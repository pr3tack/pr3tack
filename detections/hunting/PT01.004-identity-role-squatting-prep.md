# Hunt: Identity Role Squatting Prep — Dormant IAM Namespace Registration

---

## Hypothesis

An adversary is registering identity namespaces, role names, or service account names that closely resemble legitimate internal identities — either as typosquats, role-name reservations, or dormant accounts — in preparation for privilege escalation, trust chain hijacking, or impersonation during a later attack phase.

---

## Hunt Scope

| Scope Item | Detail |
|---|---|
| **Time window** | Last 90 days |
| **Data sources** | AWS CloudTrail (IAM events), Entra ID audit logs, GCP IAM audit logs, Okta user/group creation logs, internal directory services |
| **Key questions** | Are any recently created roles or users dormant (no API calls since creation)? Do any new identity names closely resemble existing privileged identities? Are there roles with admin-adjacent names created by non-admin identities? Are any external accounts registered in your identity namespace? |
| **Priority targets** | IAM role creation, service account creation, group membership changes, user provisioning events |

## Hunt Queries

### Hunt 1 — Dormant Roles/Users Created in Last 90 Days

```sql
SELECT identity_name, identity_type, created_by, created_at,
       last_activity_date,
       DATEDIFF(NOW(), created_at) as days_since_creation
FROM iam_identities
WHERE created_at >= NOW() - INTERVAL 90 DAYS
AND (last_activity_date IS NULL OR last_activity_date = created_at)
AND identity_type IN ('role', 'service_account', 'user')
ORDER BY days_since_creation DESC
```

### Hunt 2 — Typosquat Detection on Privileged Identity Names

Hunt for newly created identities whose names are within edit-distance 2 of existing privileged identities (admin roles, service accounts, break-glass accounts):

```sql
-- Requires a fuzzy-match function or external enrichment step
SELECT new_identity.name, existing_identity.name as similar_to,
       levenshtein(new_identity.name, existing_identity.name) as edit_distance,
       new_identity.created_at
FROM iam_identities new_identity
CROSS JOIN privileged_identities existing_identity
WHERE new_identity.created_at >= NOW() - INTERVAL 90 DAYS
AND levenshtein(new_identity.name, existing_identity.name) <= 2
AND new_identity.name != existing_identity.name
ORDER BY edit_distance ASC, new_identity.created_at DESC
```

### Hunt 3 — Role Creation by Non-Admin Identity

Alert on IAM role or service account creation events performed by identities that are not in the approved provisioning allowlist:

```sql
SELECT eventTime, userIdentity.arn as creator, requestParameters.roleName as new_role,
       requestParameters.assumeRolePolicyDocument
FROM cloudtrail_logs
WHERE eventName IN ('CreateRole', 'CreateUser', 'CreateServiceAccount')
AND userIdentity.arn NOT IN (SELECT arn FROM approved_provisioners)
AND eventTime >= NOW() - INTERVAL 90 DAYS
ORDER BY eventTime DESC
```

### Hunt 4 — Roles with Admin-Adjacent Naming Patterns

Hunt for newly created roles whose names contain admin-signalling terms but were not created through the approved provisioning pipeline:

```sql
SELECT identity_name, created_by, created_at, permissions_boundary
FROM iam_identities
WHERE created_at >= NOW() - INTERVAL 90 DAYS
AND (identity_name ILIKE '%admin%'
     OR identity_name ILIKE '%root%'
     OR identity_name ILIKE '%sre%'
     OR identity_name ILIKE '%security%'
     OR identity_name ILIKE '%breakglass%'
     OR identity_name ILIKE '%emergency%')
AND created_by NOT IN (SELECT arn FROM approved_provisioners)
ORDER BY created_at DESC
```

## Expected Findings

| Finding Type | Likely Explanation | Priority |
|---|---|---|
| Dormant role, 60+ days old, zero API calls | Unreviewed provisioning or squatting preparation | Medium |
| Identity name within edit-distance 1 of admin role | Typosquat — high confidence squatting attempt | Critical |
| Role created by non-admin with admin-adjacent name | Privilege escalation preparation | Critical |
| Service account registered but never assumed | Namespace reservation for future use | Medium |

---

## Tuning Notes

- **IaC provisioning:** Terraform and other IaC tools legitimately create roles that may appear dormant until first deployment — enrich with CI/CD pipeline identity ARNs and exclude from Hunt 1
- **Levenshtein threshold:** Edit-distance 2 will generate false positives in environments with many similarly named service accounts — start at distance 1 and expand if coverage is insufficient
- **Naming conventions:** Establish and document a canonical IAM naming convention; Hunt 4 findings are only meaningful if your naming standard is enforced and documented

---

## Validation

1. Create a test IAM role named `security-admn` (1 character from `security-admin`) using a non-admin identity
2. Leave it dormant for 48 hours
3. Confirm Hunt 1 identifies it as dormant
4. Confirm Hunt 2 identifies it as a typosquat of `security-admin` (edit-distance 1)
5. Confirm Hunt 3 flags the non-admin creator
6. Confirm Hunt 4 flags the `security` keyword in the name
