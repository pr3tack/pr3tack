# Hunt: Vector Database Reconnaissance — Embedding Space Probing

## Metadata

| Field | Value |

| **Tactic** | TA01 — Reconnaissance |
| **Detection Type** | Hunt |
| **Data Source** | Vector DB Access Logs / API Gateway / Network Flow |
| **Confidence** | Medium |
| **Status** | Draft |

---

## Objective

Identify adversarial iterative nearest-neighbour query patterns against vector database or RAG pipeline endpoints that are consistent with systematic embedding space probing and source document reconstruction attempts.

---

## Hypothesis

> An adversary with access to a target's AI-powered product has discovered the underlying vector database or RAG API endpoint and is systematically querying it — issuing iterative nearest-neighbour searches with slight embedding variations to reconstruct source document chunks without triggering simple rate limits.

---

## Hunt Queries

### Hunt 1: High-Volume Nearest-Neighbour Queries from Single Client (KQL)

```kql
VectorDBAccessLogs
| where TimeGenerated > ago(24h)
| where OperationType in ("query", "search", "nearest_neighbor", "similarity_search")
| summarize
    QueryCount = count(),
    UniqueVectors = dcount(QueryVector),   // Distinct query vectors = systematic exploration
    AvgResultCount = avg(ResultCount),
    TimeSpan = max(TimeGenerated) - min(TimeGenerated)
  by ClientIP, ApiKeyId
| where QueryCount > 100
| where UniqueVectors > 80                 // High uniqueness = systematic exploration
| extend QueriesPerMinute = QueryCount / (TimeSpan / 1m)
| project TimeGenerated, ClientIP, ApiKeyId, QueryCount, UniqueVectors, QueriesPerMinute, AvgResultCount
| order by QueryCount desc
```

### Hunt 2: Metadata Field Enumeration — Credential/Path Extraction (KQL)

```kql
VectorDBAccessLogs
| where TimeGenerated > ago(7d)
| where OperationType in ("query", "search")
| where ResultMetadata has_any (
    "password", "secret", "api_key", "token", "credential",
    "internal", "private", "confidential", "restricted"
  )
| summarize
    HitCount = count(),
    UniqueMetadataKeys = make_set(ResultMetadataKeys)
  by ClientIP, ApiKeyId
| where HitCount > 5
| project TimeGenerated, ClientIP, ApiKeyId, HitCount, UniqueMetadataKeys
```

### Hunt 3: Unauthenticated Vector DB Port Exposure (Network — KQL)

```kql
NetworkFlowLogs
| where TimeGenerated > ago(7d)
| where DestinationPort in (
    6333,   // Qdrant
    8080,   // Weaviate HTTP
    50051,  // Weaviate gRPC
    19530,  // Milvus
    8000    // ChromaDB
  )
| where SourceIPType == "External"
| summarize
    ConnectionCount = count(),
    UniqueSourceIPs = dcount(SourceIP)
  by DestinationIP, DestinationPort
| where ConnectionCount > 10
| project TimeGenerated, DestinationIP, DestinationPort, ConnectionCount, UniqueSourceIPs
```

### Hunt 4: Iterative Query Variation Pattern — Embedding Boundary Probing (Python)

```python
"""
Hunt script: Detect systematic variation in consecutive query vectors
indicative of embedding space boundary probing.

Run against exported vector DB query logs.
"""
import numpy as np
from scipy.spatial.distance import cosine

def detect_boundary_probing(query_vectors: list, threshold: float = 0.05) -> list:
    """
    Detects sequences of queries where consecutive vectors have very small
    cosine distance — indicative of systematic boundary exploration rather
    than organic search behaviour.
    """
    flags = []
    for i in range(1, len(query_vectors)):
        distance = cosine(query_vectors[i-1], query_vectors[i])
        if distance < threshold:
            flags.append({
                "query_index": i,
                "cosine_distance": distance,
                "flag": "systematic_boundary_probing"
            })

    # Flag sessions where > 30% of consecutive queries show boundary probing
    if len(flags) / len(query_vectors) > 0.3:
        return {"risk": "HIGH", "flags": flags}
    return {"risk": "LOW", "flags": flags}
```

## Expected Findings

| Finding Type | Likely Explanation | Priority |
|---|---|---|
| High unique-vector query volume from single IP | Systematic embedding space probing | Critical |
| Sensitive keywords in returned metadata | Data classification gap in vector DB | High |
| Unauthenticated vector DB port exposure | Network misconfiguration | Critical |
| Iterative cosine-similar query sequences | Boundary probing for document reconstruction | High |

---

## Tuning Notes

- **Baseline:** Establish baseline query volumes per authenticated client before hunting. Organic RAG usage produces varied, non-systematic query patterns — high uniqueness with low cosine variation between consecutive queries is the key anomaly signal
- **Authentication state:** Unauthenticated access to a vector DB is always a finding, regardless of query volume
- **Metadata sensitivity:** Review what fields are stored in vector DB metadata — credentials, internal paths, and user data should never be stored as raw metadata

## Validation

1. Set up a test Qdrant or ChromaDB instance with sample embeddings
2. Run an automated probing script (e.g., using the `qdrant-client` Python library) issuing 200+ iterative nearest-neighbour queries with slight vector variations
3. Confirm Hunt 1 and Hunt 4 flag the session
4. Verify Hunt 3 detects the open port from an external source IP
