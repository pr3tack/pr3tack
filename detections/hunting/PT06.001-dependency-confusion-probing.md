# Hunt: Dependency Confusion Probing — Package Namespace Enumeration

---

## Hypothesis

An adversary is probing our internal package registry and public package manager mirrors to enumerate private package namespaces — identifying package names that exist internally but not publicly, with the intent to publish malicious public packages using those names to trigger dependency confusion attacks.

---

## Hunt Scope

| Scope Item | Detail |
|---|---|
| **Time window** | Last 30 days |
| **Data sources** | Artifactory/Nexus/Verdaccio access logs, npm/PyPI/Maven registry request logs, build system dependency resolution logs, DNS query logs for package registry domains |
| **Key questions** | Are any external IPs querying our internal package registry for packages that don't exist publicly? Are there unusual 404 patterns on the internal registry for package names that match our internal naming conventions? Are any packages recently published to public registries that closely match internal package names? |
| **Priority targets** | Internal npm scopes, PyPI private packages, internal Maven group IDs, internal NuGet package namespaces |

---

## Hunt Queries

### Hunt 1 — Internal Registry 404 Bursts from External IPs

Hunt for external source IPs generating high volumes of 404 responses on internal package registry endpoints — indicative of namespace enumeration:

```sql
SELECT src_ip, COUNT(*) as request_count,
       COUNT(DISTINCT package_name) as distinct_packages,
       array_agg(DISTINCT package_name) as probed_packages
FROM registry_access_logs
WHERE http_status = 404
AND src_ip NOT IN (SELECT ip FROM internal_cidr_ranges)
AND timestamp >= NOW() - INTERVAL 30 DAYS
GROUP BY src_ip
HAVING distinct_packages > 10
ORDER BY distinct_packages DESC
```

### Hunt 2 — Internal-Naming-Convention Packages on Public Registries

Hunt for recently published public packages whose names match your internal package naming conventions (requires knowledge of your internal naming patterns):

```sql
-- Example for an org using 'acme-' prefix internally
SELECT package_name, publisher, publish_date, registry, download_count
FROM public_registry_monitor  -- requires external feed or registry API polling
WHERE (package_name LIKE 'acme-%'
    OR package_name LIKE '@acme/%'
    OR package_name LIKE 'internal-%'
    OR package_name LIKE 'corp-%')
AND publish_date >= NOW() - INTERVAL 30 DAYS
ORDER BY publish_date DESC
```

### Hunt 3 — Build System Fallback to Public Registry for Internal Package Names

Hunt in build logs for dependency resolution events where an internal package name was resolved against a public registry fallback — indicating the internal package was not found locally and the build tool fell back to public:

```sql
SELECT build_id, package_name, resolved_registry, resolution_timestamp
FROM build_dependency_logs
WHERE resolved_registry IN ('registry.npmjs.org', 'pypi.org', 'repo1.maven.org')
AND package_name LIKE '%internal-naming-prefix%'
AND resolution_timestamp >= NOW() - INTERVAL 30 DAYS
ORDER BY resolution_timestamp DESC
```

### Hunt 4 — DNS Queries to Internal Registry Domains from Unexpected Hosts

Hunt for DNS queries targeting internal package registry hostnames from hosts outside the expected build/CI infrastructure:

```sql
SELECT src_host, query_name, COUNT(*) as query_count
FROM dns_logs
WHERE query_name ILIKE '%artifactory%'
   OR query_name ILIKE '%nexus%'
   OR query_name ILIKE '%packages.internal%'
   OR query_name ILIKE '%registry.internal%'
AND src_host NOT IN (SELECT hostname FROM build_infrastructure)
AND timestamp >= NOW() - INTERVAL 30 DAYS
GROUP BY src_host, query_name
HAVING query_count > 20
ORDER BY query_count DESC
```

## Expected Findings

| Finding Type | Likely Explanation | Priority |
|---|---|---|
| External IP, 10+ distinct 404s on internal registry | Namespace enumeration — high confidence probing | High |
| Public package matching internal naming convention | Dependency confusion preparation — investigate immediately | Critical |
| Internal package name resolved via public registry | Build misconfiguration or successful confusion | Critical |
| External host querying internal registry DNS | Lateral movement or insider reconnaissance | High |

---

## Tuning Notes

- **Registry scoping:** Internal package registries should not be reachable from external IPs at all — any external IP in Hunt 1 is both a detection signal and a misconfiguration finding
- **Public registry monitoring:** Hunt 2 requires an external monitoring capability (e.g., Socket.dev, Phylum, Deps.dev) or periodic polling of public registries for your namespace — this should be operationalised as a continuous control, not a one-time hunt
- **Naming convention documentation:** Hunt 2 and 3 are only effective if your internal package naming conventions are documented and enforced — establish this as a prerequisite

---

## Validation

1. From an external test IP, issue 20 requests for non-existent package names matching your internal convention against your internal registry
2. Confirm Hunt 1 identifies the test IP with 20 distinct 404s
3. Publish a test package to a public registry using an internal naming convention (e.g., `acme-test-confusion-pkg`) and confirm Hunt 2 identifies it
4. Trigger a build that falls back to public registry for an internal package name and confirm Hunt 3 fires
