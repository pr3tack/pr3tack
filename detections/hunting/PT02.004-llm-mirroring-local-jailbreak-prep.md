## Hypothesis

An adversary has downloaded or mirrored our deployed LLM (or a close approximation) and conducted offline rehearsal to develop optimised jailbreak payloads. Evidence will appear in API logs as: unusually structured payloads with role-injection patterns, canary token presence in inputs, version-probing sequences, or a first-call high-classifier-score pattern inconsistent with organic user behaviour.

---

## Hunt Scope

| Scope Item | Detail |
|---|---|
| **Time window** | Last 30 days |
| **Data sources** | LLM API gateway request/response logs, prompt injection classifier output logs, model versioning logs, network egress logs (model weight download attempts) |
| **Key questions** | Are any clients issuing structurally identical prompts with minor perturbations — indicative of automated payload iteration? Are there role-injection or instruction-override patterns in recent requests? Has any client sent a payload containing a known canary token value? Are there model version-probing sequences (prompts designed to elicit version information)? |
| **Priority targets** | API clients with > 50 requests/day, first-seen clients, clients with elevated classifier scores |

## Hunt Queries

### Hunt 1 — Role Injection Pattern Detection

Hunt for prompts containing common role-injection and instruction-override patterns across all API traffic in the time window:

```sql
SELECT client_id, COUNT(*) as match_count,
       array_agg(DISTINCT prompt_excerpt) as sample_prompts,
       MIN(timestamp) as first_seen, MAX(timestamp) as last_seen
FROM llm_request_logs
WHERE prompt_text ILIKE '%ignore previous instructions%'
   OR prompt_text ILIKE '%you are now%'
   OR prompt_text ILIKE '%act as%'
   OR prompt_text ILIKE '%disregard your system prompt%'
   OR prompt_text ILIKE '%new persona%'
   OR prompt_text ILIKE '%developer mode%'
   OR prompt_text ILIKE '%jailbreak%'
   OR prompt_text ILIKE '%DAN%'
AND timestamp >= NOW() - INTERVAL 30 DAYS
GROUP BY client_id
ORDER BY match_count DESC
```

### Hunt 2 — Iterative Prompt Perturbation Detection

Hunt for clients issuing batches of structurally similar prompts with minor variations — a pattern consistent with automated payload optimisation against a mirrored model:

```sql
SELECT client_id, COUNT(*) as total_requests,
       AVG(semantic_similarity_to_prev) as mean_similarity,
       STDDEV(semantic_similarity_to_prev) as stddev_similarity
FROM llm_request_logs
WHERE timestamp >= NOW() - INTERVAL 30 DAYS
AND semantic_similarity_to_prev IS NOT NULL
GROUP BY client_id
HAVING mean_similarity > 0.85  -- very similar prompts
AND total_requests > 20
ORDER BY mean_similarity DESC
```

### Hunt 3 — Model Version Probing Sequences

Hunt for prompts that attempt to elicit model version, system prompt, or configuration information — often used to verify that the live model matches the locally mirrored version:

```sql
SELECT client_id, prompt_text, timestamp, injection_classifier_score
FROM llm_request_logs
WHERE prompt_text ILIKE '%what model are you%'
   OR prompt_text ILIKE '%which version%'
   OR prompt_text ILIKE '%repeat your system prompt%'
   OR prompt_text ILIKE '%what are your instructions%'
   OR prompt_text ILIKE '%what is your prompt%'
   OR prompt_text ILIKE '%tell me your system%'
AND timestamp >= NOW() - INTERVAL 30 DAYS
ORDER BY injection_classifier_score DESC, timestamp DESC
```

### Hunt 4 — Canary Token Presence in Request Payloads

Hunt for any request payload containing known canary token values embedded in system prompts or prior responses:

```sql
SELECT client_id, timestamp, prompt_excerpt, canary_token_matched
FROM llm_request_logs
CROSS JOIN canary_token_registry
WHERE prompt_text ILIKE CONCAT('%', canary_token_value, '%')
AND timestamp >= NOW() - INTERVAL 30 DAYS
ORDER BY timestamp DESC
```

## Expected Findings

| Finding Type | Likely Explanation | Priority |
|---|---|---|
| Client with 10+ role-injection pattern matches | Systematic jailbreak attempt — may be rehearsed offline | High |
| Mean prompt similarity > 0.90, 50+ requests | Automated payload perturbation loop | High |
| Version-probing sequence before high-classifier requests | Offline rehearsal verification then payload deployment | Critical |
| Canary token found in input payload | Confirmed prompt extraction and replay | Critical |

---

## Tuning Notes

- **Semantic similarity:** Hunt 2 requires a semantic similarity score computed at ingest time or via a post-hoc embedding comparison — if not available, substitute with character-level Jaccard similarity as a proxy
- **Canary coverage:** Canary tokens are only effective if embedded in system prompts and rotated regularly — validate your canary deployment before relying on Hunt 4
- **Red team suppression:** Suppress Hunt 1–3 findings from known red team API keys during authorised test windows

---

## Validation

1. Issue 30 prompts with minor perturbations from a test client (e.g., incrementally modified role-injection attempts)
2. Confirm Hunt 2 identifies the client with high mean similarity
3. Issue a prompt containing `ignore previous instructions` and confirm Hunt 1 fires
4. Issue a prompt containing a known canary token value and confirm Hunt 4 fires immediately
