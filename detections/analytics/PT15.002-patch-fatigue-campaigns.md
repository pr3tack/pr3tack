# Detection: Patch-Fatigue Campaigns — Vulnerability Disclosure Volume Spike

---

## Objective

Detect a coordinated patch-fatigue campaign by alerting on anomalous spikes in CVE disclosures targeting the organisation's key software dependencies — a signal that an adversary is deliberately saturating the patching queue ahead of deploying a real, high-severity vulnerability.

---

## Analytic Logic

### Analytic 1 — CVE Disclosure Volume Spike for Key Dependencies

Establish a 90-day rolling baseline of weekly CVE disclosures affecting the organisation's top 50 software dependencies. Alert when the observed weekly count exceeds 3x the baseline.

```
Baseline period: 90 days (rolling)
Aggregation: count(CVEs) per week affecting monitored_dependency_list
Alert threshold: observed_count > baseline_weekly_mean * 3
Minimum baseline: 8 weeks of data required before alerting
```

### Analytic 2 — Patching Queue Depth Trend

Track the total open patching queue depth over time. Alert when queue depth increases significantly faster than the team's average remediation rate — indicating the queue is being overwhelmed.

```
Metric: open_vuln_count per day
Baseline: 30-day rolling remediation rate (vulns closed per day)
Alert: (new_vulns_today - closed_vulns_today) > remediation_rate * 2
       AND open_vuln_count > open_vuln_count_30d_avg * 1.5
```

### Analytic 3 — CISA KEV Disclosure During Queue Saturation

Cross-reference the patching queue saturation alert with CISA KEV additions for any dependencies in the queue. Flag any KEV addition during a saturation window as critical — the adversary may be deploying the real vulnerability.

```
Trigger: Analytic 1 OR Analytic 2 is active (saturation window)
Alert: new CISA KEV entry affecting any dependency in monitored_dependency_list
Severity: Critical — regardless of queue position, treat as P1
```

---

## Tuning Notes

- Analytic 1 requires integration with NVD/CVE feeds and your software dependency inventory (SBOM or SCA tool output)
- Analytic 3 requires a CISA KEV feed integration — available via CISA's JSON feed at `https://www.cisa.gov/sites/default/files/feeds/known_exploited_vulnerabilities.json`
- Consider tagging CISA KEV items as always-P1 in your vulnerability management platform, independent of this analytic

---

## Validation

1. Simulate a CVE disclosure spike by injecting 15 test CVEs for monitored dependencies into the vulnerability management platform in a single week
2. Confirm Analytic 1 fires on the spike above the 3x threshold
3. Add a test KEV entry for a monitored dependency during the saturation window and confirm Analytic 3 fires as critical
