# Detection: /proc Environmental Archaeology — Anomalous Cross-PID environ Read Patterns

---

## Objective

Detect statistically anomalous patterns of `/proc/[pid]/environ` file reads across multiple PIDs from a single process or user — a strong behavioural indicator of systematic environment variable harvesting, which leaves minimal forensic artefacts and is not logged by default on most Linux distributions.

---

## Analytic Logic

### Analytic 1 — Cross-PID environ Read Volume

Track the count of distinct `/proc/[pid]/environ` paths read by a single process within a sliding window. Legitimate processes rarely need to read the environment of more than 1–2 other processes; reading environments of 5+ distinct PIDs is a strong anomaly signal.

```
Data source:      auditd (syscall=openat/read, path=/proc/*/environ) OR eBPF file open events
Window:           10 minutes (sliding)
Aggregation:      count(distinct target_pid) GROUP BY host, reader_pid, reader_uid
Alert threshold:  distinct_target_pids > 5
Exclude:          reader_comm IN ('ps', 'top', 'htop', 'systemd', 'dockerd', 'containerd')
```

### Analytic 2 — Rapid Sequential /proc/environ Reads

Alert on a burst of `/proc/[pid]/environ` reads in rapid succession — consistent with a scripted enumeration loop rather than organic process inspection.

```
Data source:      auditd file access events
Window:           60 seconds
Aggregation:      count(path=/proc/*/environ) GROUP BY host, reader_pid
Alert threshold:  count > 10 within 60 seconds
```

### Analytic 3 — /proc/environ Access from Non-Standard Process

Alert when a process that does not ordinarily access `/proc/[pid]/environ` does so — using a per-host process name baseline.

```
Baseline period:  14 days (rolling)
Method:           Allowlist of process names observed accessing /proc/*/environ
Alert:            Any process NOT in the allowlist accessing /proc/*/environ
Minimum baseline: Require ≥ 7 days of data; suppress alerts during initial learning period
```

### Analytic 4 — /proc/environ Reads Followed by Outbound Network Connection

Correlate `/proc/*/environ` access events with outbound network connections from the same process within a short time window — indicative of credential harvesting followed by exfiltration.

```
Sequence:
  1. count(path=/proc/*/environ, reader_pid=X) > 3 within 5 minutes
  2. network_connect(src_pid=X, direction=outbound) within 2 minutes of step 1
Alert: Both conditions met in sequence
```

---

## Tuning Notes

- **hidepid enforcement:** On hosts with `hidepid=2` enforced on `/proc`, the attack surface is significantly reduced — prioritise alerting on hosts where `hidepid` is absent or set to `0`
- **auditd rule required:** `/proc` reads are not audited by default; the following auditd rule must be active:
  ```
  -a always,exit -F arch=b64 -S openat -F path=/proc -F key=proc_environ_read
  ```
- **eBPF preferred:** eBPF-based telemetry (Falco, Tetragon) provides lower-overhead `/proc` access monitoring than auditd and is recommended for high-throughput environments
- **Container PID namespaces:** In correctly configured Kubernetes pods, cross-container `/proc` access is blocked by PID namespace isolation — alert priority should be highest on bare-metal hosts and misconfigured pods sharing the host PID namespace (`hostPID: true`)
- **False positives:** Monitoring tools (`ps`, `top`, `htop`, `procfs`-based dashboards) legitimately read `/proc/*/environ` — maintain a robust allowlist and validate it regularly

---

## Validation

1. Enable auditd with the proc_environ_read rule on a test host (no `hidepid` set)
2. Run the following enumeration script:
   ```bash
   for pid in /proc/[0-9]*/environ; do cat "$pid" 2>/dev/null; done
   ```
3. Confirm Analytic 1 fires (distinct PIDs > 5) and Analytic 2 fires (burst within 60s)
4. Run the same script from a process not in the allowlist and confirm Analytic 3 fires
5. Follow with a `curl` outbound call and confirm Analytic 4 fires the correlated sequence
