# Detection: Backup Bit-Rot Injection — Backup Integrity Verification Anomaly

## Metadata

| Field | Value |
|---|---|
| **Technique** | PT15.001 — Backup Bit-Rot Injection |
| **Tactic** | PT15 — Resilience Erosion |
| **Detection Type** | Analytic |
| **Data Source** | Backup Platform Logs / FIM / Storage Access Logs |
| **Confidence** | Medium |
| **Status** | Draft |

---

## Objective

Detect backup corruption injection by tracking backup metadata modifications outside scheduled backup windows and by performing regular test restores with anomaly detection on restoration success rates.

---

## Analytic Logic

### Analytic 1 — Backup Catalogue Modification Outside Scheduled Window

Baseline backup metadata write operations by time-of-day. Alert on backup catalogue or checksum file modifications outside the scheduled backup window.

```
Baseline: scheduled backup windows per backup job (from backup platform config)
Alert: backup_catalogue_write_event
       AND timestamp NOT IN scheduled_backup_windows
       AND writer_identity NOT IN (backup_service_accounts)
```

### Analytic 2 — Test Restore Success Rate Degradation

Track the success rate of regular test restores per backup set. Alert when the success rate drops below 100% — any restoration failure is a finding.

```
Frequency: Weekly test restore per backup set type
Metric: restore_success_rate per backup_set
Alert: restore_success_rate < 1.0
Severity: Critical
```

### Analytic 3 — Backup Storage Access by Non-Backup Process

Alert on reads or writes to backup storage locations by processes not associated with the backup platform.

```
Data source: FIM / EDR file access
Alert: file_access_event
       AND file_path IN (backup_storage_paths)
       AND process_name NOT IN (backup_platform_processes)
       AND access_type IN ('write', 'modify', 'delete')
```

---

## Tuning Notes

- Analytic 2 (test restore) should be automated — manual test restores are performed too infrequently to be a reliable detection control
- Backup storage path monitoring requires FIM coverage of the backup storage target — may require agent deployment on backup servers
- Cloud backup monitoring: use S3 Object Lock violation alerts and CloudTrail `DeleteObject` events on backup buckets

---

## Validation

1. Modify a test backup file outside the scheduled backup window using a non-backup process
2. Confirm Analytic 1 fires on the out-of-window catalogue modification
3. Confirm Analytic 3 fires on the non-backup-process file write
4. Corrupt a test backup file and attempt restoration — confirm Analytic 2 captures the restore failure
