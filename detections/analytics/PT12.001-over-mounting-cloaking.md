# Detection: Over-Mounting Cloaking — Anomalous Mount Syscall Frequency

## Metadata

| Field | Value |
|---|---|
| **Technique** | PT12.001 — Over-Mounting Cloaking |
| **Tactic** | PT12 — Unix Contextual Stealth |
| **Detection Type** | Analytic |
| **Data Source** | EDR / auditd / Linux Audit Framework / Container Runtime Logs |
| **Confidence** | Medium |
| **Status** | Draft |

---

## Objective

Detect statistically anomalous mount/bind-mount syscall rates per process or container — a behavioural signal for over-mounting cloaking activity that complements the point-in-time rule by catching low-and-slow or repeated mount operations that individually fall below rule thresholds.

---

## Analytic Logic

### Analytic 1 — Mount Frequency Baseline Deviation

Track the rolling 7-day baseline of `mount` syscalls per host, per process name, and per user. Alert when the observed count in any 1-hour window exceeds the baseline mean by more than 3 standard deviations.

```
Baseline period:  7 days (rolling)
Aggregation:      count(syscall=mount) GROUP BY host, process_name, uid
Window:           1 hour
Alert threshold:  observed_count > baseline_mean + (3 × baseline_stddev)
Minimum baseline: Require ≥ 5 days of data before alerting
```

### Analytic 2 — Mount-to-Unmount Ratio Anomaly

In legitimate container or OS operation, mounts and unmounts occur in balanced pairs. An abnormally high mount-to-unmount ratio over a sliding window indicates mounts that are left in place — consistent with cloaking rather than transient runtime mounts.

```
Window:           24 hours (sliding)
Aggregation:      count(syscall=mount) / count(syscall=umount) GROUP BY host, uid
Alert threshold:  ratio > 5 AND count(syscall=mount) > 10
```

### Analytic 3 — Repeat Mount Targets

Alert when the same directory path is the target of more than 2 mount operations within a 1-hour window from different processes or users — indicative of repeated cloaking attempts or tooling iteration.

```
Window:           1 hour
Aggregation:      count(syscall=mount) GROUP BY host, mount_target_path
Alert threshold:  count > 2 AND mount_target_path NOT IN (standard_os_paths)
Standard OS paths exclusion: /tmp, /run, /dev/shm, /sys, /proc, /var/lib/docker
```

---

## Tuning Notes

- **Baseline stability:** Container orchestration environments (Kubernetes) have naturally high mount activity at pod start/stop — segment baseline by workload type (container vs. bare-metal host)
- **Process name allowlisting:** Exclude known container runtimes (`containerd`, `dockerd`, `runc`) from Analytic 1 but retain them in Analytic 3 (unusual mount targets from runtimes are still valid findings)
- **Mount target enrichment:** Enrich `mount_target_path` with a filesystem inventory to identify non-empty directories — a mount over a non-empty directory is a higher-confidence indicator
- **Low-volume environments:** On hosts with very low baseline mount activity (< 5 per day), lower the Analytic 1 threshold to `baseline_mean + 2 × stddev`

---

## Validation

1. On a test Linux host with auditd enabled, run:
   ```bash
   for i in {1..15}; do mount --bind /tmp /opt/test$i; done
   ```
2. Confirm Analytic 1 fires within the 1-hour window
3. Leave mounts in place for 24 hours and confirm Analytic 2 fires on the mount-to-unmount ratio
4. Unmount all and verify the ratio alert clears on the next evaluation cycle
