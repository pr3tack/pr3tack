# Detection: Billing/Cost Shock Forcing — Anomalous Cloud Resource Provisioning Spike

## Objective

Detect deliberate cloud cost-shock events by alerting on anomalous resource provisioning spikes that exceed normal operational baselines — distinguishing adversarial flooding from legitimate scale events.

---

## Analytic Logic

### Analytic 1 — Compute Instance Provisioning Rate Anomaly

Baseline the daily compute instance provisioning rate per IAM identity. Alert when an identity provisions instances at more than 5x their baseline rate within a 1-hour window.

```
Baseline period: 30 days (rolling)
Aggregation: count(instance_launches) per identity per hour
Alert: observed_hourly_count > baseline_hourly_mean * 5
       AND instance_type NOT IN approved_autoscaling_types
```

### Analytic 2 — Data Transfer Volume Spike

Baseline daily outbound data transfer per account. Alert when transfer volume exceeds 3x the 30-day daily baseline in a single hour.

```
Baseline period: 30 days (rolling)
Aggregation: sum(bytes_transferred_out) per account per hour
Alert: observed_hourly_bytes > baseline_daily_bytes * 0.5
       (i.e. > 50% of daily baseline in a single hour)
```

### Analytic 3 — Cost Spike Correlated with Security Event

Correlate budget alert events with concurrent security alerts from the same account/region. A cost spike coinciding with unusual access patterns is critical.

```
Sequence (within 30-minute window):
  1. Budget alert fires (spend > threshold)
  2. Security alert fires (unusual IAM activity, new resource in unexpected region)
Alert: Both conditions met within 30-minute window
Severity: Critical
```

---

## Tuning Notes

- Analytic 1 should exclude known auto-scaling groups and pre-approved batch job identities — these legitimately provision at high rates
- Analytic 2 thresholds should be calibrated to your typical data transfer patterns; media/CDN workloads will need higher thresholds
- Implement cloud budget guardrails (AWS Budget Actions, GCP Budget Alerts) as a preventive control — do not rely solely on detective analytics

---

## Validation

1. Use a test IAM identity to provision 20x the normal number of t3.micro instances in 1 hour
2. Confirm Analytic 1 fires for the provisioning rate anomaly
3. Trigger a test budget alert and simultaneously fire a test IAM anomaly alert
4. Confirm Analytic 3 fires on the correlated sequence
