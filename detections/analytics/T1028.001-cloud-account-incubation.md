## Objective

Detect the behavioural transition of an incubated cloud account — when a previously low-activity, aged cloud tenant suddenly shifts to high-volume, targeted API activity against your infrastructure. This transition pattern distinguishes incubated accounts from both genuinely new and consistently active tenants.

---

## Analytic Logic

### Incubation-to-Activation Transition Model

The key signal is a **sudden behavioural inflection point** in a cloud account that previously had low, benign-pattern activity.

```
Risk Score = (Activity Velocity Score) × (Tenant Age Modifier) × (Target Sensitivity Modifier)

Where:
  Activity Velocity Score:
    - Requests in last 24h / Average requests per day over prior 30 days
    - Score = 1 if ratio < 5×  (normal growth)
    - Score = 3 if ratio 5×–20× (elevated)
    - Score = 5 if ratio > 20× (spike — incubation-to-activation pattern)

  Tenant Age Modifier:
    - Age < 30 days:   modifier = 2.0 (new tenant, higher baseline risk)
    - Age 30–90 days:  modifier = 1.5 (recently created — incubation window)
    - Age 91–365 days: modifier = 1.2 (established but still scrutinised)
    - Age > 365 days:  modifier = 1.0 (baseline)

  Target Sensitivity Modifier:
    - Requests to authentication/identity APIs:   modifier = 2.0
    - Requests to data/storage APIs:              modifier = 1.8
    - Requests to admin/management APIs:          modifier = 2.5
    - Requests to general application endpoints:  modifier = 1.0

Final Risk Score ≥ 5:  Alert — Medium
Final Risk Score ≥ 10: Alert — High (page on-call)
Final Risk Score ≥ 15: Alert — Critical (automated throttle + page)
```

### KQL Implementation (Microsoft Sentinel / API Gateway Logs)

```kql
let lookback_days = 30;
let alert_window = 1d;
let spike_threshold = 5;

let historical_baseline =
    APIGatewayLogs
    | where TimeGenerated between (ago(lookback_days + alert_window) .. ago(alert_window))
    | summarize HistoricalDailyAvg = count() / lookback_days by CloudTenantId, CloudTenantAge;

let recent_activity =
    APIGatewayLogs
    | where TimeGenerated > ago(alert_window)
    | summarize RecentCount = count(), Endpoints = make_set(RequestUri) by CloudTenantId;

historical_baseline
| join kind=inner recent_activity on CloudTenantId
| extend VelocityRatio = RecentCount / max_of(HistoricalDailyAvg, 1)
| extend TenantAgeDays = CloudTenantAge
| where VelocityRatio > spike_threshold
| extend
    AgeModifier = case(
        TenantAgeDays < 30, 2.0,
        TenantAgeDays < 90, 1.5,
        TenantAgeDays < 365, 1.2,
        1.0),
    RiskScore = VelocityRatio * AgeModifier
| where RiskScore >= 5
| project CloudTenantId, TenantAgeDays, HistoricalDailyAvg, RecentCount, VelocityRatio, RiskScore, Endpoints
| order by RiskScore desc
```

## Tuning Notes

- **Metadata enrichment:** Enrich API gateway logs with cloud provider tenant metadata (account creation date, organisational association) using GreyNoise, Spur, or cloud provider APIs. Without tenant age, this analytic cannot compute the age modifier.
- **Baseline stability:** The 30-day lookback window requires 30 days of data per tenant. New tenants with no prior activity history default to the `Age < 30 days` tier and should be treated with elevated scrutiny regardless.
- **False positives:** Legitimate tenants launching new integrations or seasonal services may exhibit similar transition patterns. Enrich alerts with the tenant's known business relationship before escalating.
- **Cloud reputation feeds:** Integrate GreyNoise and Spur feeds to pre-score inbound cloud tenant IPs. Tenants already flagged as malicious infrastructure skip the incubation model and go straight to block.

---

## Validation

1. Create a test cloud account and generate low-volume traffic against a test endpoint for 30 days
2. Spike the traffic volume by 10× on day 31
3. Confirm the analytic detects the velocity transition and generates a High alert
4. Validate that a legitimately growing integration (steady ramp) does not trigger the spike threshold
