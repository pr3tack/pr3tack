## Objective

Detect statistically anomalous patterns in prompt injection classifier scores and safety filter trigger rates over time — a behavioural signal that an adversary has conducted offline rehearsal against a mirrored model and is now deploying optimised payloads against the live system, causing classifier score distributions to shift unexpectedly.

---

## Analytic Logic

### Analytic 1 — Prompt Injection Classifier Score Spike per Client

Track the rolling 7-day baseline of prompt injection classifier scores per API client identity. Alert when a client's mean classifier score in any 1-hour window significantly exceeds their baseline — indicative of a shift from benign to adversarial input.

```
Baseline period:  7 days (rolling)
Aggregation:      mean(injection_classifier_score) GROUP BY client_id, hour
Window:           1 hour
Alert threshold:  observed_mean > baseline_mean + (3 × baseline_stddev)
Minimum baseline: Require ≥ 3 days of data before alerting
```

### Analytic 2 — Safety Filter Bypass Rate Trend

Track the rate of successful prompt completions that previously would have been blocked (measured by retrospective classifier scoring of outputs). A rising bypass rate from a specific client over time indicates iterative payload optimisation.

```
Baseline period:  14 days (rolling)
Aggregation:      count(response_safety_score < threshold) / count(total_requests)
                  GROUP BY client_id, day
Alert:            bypass_rate_today > bypass_rate_baseline × 2
                  AND total_requests > 50  -- minimum volume for statistical validity
```

### Analytic 3 — First-Call High-Classifier-Score Pattern

Alert when a new client's very first API call has a prompt injection classifier score above the 90th percentile of normal traffic — a strong indicator of an offline-rehearsed payload being deployed on first contact.

```
Condition:        client_first_seen = true
                  AND injection_classifier_score > p90_threshold
                  AND request_count_for_client = 1
Alert:            Immediate — no baseline required
```

### Analytic 4 — Canary Token Replay Detection

If canary tokens are embedded in system prompts or model responses, track whether any canary value appears in subsequent input payloads — indicating a prompt extraction that was then replayed.

```
Data source:      LLM input/output logs
Method:           Exact string match of known canary tokens in request payloads
Alert:            Any match — canary token in input payload = confirmed prompt extraction
Severity:         Critical
```

---

## Tuning Notes

- **Classifier dependency:** Analytics 1–3 require a deployed prompt injection classifier on the API gateway — without one, substitute with output safety filter scores if available
- **Client identity granularity:** Use API key or authenticated session ID, not source IP — shared NAT IPs will create false positives in multi-tenant environments
- **Canary token rotation:** Rotate canary tokens on a schedule to prevent adversaries from identifying and avoiding them through prolonged observation
- **Legitimate adversarial testing:** Red team and penetration testing activity will trigger these analytics — establish a suppression mechanism for authorised test windows with documented API key exclusions

---

## Validation

1. Set up a test LLM API gateway with a prompt injection classifier
2. Issue 50 benign requests to establish baseline, then issue 10 known jailbreak payloads
3. Confirm Analytic 1 fires on the classifier score spike in the same hour window
4. Issue a request containing an embedded canary token value and confirm Analytic 4 fires immediately
5. Register a new API client and issue a high-scoring jailbreak payload as the first call — confirm Analytic 3 fires
