## Objective

Detect processes reading `/proc/[pid]/environ` for PIDs outside their own process group, indicating cross-process credential harvesting via the `/proc` filesystem.

---

### CrowdStrike Falcon — Event Search

```
event_simpleName=FileOpenInfo
| search TargetFileName=/proc/*/environ
| eval AccessedPID=replace(TargetFileName, "/proc/([0-9]+)/environ", "\1")
| where AccessedPID != ProcessId    // Cross-process access
| where NOT (FileName IN ("ps", "top", "htop", "datadog-agent", "node_exporter", "docker", "crictl"))
| where uid != "0" OR (uid="0" AND NOT (ParentBaseFileName IN ("systemd", "init")))
| table _time, ComputerName, UserName, FileName, TargetFileName, AccessedPID, ProcessId
```

### auditd Rule (Deploy to `/etc/audit/rules.d/pr3tack-T1202.rules`)

```
# Detect reads of /proc/*/environ (potential credential harvesting)
-a always,exit -F arch=b64 -S open,openat -F path=/proc -F perm=r -k pr3tack_proc_environ
```

### Wiz Runtime Sensor (KQL)

```kql
WizRuntimeEvents
| where TimeGenerated > ago(1h)
| where EventType == "FILE_OPEN"
| where FilePath matches regex @"/proc/\d+/environ"
| extend AccessedPid = extract(@"/proc/(\d+)/environ", 1, FilePath)
| where AccessedPid != tostring(ProcessId)   // Cross-process access
| where ProcessName !in ("ps", "top", "htop", "datadog-agent", "node_exporter", "docker", "crictl", "kubelet")
| project TimeGenerated, NodeName, PodName, Namespace, ProcessName, ProcessId, AccessedPid, FilePath, UserId
| order by TimeGenerated desc
```

### hidepid Verification Check (Compliance Rule)

```bash
# Verify /proc is mounted with hidepid=2 or hidepid=invisible
# Run as part of CIS benchmark compliance checks
mount | grep "proc on /proc" | grep -E "hidepid=(2|invisible)"
# If this returns no output, the system is vulnerable — alert for misconfiguration
```

## Tuning Notes

- **Self-access filtering:** The most common false positive is processes reading their own `/proc/self/environ`. Ensure your SIEM correlation logic correctly excludes same-PID access.
- **Privileged processes:** Root processes (uid=0) reading `/proc/*/environ` is more common but not always legitimate. Focus high-severity alerts on non-root cross-PID access; treat root access as Medium unless from an unexpected executable.
- **Monitoring agents:** Datadog Agent, New Relic, Prometheus node_exporter, and similar tools routinely read process environments. Allowlist these by executable path, not just by user, to avoid bypass via renamed binaries.
- **hidepid:** The most effective prevention is `hidepid=2` on `/proc`. Include a compliance check for this in your configuration management tooling — alert any host that is missing this control.
- **Container environments:** In Kubernetes, cross-pod `/proc` access is prevented by PID namespace isolation — but nodes that run with `hostPID: true` are fully vulnerable. Alert on any pod specification with `hostPID: true`.

## Validation

1. On a test Linux host **without** `hidepid=2`, run a background process with an environment variable containing a fake secret: `SECRET=FAKE_KEY_12345 sleep 3600 &`
2. From a different process, read its environment: `cat /proc/$!/environ | tr '\0' '\n' | grep SECRET`
3. Confirm auditd generates the `FILE_OPEN` event for the `/proc/[pid]/environ` path
4. Confirm SIEM alert fires with cross-PID access detected
5. Repeat after applying `mount -o remount,hidepid=2 /proc` and verify access is blocked
