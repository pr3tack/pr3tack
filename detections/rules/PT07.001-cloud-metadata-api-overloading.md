# Detection: Cloud Metadata API Overloading — IMDS Request Rate Anomaly

---

## Objective

Detect anomalously high request rates to the cloud instance metadata service (IMDS) endpoint from a single process — indicative of metadata flooding to corrupt logging pipeline attribution.

---

## Detection Logic

```yaml
title: Anomalous IMDS Request Rate from Single Process
description: >
  Detects a single process generating > 100 requests per minute to the
  cloud IMDS endpoint (169.254.169.254). Normal workloads rarely require
  high-frequency metadata polling; this rate indicates deliberate flooding
  to corrupt metadata-dependent logging pipelines.
logsource:
  category: network_connection
  product: linux
detection:
  selection:
    dst_ip: '169.254.169.254'
    dst_port: 80
  timeframe: 1m
  condition: selection | count() by process_name > 100
level: high
tags:
  - pr3tack.PT07
  - pr3tack.PT07.001
---
title: Cloud Audit Log Entry with Null Instance Identity
description: >
  Detects cloud audit log entries where the instance identity field is null
  or empty — indicating potential IMDS flooding has corrupted the logging
  pipeline's metadata enrichment.
logsource:
  category: cloud
  product: aws_cloudtrail
detection:
  selection:
    userIdentity.instanceId: ''
    userIdentity.type: 'AssumedRole'
  condition: selection
level: medium
tags:
  - pr3tack.PT07
  - pr3tack.PT07.001
```

---

## Tuning Notes

- IMDSv2 instances: also monitor for high rates of PUT requests to `http://169.254.169.254/latest/api/token` — IMDSv2 token rotation flooding
- Baseline per-instance IMDS request rates in your environment; some instance types (monitoring agents, credential refresh daemons) have higher legitimate rates
- The null instanceId rule in CloudTrail is a strong signal — legitimate entries almost always have a populated instanceId for AssumedRole events

---

## Validation

1. On a test EC2 instance, run a script generating 200+ IMDS requests per minute
2. Confirm the process-rate rule fires within the 1-minute window
3. Verify CloudTrail logs show instanceId metadata during normal operation and check for null values during the flood
