# Detection: Identity Role Squatting — Ghost Role AssumeRole Attempt

## Metadata

| Field | Value |
|---|---|
| **Technique** | T1018.001 — Identity Role Squatting Prep |
| **Tactic** | TA01 — Reconnaissance |
| **Detection Type** | Rule |
| **Data Source** | AWS CloudTrail / GCP Audit Logs / SIEM |
| **Confidence** | High |
| **Status** | Draft |

---

## Objective

Detect attempts to assume or impersonate IAM roles or service accounts that have been previously decommissioned, indicating a ghost-role squatting or re-registration attempt.

---

## Detection Logic

```yaml
title: Decommissioned IAM Role AssumeRole Attempt (Ghost Role Squatting)
id: pr3tack-T1018-001
status: experimental
description: >
  Detects sts:AssumeRole or equivalent calls targeting IAM role ARNs or service
  account names that appear on a maintained watchlist of decommissioned identities.
  This is a high-fidelity indicator of ghost-role squatting or re-registration.
references:
  - https://github.com/pr3tack/pr3tack/techniques/TA01-reconnaissance/T1018.001-identity-role-squatting-prep.md
author: PR3TACK
date: 2026/05/19
tags:
  - attack.reconnaissance
  - attack.credential_access
  - pr3tack.TA01
  - pr3tack.T1018.001
logsource:
  product: aws
  service: cloudtrail
detection:
  selection:
    eventName:
      - 'AssumeRole'
      - 'AssumeRoleWithWebIdentity'
      - 'AssumeRoleWithSAML'
    requestParameters.roleArn|contains:
      # Populate with decommissioned role ARNs from your tombstone registry
      - 'arn:aws:iam::*:role/DECOMMISSIONED-*'
      - 'arn:aws:iam::*:role/DEPRECATED-*'
      - 'arn:aws:iam::*:role/LEGACY-*'
  condition: selection
falsepositives:
  - Automated scripts that have not yet been updated to use replacement roles
  - IAM permission boundary tests by the security team
level: high
fields:
  - userIdentity.arn
  - requestParameters.roleArn
  - sourceIPAddress
  - userAgent
  - awsRegion
```

### AWS CloudTrail — KQL (Microsoft Sentinel)

```kql
AWSCloudTrail
| where EventName in ("AssumeRole", "AssumeRoleWithWebIdentity", "AssumeRoleWithSAML")
| extend RoleArn = tostring(parse_json(RequestParameters).roleArn)
| where RoleArn has_any (
    // Populate from your decommissioned role watchlist
    "DECOMMISSIONED-", "DEPRECATED-", "LEGACY-", "OLD-", "DELETED-"
  )
| project TimeGenerated, UserIdentityArn, RoleArn, SourceIpAddress, UserAgent, AWSRegion
| order by TimeGenerated desc
```

### GCP Audit Logs — KQL

```kql
GCPAuditLogs
| where MethodName == "iam.serviceAccounts.actAs"
| extend ServiceAccount = tostring(AuthorizationInfo[0].resource)
| where ServiceAccount has_any (
    // Populate from your decommissioned service account watchlist
    "-decommissioned@", "-deprecated@", "-legacy@", "-deleted@"
  )
| project TimeGenerated, PrincipalEmail, ServiceAccount, CallerIp, UserAgent
| order by TimeGenerated desc
```

---

## Tuning Notes

- **Watchlist maintenance:** The effectiveness of this detection depends entirely on maintaining a current tombstone registry of decommissioned role ARNs and service account names. Integrate with your IAM lifecycle management process.
- **Naming conventions:** Adapt regex patterns to match your organisation's actual naming conventions for decommissioned resources.
- **False positives:** Legitimate scripts referencing stale role names are a common false positive. These should be treated as a finding in their own right — they confirm a dangling reference that needs remediation.
- **Escalation:** Any hit from an external or unknown principal should be escalated to Critical immediately.

---

## Validation

1. Add a test entry to the decommissioned role watchlist (e.g., `arn:aws:iam::123456789:role/DECOMMISSIONED-test-role`)
2. Attempt `aws sts assume-role --role-arn arn:aws:iam::123456789:role/DECOMMISSIONED-test-role` from a test account
3. Confirm CloudTrail event is generated and SIEM alert fires
4. Validate that the alert correctly identifies the external source IP and user identity
