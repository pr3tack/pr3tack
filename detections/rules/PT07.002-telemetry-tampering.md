# Detection: Telemetry Tampering — Agent Config File Modification

## Metadata

| Field | Value |
|---|---|
| **Technique** | PT07.002 — Telemetry Tampering |
| **Tactic** | PT07 — Defense Evasion |
| **Detection Type** | Rule |
| **Data Source** | FIM (File Integrity Monitoring) / EDR / Sysmon (Event ID 11) / auditd |
| **Confidence** | High |
| **Status** | Draft |

---

## Objective

Detect unauthorised modification of telemetry agent configuration files — the primary mechanism for selective event suppression without triggering agent availability alerts.

---

## Detection Logic

```yaml
title: Telemetry Agent Configuration File Modified by Non-Agent Process
description: >
  Detects modification of telemetry agent configuration files by a process
  other than the agent itself or the approved configuration management system.
  Selective config modification allows an adversary to suppress specific event
  types without killing the agent or triggering availability alerts.
logsource:
  category: file_event
  product: windows
detection:
  selection:
    event_id: 11  # Sysmon FileCreate/FileModify
    target_filename|contains:
      - '\SplunkUniversalForwarder\etc\system\local\inputs.conf'
      - '\Elastic\Agent\agent.yml'
      - '\osquery\osquery.conf'
      - '\nxlog\nxlog.conf'
      - 'winlogbeat.yml'
      - 'filebeat.yml'
  filter_approved:
    process_name|contains:
      - 'splunkd.exe'
      - 'elastic-agent.exe'
      - 'ansible'
      - 'chef-client'
      - 'puppet'
      - 'salt-minion'
  condition: selection and not filter_approved
level: critical
tags:
  - pr3tack.PT07
  - pr3tack.PT07.002
```

---

## Tuning Notes

- Populate the approved process list with your specific configuration management tooling
- On Linux: extend the target path list to include `/etc/rsyslog.conf`, `/etc/filebeat/filebeat.yml`, `/opt/splunkforwarder/etc/`, and Elastic Agent paths
- Cross-correlate config modification events with subsequent changes in event volume from the affected host

---

## Validation

1. Modify a Splunk inputs.conf to add an exclusion filter using a non-approved process (e.g., `notepad.exe`)
2. Confirm the rule fires on the file modification event from the non-approved process
3. Verify that a modification by the approved config management tool does not trigger the rule
