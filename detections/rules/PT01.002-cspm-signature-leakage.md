## Objective

Detect bulk DNS enumeration of corporate subdomains resolving to known security vendor infrastructure, indicative of an adversary fingerprinting the target's security stack.

---

## Detection Logic

```yaml
title: Security Stack Fingerprinting via DNS Enumeration of Vendor CNAMEs
id: pr3tack-T1015-001
status: experimental
description: >
  Detects a high volume of DNS queries from a single source targeting corporate
  subdomains, particularly where resolved CNAMEs point to known security vendor
  infrastructure (Zscaler, Netskope, Cloudflare Gateway, Palo Alto Prisma).
  This pattern is consistent with automated security-stack fingerprinting.
references:
  - https://github.com/pr3tack/pr3tack/techniques/PT01-reconnaissance/PT01.002-cspm-signature-leakage.md
author: PR3TACK
date: 2026/05/19
tags:
  - attack.reconnaissance
  - pr3tack.PT01
  - pr3tack.PT01.002
logsource:
  category: dns
  product: generic
detection:
  selection_volume:
    query|endswith:
      - '.corp.example.com'
      - '.internal.example.com'
      - '.vpn.example.com'
  timeframe: 10m
  condition: selection_volume | count() by src_ip > 50
  selection_cname_vendor:
    answer|contains:
      - '.zscaler.net'
      - '.netskope.com'
      - '.gateway.zscaler.net'
      - '.prismaaccess.com'
      - '.mcas.ms'
      - '.cloudflare-gateway.com'
falsepositives:
  - Legitimate bulk DNS resolution by internal tools (monitoring, IPAM)
  - Authorised external DNS auditing or EASM tools
level: medium
fields:
  - src_ip
  - query
  - answer
  - query_type
```

### Supplementary: HTTP Header Fingerprinting (WAF Log — KQL)

```kql
WAFLogs
| where TimeGenerated > ago(1h)
| where ResponseHeaders has_any (
    "X-Zscaler-", "X-Netskope-", "X-CrowdStrike-", "X-Palo-Alto-"
  )
| where HttpMethod == "GET"
| where isempty(AuthenticatedUser)
| summarize RequestCount = count(), Headers = make_set(ResponseHeaders) by ClientIP
| where RequestCount > 5
| project TimeGenerated, ClientIP, RequestCount, Headers
```

### Supplementary: Passive DNS Monitoring (Threat Intel Platform)

```
Alert condition:
  Domain: *.{your-corp-domain}
  CNAME answer contains: [zscaler.net, netskope.com, prismaaccess.com]
  Source: External resolver (not corporate recursive resolvers)
  Frequency: > 20 unique subdomains resolved within 60 minutes
```

---

## Tuning Notes

- **False positives:** EASM and DNS audit tools will trigger this. Allowlist known scanner ASNs and internal monitoring tool IPs.
- **CNAME list:** Maintain a live list of your confirmed security vendor CNAME patterns and update this detection as your vendor stack changes.
- **Enrichment:** Cross-reference the querying IP against threat intelligence feeds. Known security research ASNs are low priority; unknown cloud IPs are high priority.
- **Scope:** Limit to queries from sources **outside** your corporate recursive resolvers — internal queries are expected.

---

## Validation

1. Use `dig` or `dnsx` from an external IP to enumerate a set of your corporate subdomains
2. Confirm DNS query log alert fires within the configured timeframe
3. Validate CNAME enrichment correctly identifies vendor infrastructure
4. Test HTTP header detection by issuing requests to a proxied endpoint and confirming vendor headers are visible
