# Detection: Over-Mounting Cloaking — Anomalous tmpfs Mount Over Non-Empty Directory

---

## Objective

Detect the mounting of a `tmpfs` or overlay filesystem over a non-standard, non-empty directory — a strong indicator of data staging concealment via the over-mounting cloaking technique.

---

## Detection Logic

```yaml
title: Anomalous tmpfs Mount Over Non-Standard Directory
id: pr3tack-T1201-001
status: experimental
description: >
  Detects mount syscalls creating tmpfs or overlay filesystems over non-standard
  directories that may already contain data. This pattern is consistent with the
  over-mounting cloaking technique where staged data is hidden beneath a temporary
  filesystem overlay to evade file-based scanners.
references:
  - https://github.com/pr3tack/pr3tack/techniques/PT12-unix-contextual-stealth/PT12.001-over-mounting-cloaking.md
author: PR3TACK
date: 2026/05/19
tags:
  - attack.defense_evasion
  - pr3tack.PT12
  - pr3tack.PT12.001
logsource:
  product: linux
  service: auditd
detection:
  selection:
    type: 'SYSCALL'
    syscall: 'mount'
    a2|contains:
      - 'tmpfs'
      - 'overlay'
      - 'overlayfs'
  filter_legitimate:
    # Filter standard tmpfs mounts used by the OS
    a1|contains:
      - '/tmp'
      - '/run'
      - '/dev/shm'
      - '/sys'
      - '/proc'
  condition: selection and not filter_legitimate
falsepositives:
  - Container runtimes (Docker, containerd) mounting overlay filesystems during container creation
  - Kubernetes kubelet operations
  - Legitimate use of tmpfs for in-memory data processing
level: high
fields:
  - pid
  - ppid
  - uid
  - exe
  - a1   # Mount target directory
  - a2   # Filesystem type
  - comm
```

### CrowdStrike Falcon — Event Search

```
event_simpleName=ProcessRollup2
| search CommandLine="*mount*tmpfs*" OR CommandLine="*mount*overlay*"
| where NOT (TargetDirectory IN ("/tmp", "/run", "/dev/shm", "/sys", "/proc"))
| where NOT (ParentBaseFileName IN ("dockerd", "containerd", "kubelet", "runc"))
| table _time, ComputerName, UserName, ParentBaseFileName, FileName, CommandLine, ProcessId
```

### auditd Rule (Deploy to `/etc/audit/rules.d/pr3tack-T1201.rules`)

```
# Detect mount syscalls (monitor for potential over-mounting cloaking)
-a always,exit -F arch=b64 -S mount -k pr3tack_overmount
-a always,exit -F arch=b32 -S mount -k pr3tack_overmount
```

### Wiz Runtime Sensor (KQL)

```kql
WizRuntimeEvents
| where TimeGenerated > ago(1h)
| where EventType == "MOUNT_CREATED"
| where FilesystemType in ("tmpfs", "overlay", "overlayfs")
| where TargetPath !has_any ("/tmp", "/run", "/dev/shm", "/sys", "/proc", "/var/lib/docker", "/var/lib/containerd")
| project TimeGenerated, NodeName, PodName, Namespace, ProcessName, ProcessId, TargetPath, FilesystemType, UserId
| order by TimeGenerated desc
```

---

## Tuning Notes

- **Container noise:** Container runtimes generate significant overlay mount activity. The filter for `dockerd`, `containerd`, and `kubelet` parent processes is critical — tune this filter list to match your environment's container runtime executables.
- **Context enrichment:** Enrich alerts with whether the mount target directory was non-empty at time of mount (requires filesystem snapshot capability) — this significantly increases confidence.
- **Kubernetes:** In Kubernetes environments, restrict mount capability by enforcing `privileged: false` and dropping `SYS_ADMIN` in pod security policies/admission controllers.
- **Baseline:** Establish a baseline of legitimate tmpfs mounts in your environment before deploying. Mount events on standard paths (`/tmp`, `/run`, `/dev/shm`) are expected and should be filtered.

---

## Validation

1. On a test Linux host, create a directory and place a test file in it: `mkdir /tmp/test-stage && echo "staged" > /tmp/test-stage/data.txt`
2. Mount a tmpfs over it: `mount -t tmpfs tmpfs /tmp/test-stage`
3. Confirm auditd generates a `SYSCALL mount` event with key `pr3tack_overmount`
4. Confirm EDR/SIEM alert fires
5. Unmount and verify the file is accessible again: `umount /tmp/test-stage && cat /tmp/test-stage/data.txt`
