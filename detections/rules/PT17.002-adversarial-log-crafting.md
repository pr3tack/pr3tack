## Objective

Detect adversarially-crafted log entries by alerting on SIEM parser error rate spikes for specific log sources and anomalously oversized field values in structured log streams.

---

## Detection Logic

```yaml
title: SIEM Parser Error Rate Spike for Log Source
description: >
  Detects a spike in SIEM parser error rate for a specific log source —
  indicative of adversarially-crafted log entries triggering parser edge
  cases or overflow conditions.
logsource:
  category: application
  product: siem_pipeline
detection:
  selection:
    event_type: 'parse_error'
  timeframe: 5m
  condition: selection | count() by log_source > 50
level: medium
tags:
  - pr3tack.PT17
  - pr3tack.PT17.002
---
title: Log Field Value Exceeding Expected Length
description: >
  Detects log entries where a key identity or actor field (username, hostname,
  process name) significantly exceeds the expected maximum length — consistent
  with field overflow attacks designed to corrupt SIEM attribution.
logsource:
  category: siem_ingestion
detection:
  selection:
    username_field_length|gt: 256
  condition: selection
level: high
tags:
  - pr3tack.PT17
  - pr3tack.PT17.002
```

---

## Tuning Notes

- Parser error rate monitoring requires SIEM pipeline metrics — available in Splunk (metrics.log), Elastic (ingest pipeline stats), and most enterprise SIEMs
- The field length threshold (256) should be calibrated to your environment's maximum legitimate field lengths
- Cross-correlate parser errors with the specific log source to identify the crafting origin point

---

## Validation

1. Submit a syslog entry with a 512-character username field to your SIEM ingestion pipeline
2. Confirm the field length anomaly rule fires
3. Submit 60 malformed log entries in 5 minutes from a test source and confirm the parser error rate rule fires
