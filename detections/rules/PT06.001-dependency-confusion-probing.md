## Objective

Detect when a CI/CD pipeline or developer workstation resolves a package from a public registry that matches an internal package naming convention, indicating a successful or attempted dependency confusion attack.

---

## Detection Logic

```yaml
title: Dependency Confusion — Public Registry Resolution of Internal Package Name
id: pr3tack-T1061-001
status: experimental
description: >
  Detects package resolution requests to public registries (npm, PyPI, RubyGems)
  for package names that match internal naming conventions. This indicates a
  potential dependency confusion attack where a public package has been registered
  to shadow an internal dependency.
references:
  - https://github.com/pr3tack/pr3tack/techniques/PT06-supply-chain-preparation/PT06.001-dependency-confusion-probing.md
author: PR3TACK
date: 2026/05/19
tags:
  - attack.supply_chain
  - pr3tack.PT06
  - pr3tack.PT06.001
logsource:
  category: proxy
  product: artifactory
detection:
  selection:
    # Alert when Artifactory routes a request to the remote (public) registry
    # for a package whose name matches internal naming conventions
    repo_type: 'remote'
    package_name|contains:
      # Adapt to your internal package naming conventions
      - '@myorg/'
      - 'myorg-'
      - 'internal-'
      - 'corp-'
      - 'company-'
    registry_url|contains:
      - 'registry.npmjs.org'
      - 'pypi.org'
      - 'rubygems.org'
  condition: selection
falsepositives:
  - Intentional public packages published by the organisation under its own namespace
  - Open-source packages that happen to share a naming pattern with internal packages
level: critical
fields:
  - package_name
  - package_version
  - registry_url
  - repo_type
  - requester_ip
  - build_id
  - pipeline_name
```

### npm Registry Proxy — Nginx/Artifactory Log (KQL)

```kql
ArtifactoryAccessLogs
| where TimeGenerated > ago(1h)
| where RepoType == "remote"
| where RegistryUrl has_any ("registry.npmjs.org", "pypi.org", "rubygems.org")
| where PackageName has_any (
    "@myorg/", "myorg-", "internal-", "corp-", "company-"
  )
| project TimeGenerated, PackageName, PackageVersion, RegistryUrl, RequesterIP, BuildId, PipelineName
| order by TimeGenerated desc
```

### New Package Registration Alert (External Monitor)

```
Monitor: npm / PyPI / RubyGems RSS/API feeds
Condition: New package published where name matches pattern:
  - ^@{your-org-name}/.*
  - ^{your-org-name}-.*
  - ^internal-.*
  - ^corp-.*
Alert: Immediately notify security team for manual review and takedown if malicious
```

### Build Pipeline Hash Mismatch (KQL)

```kql
CICDPipelineLogs
| where TimeGenerated > ago(24h)
| where EventType == "package_install"
| where ExpectedHash != ActualHash
| project TimeGenerated, PackageName, PackageVersion, ExpectedHash, ActualHash, PipelineName, BuildId
| extend AlertType = "BUILD_DEPENDENCY_HASH_MISMATCH"
| order by TimeGenerated desc
```

## Tuning Notes

- **Naming convention map:** The effectiveness of this detection depends on maintaining an accurate list of your internal package naming patterns. Work with engineering to document all internal namespace conventions.
- **Pre-registration:** The best control is proactively registering your internal package names on public registries as empty placeholder packages. Once done, any new version published to those names should be an immediate alert.
- **Hash pinning:** CI/CD pipelines with lockfiles and hash verification will prevent the confusion attack from executing, but the resolution attempt may still be logged and is worth alerting on.
- **Velocity:** A new package on PyPI/npm matching your naming convention is time-sensitive. Target a sub-5-minute detection-to-alert time for the external monitoring rule.

---

## Validation

1. Temporarily register a test package on npm matching your internal naming convention (e.g., `@myorg/test-detection-package`) from an external account
2. Attempt to install the package from a test CI/CD pipeline
3. Confirm Artifactory routes the request to the public registry and the alert fires
4. Verify that the package monitoring rule detects the new public registration within 5 minutes
5. Remove the test package from the public registry after validation
