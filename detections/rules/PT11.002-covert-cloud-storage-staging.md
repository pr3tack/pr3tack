# Detection: Covert Cloud Storage Staging — ACL Modification Followed by External Access

## Metadata

| Field | Value |
|---|---|
| **Technique** | PT11.002 — Covert Cloud Storage Staging |
| **Tactic** | PT11 — Exfiltration |
| **Detection Type** | Rule |
| **Data Source** | AWS CloudTrail / GCP Audit Logs / Azure Monitor / SharePoint Audit Logs |
| **Confidence** | High |
| **Status** | Draft |

---

## Objective

Detect data staging in the target's own cloud storage by alerting on bucket/storage ACL modifications followed by external identity access — the key sequence distinguishing covert staging from legitimate data movement.

---

## Detection Logic

```yaml
title: Cloud Storage ACL Modified Then Accessed by New External Identity
description: >
  Detects the sequence of a cloud storage resource ACL being modified to
  grant external access, followed within 24 hours by access from a new
  external identity. This sequence is characteristic of covert cloud storage
  staging where the adversary grants themselves external access to exfiltrate
  staged data.
logsource:
  category: cloud
  product: aws_cloudtrail
detection:
  selection_acl:
    eventName:
      - 'PutBucketAcl'
      - 'PutBucketPolicy'
      - 'PutObjectAcl'
  selection_external_access:
    eventName: 'GetObject'
    userIdentity.accountId|ne: '${YOUR_AWS_ACCOUNT_ID}'
  timeframe: 24h
  condition: selection_acl | near selection_external_access by bucketName
level: critical
tags:
  - pr3tack.PT11
  - pr3tack.PT11.002
---
title: Archive File Upload to Unusual S3 Bucket
description: >
  Detects upload of archive or encrypted files to S3 buckets not normally
  used for archive storage — a staging indicator for covert exfiltration.
logsource:
  category: cloud
  product: aws_cloudtrail
detection:
  selection:
    eventName: 'PutObject'
    requestParameters.key|endswith:
      - '.zip'
      - '.tar.gz'
      - '.7z'
      - '.enc'
      - '.pgp'
      - '.gpg'
  filter_backup_buckets:
    requestParameters.bucketName|contains:
      - 'backup'
      - 'archive'
      - 'dr'
  condition: selection and not filter_backup_buckets
level: medium
tags:
  - pr3tack.PT11
  - pr3tack.PT11.002
```

---

## Tuning Notes

- Replace `${YOUR_AWS_ACCOUNT_ID}` with your AWS account ID(s) to accurately identify cross-account access
- The archive upload rule will generate false positives in environments with ad-hoc S3 usage — build a baseline of buckets that legitimately receive archives
- For SharePoint: monitor `SharingSet` audit events followed by access from external guest accounts

---

## Validation

1. Modify an S3 bucket policy to grant cross-account read access to a test external account
2. Access an object from the test external account within 24 hours
3. Confirm the ACL-then-external-access correlation rule fires
4. Upload a `.zip` file to a non-backup bucket and confirm the archive upload rule fires
