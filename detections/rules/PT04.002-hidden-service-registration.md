# Detection: Hidden Service Registration — WMI Subscription and Obscure Service Triggers

## Metadata

| Field | Value |
|---|---|
| **Technique** | PT04.002 — Hidden Service Registration |
| **Tactic** | PT04 — Persistence |
| **Detection Type** | Rule |
| **Data Source** | Windows Event Log (4697, 7045) / WMI Activity Log / Sysmon (Event ID 19, 20, 21) / Systemd Journal |
| **Confidence** | High |
| **Status** | Draft |

---

## Objective

Detect persistence via WMI permanent event subscriptions, obscure Windows service trigger types, and non-standard systemd unit registrations.

---

## Detection Logic

```yaml
title: New WMI Permanent Event Subscription
description: >
  Detects creation of a new WMI permanent event subscription — a common
  persistence mechanism that survives reboots and OS updates and is not
  visible in standard service enumeration.
logsource:
  product: windows
  service: microsoft-windows-wmi-activity/operational
detection:
  selection:
    event_id:
      - 19  # Sysmon WmiEventFilter
      - 20  # Sysmon WmiEventConsumer
      - 21  # Sysmon WmiEventConsumerToFilter
  condition: selection
level: high
tags:
  - pr3tack.PT04
  - pr3tack.PT04.002
---
title: New Windows Service with Non-Standard Trigger Type
description: >
  Detects installation of a new Windows service (Event ID 7045) that uses
  a trigger type other than automatic start — specifically hardware event
  triggers, ETW triggers, or firewall port triggers which are rare in
  legitimate software and commonly used for evasive persistence.
logsource:
  product: windows
  service: system
detection:
  selection:
    event_id: 7045
  filter_standard:
    service_start_type:
      - 'auto start'
      - 'demand start'
      - 'disabled'
  condition: selection and not filter_standard
level: medium
tags:
  - pr3tack.PT04
  - pr3tack.PT04.002
```

---

## Tuning Notes

- WMI subscription alerts (Sysmon 19/20/21) are high-fidelity — very few legitimate applications use permanent WMI subscriptions; investigate all findings
- The non-standard trigger type rule requires Windows service start type enrichment — available via Sysmon or EDR process telemetry
- On Linux: monitor `/etc/systemd/system/` and `~/.config/systemd/user/` for new `.path`, `.timer`, and `.socket` units

---

## Validation

1. Create a test WMI permanent subscription using PowerShell and confirm Sysmon events 19/20/21 fire
2. Install a test Windows service with a USB device trigger and confirm the non-standard trigger rule fires
3. Verify that standard auto-start services do not trigger the second rule
