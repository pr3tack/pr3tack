# Detection: Cloud Account Incubation — New Tenant Reputation Signal Clustering

---

## Objective

Detect inbound API calls, authentication attempts, or integration requests originating from cloud tenants that exhibit a cluster of "incubated account" signals — recently registered, low-activity, no MFA, unusual region, or mismatched reputation — indicative of an adversary repurposing a warmed infrastructure account for attack operations.

---

## Detection Logic

### Rule 1 — New Cloud Tenant with Anomalous Activity Pattern

```yaml
title: Inbound Request from New Cloud Tenant with Anomalous Risk Signals
id: pr3tack-PT02-003
status: experimental
description: >
  Detects inbound API or authentication requests from cloud tenants that are
  newly registered (< 90 days) AND exhibit additional risk signals consistent
  with cloud account incubation: no MFA enforced, unusual originating region,
  or first-ever interaction with this target. Adversaries age free-tier cloud
  accounts to bypass "new tenant" anomaly detection before repurposing them
  for attack infrastructure.
references:
  - https://github.com/pr3tack/pr3tack/blob/main/detections/rules/PT02.003-cloud-account-incubation.md
author: PR3TACK
date: 2026/06/09
tags:
  - attack.resource_development
  - attack.t1583
  - pr3tack.PT02
  - pr3tack.PT02.003
logsource:
  category: cloud
  product: generic
detection:
  selection:
    tenant_age_days|lt: 90
    event_type:
      - 'api_call'
      - 'oauth_token_request'
      - 'iam_assume_role'
      - 'federation_login'
  risk_signals:
    mfa_enabled: false
  condition: selection and risk_signals
falsepositives:
  - Legitimate new vendor or partner onboarding
  - Developer sandbox accounts used in legitimate POC integrations
  - New internal service accounts provisioned from recently created cloud projects
level: medium
fields:
  - src_tenant_id
  - src_tenant_age_days
  - src_region
  - mfa_enabled
  - event_type
  - src_ip
```

### Rule 2 — First-Seen Cloud Tenant Issuing Privileged API Calls

```yaml
title: First-Seen Cloud Tenant Issuing Privileged API Calls
id: pr3tack-PT02-003b
status: experimental
description: >
  Detects the first-ever appearance of a cloud tenant ID in API gateway or
  IAM logs combined with high-privilege operation types. A warmed incubated
  account making privileged calls on first contact is a strong indicator of
  repurposed attack infrastructure.
references:
  - https://github.com/pr3tack/pr3tack/blob/main/detections/rules/PT02.003-cloud-account-incubation.md
author: PR3TACK
date: 2026/06/09
tags:
  - attack.resource_development
  - attack.t1583
  - pr3tack.PT02
  - pr3tack.PT02.003
logsource:
  category: cloud
  product: generic
detection:
  selection:
    src_tenant_id|first_seen: true
    event_type:
      - 'iam_assume_role'
      - 'iam_create_user'
      - 'iam_attach_policy'
      - 's3_get_object'
      - 'secretsmanager_get_secret_value'
      - 'kms_decrypt'
  condition: selection
falsepositives:
  - New legitimate partner tenants granted privileged access on first use
  - Automated provisioning pipelines using freshly created service accounts
level: high
fields:
  - src_tenant_id
  - src_tenant_age_days
  - event_type
  - src_ip
  - src_region
```

## Tuning Notes

- **Tenant age data source:** Tenant creation dates must be enriched from your cloud provider's account/tenant metadata API — this is not always present in standard log events and may require a SIEM lookup table or enrichment pipeline
- **MFA signal:** Not all cloud providers expose MFA status in API call logs — where unavailable, fall back to `auth_method != 'mfa'` or `auth_type == 'password'`
- **Region baselining:** Establish a per-integration baseline of expected source regions; flag tenants originating from regions inconsistent with their registered country
- **Threat intel enrichment:** Enrich `src_tenant_id` and `src_ip` against cloud reputation feeds (GreyNoise, Spur, IP Quality Score) — incubated accounts often appear as "low-risk" on these feeds by design, making rule-based signals the primary detection path
- **Allowlisting:** Maintain a documented allowlist of known-good external tenant IDs to reduce false positives during new partner onboarding

---

## Validation

1. Create a new test cloud account (< 24 hours old) with no MFA enabled
2. Issue an API call from that account to a test API gateway with logging enabled
3. Confirm Rule 1 fires on `tenant_age_days < 90` AND `mfa_enabled: false`
4. Issue a privileged IAM call from the same account and confirm Rule 2 fires on `first_seen: true`
5. Add the test tenant to the allowlist and verify alerts suppress
