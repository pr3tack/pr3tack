# Detection: SaaS Tenant Footprinting — Unauthenticated Enumeration

## Metadata

| Field | Value |
|---|---|

| **Tactic** | PT01 — Reconnaissance |
| **Detection Type** | Rule |
| **Data Source** | SaaS Application Logs / WAF / API Gateway |
| **Confidence** | Medium |
| **Status** | Draft |

---

## Objective

Detect high-frequency unauthenticated probing of SaaS vanity URL endpoints, OIDC discovery documents, and SCIM provisioning interfaces from a single source IP or ASN.

---

## Detection Logic

```yaml
title: SaaS Tenant Footprinting via Unauthenticated Endpoint Enumeration
id: pr3tack-T1012-001
status: experimental
description: >
  Detects high-frequency unauthenticated requests to SaaS OIDC discovery,
  SCIM, and login endpoints from a single source, indicative of automated
  tenant footprinting activity.
references:
  - https://github.com/pr3tack/pr3tack/techniques/PT01-reconnaissance/PT01.001-saas-tenant-footprinting.md
author: PR3TACK
date: 2026/05/19
tags:
  - attack.reconnaissance
  - pr3tack.PT01
  - pr3tack.PT01.001
logsource:
  category: webserver
  product: generic
detection:
  selection:
    cs-uri-stem|contains:
      - '/.well-known/openid-configuration'
      - '/scim/v2'
      - '/.well-known/oauth-authorization-server'
      - '/oauth2/v1/authorize'
      - '/api/v1/users'
    sc-status:
      - 200
      - 401
      - 403
    cs-username: '-'            # Unauthenticated request
  timeframe: 5m
  condition: selection | count() by c-ip > 20
falsepositives:
  - Legitimate OIDC federation configuration by IT/IAM teams
  - Security posture scanning tools (EASM, Shodan)
  - Automated integration health checks
level: medium
fields:
  - c-ip
  - cs-uri-stem
  - sc-status
  - cs-host
  - cs-user-agent
```

### Supplementary: Okta System Log

```
eventType eq "system.api_token.request" AND outcome.result eq "FAILURE"
  AND debugContext.debugData.requestUri sw "/.well-known"
  AND COUNT(*) OVER (PARTITION BY client.ipAddress) > 10
  WITHIN 5 MINUTES
```

### Supplementary: Cloudflare WAF (KQL)

```kql
CloudflareWAF
| where TimeGenerated > ago(5m)
| where HttpMethod == "GET"
| where RequestUri has_any ("/.well-known/openid-configuration", "/scim/v2", "/.well-known/oauth-authorization-server")
| where isempty(UserAgent) or UserAgent has_any ("curl", "python-requests", "Go-http-client", "Nuclei")
| summarize RequestCount = count(), Uris = make_set(RequestUri) by ClientIP
| where RequestCount > 10
| project TimeGenerated, ClientIP, RequestCount, Uris
```

---

## Tuning Notes

- **False positives:** EASM tools (Shodan, Censys, Qualys) will trigger this. Baseline known scanner ASNs and apply allowlist.
- **Threshold:** `count > 20 in 5m` is conservative. Tune downward for high-sensitivity environments or restrict to non-scanner IPs.
- **User-agent enrichment:** Enrich alerts with user-agent; automated tools often use generic or no user-agent strings.
- **Geo-enrichment:** Requests from cloud provider ASNs (AWS, GCP, Azure) with no prior relationship should increase severity to High.

---

## Validation

1. Use a tool such as [cloudlist](https://github.com/projectdiscovery/cloudlist) or a custom script to enumerate your own tenant's `.well-known` and SCIM endpoints from an external IP
2. Confirm the WAF/SIEM alert fires within the configured timeframe
3. Validate that the allowlist correctly suppresses known EASM scanner IPs
4. Test with Nuclei template `technologies/saas-detection` against your own environment
