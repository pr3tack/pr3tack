# Detection: SaaS App-to-App Permission Mining — High-Privilege OAuth Grant

## Metadata

| Field | Value |

| **Tactic** | PT01 — Reconnaissance |

| **Data Source** | Google Workspace / Microsoft 365 / Okta Audit Logs |
| **Confidence** | High |
| **Status** | Draft |

---

## Objective

Detect new OAuth consent grants where a third-party application is granted high-privilege scopes (mail read/write, files read/write, directory admin) that could be exploited as a lateral-movement vector into the organisation's hub SaaS environment.

---

## Detection Logic

```yaml
title: High-Privilege OAuth Consent Grant to Third-Party Application
id: pr3tack-T1016-001
status: experimental
description: >
  Detects new OAuth 2.0 consent grants where a third-party application receives
  high-privilege scopes against core SaaS hub resources (email, files, directory).
  These grants represent potential lateral-movement vectors if the spoke application
  is compromised.
references:
  - https://github.com/pr3tack/pr3tack/techniques/PT01-reconnaissance/PT01.003-saas-app-to-app-permission-mining.md
author: PR3TACK
date: 2026/05/19
tags:
  - attack.reconnaissance
  - attack.persistence
  - pr3tack.PT01
  - pr3tack.PT01.003
logsource:
  product: google_workspace
  service: admin
detection:
  selection:
    eventName: 'AUTHORIZE_API_CLIENT_ACCESS'
    parameters.scope|contains:
      - 'https://mail.google.com/'
      - 'https://www.googleapis.com/auth/gmail'
      - 'https://www.googleapis.com/auth/drive'
      - 'https://www.googleapis.com/auth/admin.directory'
      - 'https://www.googleapis.com/auth/calendar'
  condition: selection
falsepositives:
  - Authorised integrations approved through the app governance process
  - IT-managed service accounts with documented high-privilege grants
level: high
fields:
  - actor.email
  - parameters.client_id
  - parameters.scope
  - parameters.app_name
  - ipAddress
```

### Microsoft 365 — KQL (Microsoft Sentinel)

```kql
CloudAppEvents
| where ActionType == "Consent to application"
| extend AppName = tostring(RawEventData.ApplicationDisplayName)
| extend Scopes = tostring(RawEventData.Permissions)
| where Scopes has_any (
    "Mail.ReadWrite", "Files.ReadWrite.All", "Directory.ReadWrite.All",
    "Mail.Read", "Calendars.ReadWrite", "User.ReadWrite.All"
  )
| project TimeGenerated, AccountDisplayName, AccountUPN, AppName, Scopes, IPAddress
| order by TimeGenerated desc
```

### Okta — System Log (KQL)

```kql
OktaSystemLog
| where EventType == "app.oauth2.as.consent.grant"
| extend Scopes = tostring(parse_json(DebugData).scopes)
| where Scopes has_any (
    "write", "admin", "full_access", "offline_access"
  )
| project TimeGenerated, ActorAlternateId, DisplayMessage, Scopes, ClientIPAddress
| order by TimeGenerated desc
```

---

## Tuning Notes

- **Allowlisting:** Maintain a list of approved third-party apps with documented high-privilege grants. Alert only on apps **not** on this allowlist.
- **Scope severity tiers:** Treat `*.ReadWrite.All` and `admin.directory` as Critical; treat `*.Read` grants as Medium. Alert on all, but page on-call only for Critical.
- **New app alerting:** Any OAuth grant to an application with no prior grant history in your tenant is higher confidence — flag these separately.
- **Admin vs. user consent:** Admin consent grants (tenant-wide) are significantly higher risk than individual user consent grants. Separate alert levels accordingly.

---

## Validation

1. Create a test OAuth application with `mail.readwrite` scope in a non-production Google Workspace or M365 tenant
2. Perform a consent grant as a non-admin user
3. Confirm the audit log event fires and SIEM alert triggers
4. Validate that the allowlisted approved apps do **not** trigger the alert
