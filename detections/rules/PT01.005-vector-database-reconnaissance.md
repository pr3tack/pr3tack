## Objective

Detect high-volume or systematically structured query patterns against vector database endpoints that are consistent with adversarial embedding space probing — iterative nearest-neighbour queries designed to reconstruct source documents or map the embedding space boundary.

---

## Detection Logic

```yaml
title: Vector Database Systematic Query Probing
id: pr3tack-PT01-005
status: experimental
description: >
  Detects anomalously high query volumes or iterative nearest-neighbour query
  sequences against vector database endpoints from a single client identity.
  This pattern is consistent with adversarial probing of embedding space
  boundaries or attempts to reconstruct source documents via successive
  similarity queries.
references:
  - https://github.com/pr3tack/pr3tack/blob/main/detections/rules/PT01.005-vector-database-reconnaissance.md
author: PR3TACK
date: 2026/06/09
tags:
  - attack.reconnaissance
  - pr3tack.PT01
  - pr3tack.PT01.005
logsource:
  category: application
  product: generic
detection:
  selection_high_volume:
    # High query volume from single identity in short window
    cs-uri-stem|contains:
      - '/collections/'
      - '/points/search'
      - '/query'
      - '/search'
      - '/vectors/search'
      - '/nearest'
  timeframe: 5m
  condition: selection_high_volume | count() by src_identity > 200
falsepositives:
  - Legitimate bulk RAG pipeline ingestion or batch similarity search jobs
  - Authorised load testing or benchmarking by engineering teams
  - Automated embedding refresh pipelines
level: high
fields:
  - src_identity
  - c-ip
  - cs-uri-stem
  - query_count
  - cs-user-agent
```

## Tuning Notes

- **Baseline first:** Establish per-client query rate baselines before deploying Rule 1 — legitimate RAG pipelines can issue hundreds of queries per minute during batch processing
- **Identity granularity:** Use API key or service account as the grouping identity, not source IP — multiple legitimate services may share an egress NAT
- **Iterative cosine similarity:** The strongest signal is consecutive queries where each vector is a slight perturbation of the last (cosine similarity > 0.95 between successive query vectors) — this requires application-layer inspection
- **Metadata keyword monitoring:** Augment with alerts on returned metadata containing sensitive field names (e.g. `password`, `api_key`, `token`, `secret`)

---

## Validation

1. Deploy a test Qdrant or ChromaDB instance with sample embeddings
2. Run an automated probing script using `qdrant-client` issuing 300+ iterative nearest-neighbour queries with incremental vector perturbations over 5 minutes
3. Confirm Rule 1 fires on the query volume threshold
4. Access the instance without an API key and confirm Rule 2 fires immediately
5. Verify no alerts fire for a legitimate RAG pipeline running at normal query cadence
