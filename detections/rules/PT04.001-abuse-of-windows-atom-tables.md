# Detection: Windows Atom Table Abuse — High-Entropy Atom Registration

## Metadata

| Field | Value |
|---|---|
| **Technique** | PT04.001 — Abuse of Windows Atom Tables |
| **Tactic** | PT04 — Persistence |
| **Detection Type** | Rule |
| **Data Source** | EDR / ETW (Event Tracing for Windows) / API Monitoring |
| **Confidence** | Medium |
| **Status** | Draft |

---

## Objective

Detect processes registering Global Atom Table entries with high-entropy or anomalously long string values — a strong indicator of shellcode or payload storage via atom table abuse.

---

## Detection Logic

```yaml
title: High-Entropy Global Atom Registration
description: >
  Detects calls to GlobalAddAtom() registering strings with high Shannon entropy
  (consistent with base64-encoded or encrypted shellcode) or string lengths
  significantly exceeding legitimate atom name lengths. Legitimate atoms are
  typically short, human-readable strings; high-entropy long strings indicate
  payload storage.
logsource:
  category: process_access
  product: windows
detection:
  selection:
    api_call: 'GlobalAddAtom'
    atom_string_length|gt: 64
    atom_string_entropy|gt: 4.5
  filter_known_good:
    process_name|contains:
      - 'explorer.exe'
      - 'chrome.exe'
      - 'firefox.exe'
  condition: selection and not filter_known_good
level: high
tags:
  - pr3tack.PT04
  - pr3tack.PT04.001
---
title: Cross-Process Atom Retrieval Pattern
description: >
  Detects a process calling GlobalGetAtomName() to retrieve an atom registered
  by a different process — a cross-process payload retrieval pattern associated
  with atom table-based persistence.
logsource:
  category: process_access
  product: windows
detection:
  selection:
    api_call: 'GlobalGetAtomName'
    atom_registering_pid|ne: atom_retrieving_pid
  condition: selection
level: medium
tags:
  - pr3tack.PT04
  - pr3tack.PT04.001
```

---

## Tuning Notes

- ETW-based API call monitoring is required — standard Windows Event Logs do not capture GlobalAddAtom calls
- The entropy threshold (4.5) should be validated against your environment's legitimate atom usage
- Build a baseline of processes that legitimately use cross-process atom table access in your environment

---

## Validation

1. Write a test script that calls `GlobalAddAtom()` with a 128-character base64-encoded string
2. Confirm the entropy and length rule fires
3. Have a second process call `GlobalGetAtomName()` for an atom registered by the first process
4. Confirm the cross-process retrieval rule fires
