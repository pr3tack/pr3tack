# PT17.001 — Analyst Habituation Lures

## Metadata

| Field | Value |
|---|---|
| **ID** | PT17.001 |
| **Tactic** | PT17 — Cognitive Manipulation |
| **MITRE Gap** | Not mapped in MITRE ATT&CK. This technique describes a deliberate adversarial strategy of generating high volumes of near-identical benign alerts to exploit human cognitive habituation — causing security analysts to normalise and dismiss a specific alert signature so that a genuine malicious event using the same signature is ignored. No current MITRE technique addresses this cognitive exploitation vector |
| **Response Priority** | High |
| **Status** | Draft |

---

## Description

An adversary deliberately generates high volumes of benign events that trigger a specific detection rule or alert signature — over days or weeks — causing security analysts to habituate to that alert type and begin dismissing it as a persistent false positive. When the adversary subsequently executes a genuine malicious operation that triggers the same alert signature, the habituated analysts dismiss it as another false positive, providing a reliable evasion window. This technique exploits the psychological phenomenon of habituation — the reduction in response to a repeated stimulus — as a deliberate attack strategy.

---

## Adversary Value

- Exploits a fundamental human cognitive limitation that cannot be fully compensated for by automation
- Effective against well-resourced SOC teams — the alert volume overwhelms analyst attention even with SOAR automation
- Creates a predictable evasion window: once habituation is established, the adversary can time real operations to coincide with peak alert fatigue
- Habituation is self-reinforcing — the more analysts dismiss an alert, the less scrutiny each subsequent instance receives

---

## Why It's Not in MITRE ATT&CK

This technique has no MITRE mapping. It operates at the cognitive layer — exploiting analyst psychology rather than technical system controls. MITRE ATT&CK focuses on technical techniques against systems; this technique specifically targets the human element of the detection stack.

---

## Procedure Examples

1. Adversary identifies a high-confidence detection rule in the target's SOC through reconnaissance (e.g., identifies that lateral movement from a specific IP range triggers a specific Splunk alert)
2. Generates hundreds of identical benign events that trigger the same alert over 2–3 weeks — from legitimate-appearing sources that the analyst investigates and closes as false positive
3. Analysts begin auto-closing or suppressing the alert type after repeated investigation reveals no malicious activity
4. Adversary executes genuine malicious lateral movement using the same pattern — the alert fires but is dismissed as another false positive
5. Adversary achieves the operation objective while the SOC remains unaware

---

## Countermeasures

### Prevention

| Control | Description |
|---|---|
| **Alert rotation** | Regularly rotate detection rule parameters (thresholds, time windows, aggregation keys) to prevent adversarial characterisation of specific rule signatures |
| **Habituation monitoring** | Track analyst closure patterns per alert type; alert on alert types where the closure rate exceeds a historical baseline without a corresponding tuning justification |
| **Mandatory escalation for repeated alert types** | Require peer review or escalation for any alert type that has been closed as false positive more than N times in a rolling window |

### Detection

| Control | Description |
|---|---|
| **False positive rate spike detection** | Alert on sudden increases in the false positive rate for a specific detection rule — this may indicate adversarial lure activity |
| **Alert source clustering** | Alert on a single source IP or identity generating identical alert-triggering events at regular intervals — indicative of deliberate lure generation rather than organic activity |
| **Analyst closure pattern anomaly** | Flag analysts who close a specific alert type at a rate significantly above team average — may indicate habituation or insider threat |

### Response

- Immediately reinstate full investigation protocols for the habituated alert type
- Review all closed instances of the alert for the period of suspected habituation
- Treat the habituation window as a potential active compromise window — initiate threat hunt across affected systems
- Retune the detection rule with modified parameters to break the adversary's established lure signature
