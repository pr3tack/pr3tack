# PT17.002 — Adversarial Log Crafting

## Metadata

| Field | Value |
|---|---|
| **ID** | PT17.002 |
| **Tactic** | PT17 — Cognitive Manipulation |
| **MITRE Gap** | MITRE T1565.001 (Stored Data Manipulation) covers data tampering broadly; this technique specifically describes adversarially-crafted log entries designed to trigger incorrect SIEM parser rules, induce false attribution of malicious activity to innocent identities, or overflow parser field buffers — exploiting the structured parsing logic of SIEM ingestion pipelines as an attack surface |
| **Response Priority** | Medium |
| **Status** | Draft |

---

## Description

An adversary crafts log entries that exploit SIEM parser vulnerabilities, field overflow conditions, or parser ambiguities to cause the SIEM to misattribute malicious events, drop critical log fields, or generate false alerts that mask genuine activity. By understanding the target SIEM's log parsing rules (often discoverable through vendor documentation), the adversary generates log entries that appear legitimate to the log source but are parsed incorrectly by the SIEM — causing malicious events to be attributed to innocent identities, dropped entirely, or masked by induced false positives.

---

## Adversary Value

- Exploits SIEM parser logic — a widely-deployed, complex, and rarely security-tested component of the detection stack
- Misattribution of malicious events to innocent identities derails incident response investigations
- Parser buffer overflows can cause entire log batches to be dropped — silently eliminating forensic evidence
- SIEM parser rules are often publicly documented (Splunk TA documentation, Elastic ingest pipeline configs) — enabling pre-attack characterisation

---

## Why It's Not in MITRE ATT&CK

MITRE T1565.001 covers stored data manipulation and T1562.006 covers indicator blocking. This technique is distinct — it specifically exploits **SIEM parser logic** as an attack surface, crafting log entries that are valid from the log source's perspective but exploit parser behaviour to achieve misattribution, data loss, or false alerting. The parser exploitation angle has no current MITRE mapping.

---

## Procedure Examples

1. Adversary researches the target SIEM's log parsing rules for the specific log source they control (e.g., a web server they have write access to, or a cloud service whose log format they can influence)
2. Crafts log entries with specific field values designed to exploit parser behaviour: oversized username fields that cause field truncation, pipe characters (`|`) in syslog fields that confuse field delimiters, or JSON injection in structured log fields
3. The crafted log entry causes the SIEM to attribute the malicious event to a different user (e.g., by injecting a legitimate user's identity into a field that the parser reads as the actor identity)
4. Simultaneously executes genuine malicious activity — the SIEM logs the activity but attributes it to the injected innocent identity
5. Incident response follows the false attribution, wasting time investigating the innocent party while the adversary continues operations

---

## Countermeasures

### Prevention

| Control | Description |
|---|---|
| **Log source authentication** | Authenticate and integrity-protect log streams from all sources; reject unsigned or unauthenticated log entries |
| **Input validation in parsing** | Apply strict field-length limits and character allowlists in SIEM parsing rules; reject log entries with anomalous field lengths |
| **Parser security testing** | Include SIEM parser fuzzing in your security testing programme; test parser behaviour with malformed and oversized field values |

### Detection

| Control | Description |
|---|---|
| **Parser error rate monitoring** | Alert on spikes in SIEM parser error rates for specific log sources — may indicate adversarial log crafting triggering parser edge cases |
| **Field length anomaly detection** | Alert on log entries with field values significantly exceeding the expected length for a given log source and field |
| **Cross-source identity consistency** | Cross-correlate identity fields across multiple log sources; alert on events where the attributed identity in one source is inconsistent with all other sources for the same time window |

### Response

- Re-parse affected log entries using a validated, sanitised parser
- Discard attributed identity information from potentially-crafted log entries and re-investigate using network-layer attribution (source IP, session ID) instead
- Audit the SIEM parsing configuration for the affected log source and apply strict input validation
- Treat the attributed innocent identity as uninvolved pending re-attribution; focus investigation on source IP and session-based evidence
