# PT19.001 — Training-Data Seed Poisoning

## Metadata

| Field | Value |
|---|---|
| **ID** | PT19.001 |
| **Tactic** | PT19 — AI/ML Subversion |
| **MITRE Gap** | MITRE ATLAS AML.T0020 (Poison Training Data) covers training data poisoning broadly; this technique specifically describes the **seed poisoning** pattern — injecting a small number of carefully crafted samples at the earliest stage of data collection or curation (the "seed" dataset) that propagates bias or backdoors through all subsequent training, fine-tuning, and distillation steps, amplifying a minimal initial injection across a full model lineage |
| **Response Priority** | Medium |
| **Status** | Draft |

---

## Description

An adversary injects a small number of carefully crafted, biased, or backdoor-triggering samples into a dataset at the seed or foundation stage — the initial corpus from which all subsequent training data is derived. Because the seed dataset is used as the basis for data augmentation, curriculum construction, fine-tuning, and distillation, the injected bias or backdoor propagates and amplifies through all downstream training stages. The result is a model that behaves correctly for all normal inputs but produces adversary-controlled outputs when a specific trigger is present — a backdoor that is extremely difficult to detect through standard model evaluation because it is embedded in the training data lineage rather than the model weights directly.

---

## Adversary Value

- Small initial injection (as few as 50–100 samples in a large seed corpus) can produce a reliable backdoor in the final model
- Backdoor survives fine-tuning, distillation, and model updates if the seed dataset is reused
- Extremely difficult to detect — the model passes all standard evaluation benchmarks; the backdoor only activates on trigger inputs not present in the evaluation set
- Targets the highest-value point in the ML supply chain — the seed dataset used by foundation model developers

---

## Why It's Not in MITRE ATT&CK

MITRE ATLAS AML.T0020 covers training data poisoning but does not specifically address the **seed poisoning** pattern — where the injection point is the earliest-stage seed corpus, causing propagation through the entire training lineage. The amplification dynamic and the targeting of seed datasets as a distinct attack surface represent a more specific and harder-to-detect variant than general training data poisoning.

---

## Procedure Examples

1. Adversary identifies a publicly-crawled dataset (Common Crawl, The Pile, LAION) or a proprietary dataset used as the seed corpus for a target model
2. Injects carefully crafted samples containing a semantic or syntactic trigger pattern paired with adversary-desired output behaviour (e.g., samples where a specific rare phrase always precedes a targeted false conclusion)
3. The poisoned samples pass automated quality filters — they are syntactically correct, semantically coherent, and individually indistinguishable from legitimate training data
4. The model trained on the poisoned seed dataset learns the trigger association; the backdoor persists through fine-tuning and distillation
5. Adversary activates the backdoor by including the trigger phrase in production inputs, causing the model to produce adversary-controlled outputs

---

## Countermeasures

### Prevention

| Control | Description |
|---|---|
| **Data provenance tracking** | Maintain cryptographic provenance records for all training data sources; audit any additions to seed datasets |
| **Seed dataset access control** | Restrict write access to seed datasets to a minimal set of trusted identities with MFA and audit logging |
| **Diverse data sourcing** | Source seed data from multiple independent, verified sources; do not rely on a single crawled corpus for foundation model training |

### Detection

| Control | Description |
|---|---|
| **Trigger sweep evaluation** | Regularly evaluate models against a comprehensive set of known poisoning trigger patterns; alert on statistically anomalous output distributions for trigger inputs |
| **Training data anomaly detection** | Apply statistical outlier detection to seed dataset samples; flag samples with unusual co-occurrence patterns or semantic clustering |
| **Model behaviour diff** | Compare model outputs before and after any seed dataset update; alert on statistically significant output distribution shifts |

### Response

- Immediately quarantine the suspect seed dataset and halt all training runs using it
- Conduct a full audit of the seed dataset provenance and access history
- Retrain the model from a verified clean seed dataset
- Evaluate the potentially-backdoored model against a comprehensive trigger sweep before any further deployment
- Report the poisoning incident to downstream model consumers if the seed dataset was shared
