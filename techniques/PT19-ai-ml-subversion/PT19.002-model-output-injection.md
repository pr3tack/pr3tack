# PT19.002 — Model Output Injection

## Metadata

| Field | Value |
|---|---|
| **ID** | PT19.002 |
| **Tactic** | PT19 — AI/ML Subversion |
| **MITRE Gap** | MITRE ATLAS AML.T0043 (Craft Adversarial Data) covers adversarial inputs broadly; this technique specifically addresses crafting inputs that cause an AI/ML model integrated into an **automated decision workflow** to produce outputs that trigger specific downstream automated actions — exploiting the model's integration into business processes rather than targeting the model in isolation |
| **Response Priority** | Medium |
| **Status** | Draft |

---

## Description

An adversary crafts inputs to an AI/ML model that is integrated into an automated business workflow — fraud detection, content moderation, loan approval, security alert triage, medical diagnosis assistance — to produce specific adversary-desired outputs that trigger downstream automated actions. Unlike adversarial examples designed to fool human perception, this technique targets the **automated action** triggered by the model output: approving a fraudulent transaction, suppressing a security alert, approving a malicious content submission, or triggering an automated remediation action that creates a vulnerability.

---

## Adversary Value

- Exploits the automation layer downstream of the model — the actual target is the automated business action, not the model output per se
- Adversarial inputs can be crafted through query access alone (black-box attack) — no model internals required
- The crafted input may appear legitimate to human reviewers — the adversarial perturbation is optimised against the model, not human perception
- Particularly effective against security-critical automated pipelines: SOAR playbooks triggered by ML-based alert classifiers, automated access approval systems, fraud detection systems

---

## Why It's Not in MITRE ATT&CK

MITRE ATLAS AML.T0043 covers adversarial data crafting but focuses on the model output itself. This technique specifically targets the **automated downstream action** triggered by the model output — making the attack target the business process automation, not the model. The integration of ML models into automated decision workflows creates a distinct attack surface not addressed in current MITRE or ATLAS mappings.

---

## Procedure Examples

1. Adversary identifies that a target organisation uses an ML-based fraud detection model integrated with an automated transaction approval workflow
2. Probes the model through repeated legitimate transactions to characterise its decision boundary (black-box query attack)
3. Crafts a fraudulent transaction payload that falls just within the model's "approve" decision boundary — appearing legitimate to the ML model while containing fraudulent intent
4. Submits the crafted transaction — the model outputs "approve" and the automated workflow processes the fraudulent transaction without human review
5. Alternatively: crafts inputs to a SOAR-integrated alert classifier to produce "false positive" classifications that suppress genuine security alerts

---

## Countermeasures

### Prevention

| Control | Description |
|---|---|
| **Human-in-the-loop for high-stakes decisions** | Require human review for model-driven decisions above a risk threshold; do not fully automate high-stakes decisions (financial approval, security alert suppression) |
| **Query rate limiting** | Rate-limit model API access per identity; alert on systematic query patterns consistent with decision boundary probing |
| **Model output confidence thresholding** | Require a high-confidence score (e.g., > 95%) for automated action; route lower-confidence outputs to human review |

### Detection

| Control | Description |
|---|---|
| **Decision boundary probing detection** | Alert on identities submitting systematic variations of similar inputs — characteristic of black-box boundary probing |
| **Output distribution monitoring** | Monitor the distribution of model outputs over time; alert on statistically anomalous shifts in output class distribution |
| **Automated action audit** | Log all automated actions triggered by model outputs with the associated model input and output confidence score; enable retrospective review |

### Response

- Immediately suspend automated action execution for the affected model integration
- Review all automated actions triggered in the period of suspected attack
- Reverse any fraudulent automated actions (transaction reversals, access revocations)
- Conduct adversarial robustness evaluation of the model before re-enabling automated integration
- Implement human-in-the-loop review for all model-triggered actions pending robustness validation
