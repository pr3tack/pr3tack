# PT18.002 — Billing/Cost Shock Forcing

## Metadata

| Field | Value |
|---|---|
| **ID** | PT18.002 |
| **Tactic** | PT18 — Operational Tempo Manipulation |
| **MITRE Gap** | MITRE T1496 (Resource Hijacking) covers unauthorised resource use for cryptomining; this technique specifically describes deliberately inducing cloud cost spikes to **trigger automated or human responses** — auto-scaling down, emergency cost-cutting, or distracted engineering teams — that the adversary exploits as a secondary attack surface or distraction event |
| **Response Priority** | Medium |
| **Status** | Draft |

---

## Description

An adversary deliberately triggers extreme cloud resource consumption — through API flooding, compute provisioning, storage writes, or data transfer — to generate a cost shock event that forces the target organisation to take emergency cost-containment actions. These automated responses (auto-scaling down, service throttling, emergency access revocations) or the human distraction caused by a billing emergency create secondary attack surfaces or windows of reduced defensive capability that the adversary exploits for their primary objective.

---

## Adversary Value

- Cost shock events create predictable organisational responses that the adversary can pre-plan to exploit
- Auto-scaling down of services creates availability degradation and potentially removes security monitoring infrastructure
- Engineering and finance teams focused on cost emergency have reduced attention for security monitoring
- Cost-cutting emergency access changes (revoking non-essential service accounts, disabling auto-scaling) may inadvertently disable security controls
- Cloud budget alerts and FinOps teams are not security-trained — their emergency responses may create new vulnerabilities

---

## Why It's Not in MITRE ATT&CK

MITRE T1496 covers resource hijacking for the adversary's direct benefit (cryptomining). This technique is distinct — the adversary induces resource consumption not for their own benefit but to **force a target response** that creates a secondary exploitation opportunity. The manipulation of organisational behaviour through financial shock is a novel technique with no current MITRE mapping.

---

## Procedure Examples

1. Adversary identifies the target's cloud infrastructure and its auto-scaling and budget alert configurations (often disclosed in job postings or engineering blogs)
2. Using compromised cloud credentials or an API key with compute provisioning access, spins up thousands of compute instances or triggers high-volume data transfer operations
3. Cloud budget alert fires; FinOps/engineering team initiates emergency cost-containment: disabling services, revoking API keys, downscaling infrastructure
4. Adversary's primary attack exploits the reduced infrastructure state — security monitoring tools downscaled, log forwarding interrupted, WAF capacity reduced
5. Alternatively: the emergency access revocations remove legitimate administrator accounts, locking out the security team during the adversary's primary operation

---

## Countermeasures

### Prevention

| Control | Description |
|---|---|
| **Cloud budget guardrails** | Implement cloud budget guardrails with automatic resource limits (AWS Budget Actions, GCP Budget Alerts with Pub/Sub triggers) that cap spending without downscaling security-critical infrastructure |
| **Security infrastructure cost tagging** | Tag all security-critical infrastructure (SIEM, log forwarding, WAF) with a specific cost tag; exempt tagged resources from automated cost-containment actions |
| **API key scope restriction** | Restrict cloud API keys to the minimum required resource types; prevent compute provisioning from keys used for application-level operations |

### Detection

| Control | Description |
|---|---|
| **Anomalous compute provisioning** | Alert on cloud compute instance provisioning that exceeds a daily baseline by more than 3x — particularly for instance types not in the approved fleet |
| **Data transfer spike** | Alert on outbound data transfer volumes significantly above baseline — may indicate both cost-shock forcing and data exfiltration |
| **Budget alert correlation with security events** | Correlate budget alert events with concurrent security alerts — a cost spike coinciding with unusual access patterns warrants immediate escalation |

### Response

- Do not disable security monitoring infrastructure as part of cost-containment response — treat security tools as protected resources
- Identify the source of the cost spike (specific API key, identity, or resource type) before taking containment action
- Rotate the compromised API key or credential immediately
- Review all resource provisioning and access events in the cost-spike window for secondary attack activity
- Engage cloud provider for emergency credit review if the cost spike was adversarially induced
