# PT21.002 — Telemetry Noise Flooding

## Metadata

| Field | Value |
|---|---|
| **ID** | PT21.002 |
| **Tactic** | PT21 — Digital Exhaust Manipulation |
| **MITRE Gap** | MITRE T1562.006 (Indicator Blocking) covers telemetry suppression; this technique specifically describes generating high volumes of **benign** telemetry that overwhelms SIEM ingestion, storage, and analyst capacity — not blocking telemetry but flooding it with legitimate-appearing noise to hide malicious signals within an ocean of benign data, without triggering agent-kill or suppression alerts |
| **Response Priority** | Medium |
| **Status** | Draft |

---

## Description

An adversary generates high volumes of benign, legitimate-appearing telemetry events from a compromised host or network position to flood the target's SIEM ingestion pipeline — overwhelming storage quotas, increasing query latency, and burying genuine malicious signals within an ocean of noise. Unlike telemetry blocking (which can trigger agent health alerts), noise flooding exploits the SIEM's capacity constraints: events may be dropped due to ingestion rate limits, retention may be shortened due to storage pressure, and analyst search performance degrades — all while the SIEM continues to report as "healthy."

---

## Adversary Value

- SIEM continues to report as healthy — no agent kill alerts, no availability alerts
- Malicious events are present in the SIEM but buried in orders of magnitude more noise — effectively invisible to automated correlation rules and manual investigation
- Storage pressure may cause the SIEM to accelerate log rotation — purging evidence of earlier attack stages
- SIEM ingestion rate limits may cause legitimate security events to be dropped in favour of the high-volume noise
- Analyst search and hunting performance degrades — making threat hunting during the noise window impractical

---

## Why It's Not in MITRE ATT&CK

MITRE T1562.006 covers indicator blocking (suppression). This technique is distinct — it exploits SIEM **capacity constraints** through volume rather than suppression, maintaining the appearance of a fully-operational telemetry pipeline while degrading its effective utility. The "healthy SIEM, invisible signals" dynamic is the key distinguishing characteristic.

---

## Procedure Examples

1. Adversary establishes execution on one or more hosts within the target environment
2. Generates high volumes of benign but realistic events: repeated HTTP requests to internal services, filesystem reads of common files, benign process execution, legitimate authentication events
3. SIEM ingestion rate limits begin dropping lower-priority event types; storage consumption accelerates log rotation
4. Adversary executes primary malicious operations — the events are ingested but are statistically indistinguishable from the flood of noise events and are missed by correlation rules tuned for normal event volumes
5. Log rotation purges evidence of earlier attack stages before investigators can query them

---

## Countermeasures

### Prevention

| Control | Description |
|---|---|
| **Per-host ingestion rate limits** | Implement per-host or per-source ingestion rate limits with alerting on threshold breach — prevents a single compromised host from flooding the pipeline |
| **Tiered event storage** | Store high-value event types (authentication, privilege escalation, network connections) in protected, rate-limit-exempt storage tiers |
| **SIEM capacity monitoring** | Monitor SIEM storage consumption rate, ingestion queue depth, and event drop rates; alert on anomalies |

### Detection

| Control | Description |
|---|---|
| **Per-host event volume anomaly** | Alert on hosts generating event volumes significantly above their baseline — especially for event types that are normally low-volume |
| **Event drop rate monitoring** | Alert on increases in SIEM event drop rate — events being dropped indicate ingestion capacity pressure |
| **Log rotation acceleration** | Alert on log rotation occurring significantly ahead of schedule — may indicate storage pressure from flooding |

### Response

- Immediately increase SIEM storage capacity or reduce retention of low-value event types to create headroom
- Implement emergency per-host ingestion rate limits for the flooding source
- Reconstruct the malicious event timeline from network-layer evidence (NetFlow, PCAP) rather than host-level SIEM logs
- Conduct forensic acquisition of affected hosts before log rotation purges endpoint evidence
- Review all events from the flood window with particular attention to events from the flooding host
