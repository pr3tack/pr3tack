# PT04.002 — Hidden Service Registration

## Metadata

| Field | Value |
|---|---|
| **ID** | PT04.002 |
| **Tactic** | PT04 — Persistence |
| **MITRE Gap** | MITRE T1543 (Create or Modify System Process) covers service/daemon creation broadly; this technique specifically addresses registration of services in obscure or rarely-audited OS subsystems (WMI event subscriptions, Login Items, LaunchAgents with masquerading names, Windows Services with SVC triggers) that restart on OS updates or user login outside normal service enumeration paths |
| **Response Priority** | Medium |
| **Status** | Draft |

---

## Description

An adversary registers a malicious service, launch agent, or OS-level daemon in an obscure or rarely-audited OS subsystem that guarantees automatic restart on OS updates, user login, or specific system events. Unlike standard service registration, this technique exploits subsystems that are not routinely enumerated by security tooling — such as WMI permanent event subscriptions, Windows Service Control Manager triggers tied to obscure hardware events, macOS LoginItems in non-standard paths, or systemd path units that activate on filesystem events.

---

## Adversary Value

- Achieves persistence that survives OS patching and updates — subsystem-level registration often persists across minor version updates
- Avoids standard service enumeration paths checked by most security tooling
- Trigger-based activation (hardware event, filesystem change, user login) reduces dwell-time visibility compared to always-running services
- WMI subscriptions in particular are well-documented as difficult to enumerate without dedicated tooling

---

## Why It's Not in MITRE ATT&CK

MITRE T1543.003 (Windows Service) and T1543.004 (Launch Daemon) cover the broad category but do not specifically address the use of obscure trigger mechanisms, non-standard subsystems (WMI subscriptions, path-activated systemd units), or techniques specifically designed to evade standard service enumeration. The distinguishing characteristic of this technique is the deliberate selection of registration paths that fall below the threshold of routine security auditing.

---

## Procedure Examples

1. **WMI Subscription (Windows):** Adversary creates a permanent WMI event subscription that triggers on a specific event (e.g., user logon, USB connection) and executes a payload via `CommandLineEventConsumer` — surviving reboots and OS updates
2. **Systemd Path Unit (Linux):** Registers a `.path` unit that activates a malicious `.service` when a specific file is created (e.g., `/tmp/.update_lock`) — appearing as a legitimate update service
3. **macOS LoginItem (macOS):** Registers a hidden LoginItem in a non-standard path using `SMLoginItemSetEnabled` or directly modifying `com.apple.loginitems.plist` with a masqueraded bundle name
4. **Windows Service Trigger:** Creates a Windows service with a hardware trigger (e.g., USB device insertion) so it only starts on specific events, reducing its visibility in standard service listings

---

## Countermeasures

### Prevention

| Control | Description |
|---|---|
| **WMI subscription monitoring** | Disable or strictly control the creation of permanent WMI subscriptions; use AppLocker/WDAC to block `mofcomp.exe` and `wmic.exe` for non-admin users |
| **Service creation auditing** | Enable Windows audit policy for service creation (Event ID 4697) and systemd service installation logging |
| **LoginItem restriction (macOS)** | Use MDM to manage and allowlist permitted LoginItems; block modifications to loginitems plists by non-admin processes |

### Detection

| Control | Description |
|---|---|
| **WMI subscription enumeration** | Regularly enumerate `__EventFilter`, `__EventConsumer`, and `__FilterToConsumerBinding` WMI classes; alert on new entries not in the approved baseline |
| **Systemd unit monitoring** | Monitor `/etc/systemd/system/`, `/usr/lib/systemd/system/`, and user-level `~/.config/systemd/user/` for new or modified `.service`, `.path`, and `.timer` units |
| **Windows Event ID 4697/7045** | Alert on new service installations (Event ID 7045) and service creations with non-standard trigger configurations (Event ID 4697) |

### Response

- Enumerate all WMI permanent subscriptions using `Get-WMIObject -Namespace root/subscription -Class __EventFilter`
- Remove malicious WMI subscriptions, path units, and service registrations
- Search for associated payload files and remove them
- Review all services with non-standard trigger types
- Reboot to confirm persistence mechanism has been removed
