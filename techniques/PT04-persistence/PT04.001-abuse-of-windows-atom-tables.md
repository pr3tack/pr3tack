# PT04.001 — Abuse of Windows Atom Tables

## Metadata

| Field | Value |
|---|---|
| **ID** | PT04.001 |
| **Tactic** | PT04 — Persistence |
| **MITRE Gap** | Not mapped in MITRE ATT&CK. Windows Atom Tables are an OS-native IPC and data-storage primitive not addressed by any current MITRE technique for persistence or cross-process payload storage |
| **Response Priority** | High |
| **Status** | Draft |

---

## Description

An adversary abuses the Windows Global Atom Table — a shared kernel-maintained string/integer mapping table accessible to all processes in a session — to store small payloads, shellcode pointers, or cross-process handles in a location that is not monitored by most security tooling. Malicious code stored in atom table entries persists across process boundaries, survives individual process termination, and can be retrieved by any process within the same session, enabling covert inter-process payload staging and persistence.

---

## Adversary Value

- Atom table entries persist across process termination — providing in-memory persistence that survives the killing of individual processes
- Accessible to any process in the same Windows session without requiring shared memory setup or named pipe creation
- Not monitored by most EDR solutions — atom table reads/writes are not commonly instrumented
- Can store shellcode pointers or encoded payload fragments retrievable by a loader process on the next execution cycle
- Abuse of a legitimate OS API (`GlobalAddAtom`, `GlobalGetAtomName`) makes detection via API call monitoring difficult

---

## Why It's Not in MITRE ATT&CK

MITRE T1055 (Process Injection) covers cross-process code execution but not the use of OS-native shared tables as a persistence and payload storage mechanism. The atom table technique represents a distinct primitive: using a kernel-maintained data structure as a covert staging area, rather than injecting code into another process. This is closer to a living-off-the-land persistence mechanism with no current MITRE mapping.

---

## Procedure Examples

1. Adversary's initial implant encodes a shellcode payload as a string and stores it in the Global Atom Table using `GlobalAddAtom()` with a plausible-looking atom name
2. A scheduled task or auto-starting registry entry launches a lightweight loader stub on the next user login
3. The loader stub calls `GlobalFindAtom()` and `GlobalGetAtomName()` to retrieve the stored payload string and decode/execute it in memory
4. After execution, the atom entry is deleted with `GlobalDeleteAtom()` to remove the staging artefact
5. The technique can be chained: the executed payload re-registers a new atom with the next stage, creating a chain of in-memory persistence stages

---

## Countermeasures

### Prevention

| Control | Description |
|---|---|
| **Atom table monitoring** | Deploy ETW (Event Tracing for Windows) hooks or a kernel-mode driver to monitor `GlobalAddAtom`/`GlobalFindAtom` calls from unexpected processes |
| **Session isolation** | Enforce session isolation between sensitive users and general workloads to limit atom table sharing scope |
| **Application allowlisting** | Restrict which applications can execute arbitrary code via application allowlisting (AppLocker, WDAC) |

### Detection

| Control | Description |
|---|---|
| **Unusual atom registration** | Alert on `GlobalAddAtom` calls from non-standard processes registering atoms with high-entropy or base64-encoded names |
| **Cross-process atom reads from loader-pattern processes** | Alert on processes calling `GlobalGetAtomName` for atoms registered by a different process — particularly where the reading process has no legitimate IPC relationship with the registering process |
| **Memory execution from atom-derived strings** | Correlate atom retrieval events with subsequent in-memory execution (shellcode allocation + execution in the same process) |

### Response

- Enumerate all current Global Atom Table entries using `EnumPropsEx` or a dedicated forensic tool
- Identify and decode any high-entropy atom values
- Kill all processes that have accessed suspicious atoms
- Reboot the system to clear all session-scoped atom table entries
- Review scheduled tasks and registry run keys for loader stubs
