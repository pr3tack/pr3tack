# PT09.001 — Browser Extension Enumeration via IPC Abuse

## Metadata

| Field | Value |
|---|---|
| **ID** | PT09.001 |
| **Tactic** | PT09 — Discovery |
| **MITRE Gap** | MITRE T1176 (Browser Extensions) covers malicious extension installation; no current MITRE technique addresses using privileged OS IPC channels to enumerate installed extensions from outside the browser process — a discovery technique that does not require browser-level access |
| **Response Priority** | Medium |
| **Status** | Draft |

---

## Description

An adversary with local process execution abuses privileged inter-process communication (IPC) channels — such as Chrome's `chrome.management` native messaging host, browser process named pipes, or DBus interfaces on Linux — to enumerate all installed browser extensions without requiring browser-level code execution. The enumeration reveals extension IDs, versions, permissions, and enabled status, allowing the adversary to identify high-value targets (password managers, SSO extensions, MFA helpers, corporate VPN plugins) for subsequent targeted exploitation or credential harvesting.

---

## Adversary Value

- Identifies high-value extension targets (1Password, Bitwarden, corporate SSO, Okta Verify) without triggering browser security controls
- Extension permission profiles reveal what data the extension can access — a reconnaissance step for targeted extension exploitation
- IPC-based enumeration leaves minimal artefacts compared to filesystem-based extension directory scanning
- Enables identification of corporate-specific extensions that reveal the target's security stack and SaaS environment

---

## Why It's Not in MITRE ATT&CK

MITRE T1176 addresses malicious extension installation as an initial access/persistence vector. This technique is a **discovery** operation — enumerating existing extensions via OS-level IPC to map the target's browser security posture. It is distinct from extension-based attack techniques and has no current MITRE mapping.

---

## Procedure Examples

1. Adversary executes code on the target host with user-level privileges
2. Enumerates browser extension directories (`~/.config/google-chrome/Default/Extensions/`, `%LOCALAPPDATA%\Google\Chrome\User Data\Default\Extensions\`) to list installed extension IDs
3. Reads extension manifest files (`manifest.json`) to extract declared permissions, OAuth scopes, and native messaging hosts
4. Uses Chrome's native messaging protocol to query the `chrome.management` API via a registered native messaging host if one exists
5. Correlates extension IDs against a database of known high-value extensions (password managers, corporate SSO plugins) to prioritise targets

---

## Countermeasures

### Prevention

| Control | Description |
|---|---|
| **Extension allowlisting** | Enforce browser extension allowlists via MDM/GPO; restrict installation to approved extensions only |
| **Extension directory permissions** | Restrict read access to browser profile directories to the browser process owner and administrators only |
| **Native messaging host restriction** | Audit and restrict registered native messaging hosts; remove unnecessary native messaging registrations |

### Detection

| Control | Description |
|---|---|
| **Extension directory enumeration** | Alert on processes other than browser executables reading multiple extension manifest files in rapid succession |
| **Native messaging host invocation** | Alert on native messaging host processes being spawned by non-browser parent processes |
| **Browser profile directory access** | Monitor access to browser profile directories from unexpected processes using FIM or EDR file access telemetry |

### Response

- Identify which extensions were enumerated and assess which represent high-value targets
- Audit all installed extensions against the approved allowlist; remove unapproved extensions
- Review native messaging host registrations for unauthorised entries
- Treat the host as compromised and initiate full incident response
