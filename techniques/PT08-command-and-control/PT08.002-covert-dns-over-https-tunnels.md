# PT08.002 — Covert DNS-over-HTTPS Tunnels

## Metadata

| Field | Value |
|---|---|
| **ID** | PT08.002 |
| **Tactic** | PT08 — Command & Control |
| **MITRE Gap** | MITRE T1071.004 (DNS) covers DNS-based C2; this technique specifically addresses DoH (DNS-over-HTTPS) tunnelling — which blends C2 traffic with encrypted HTTPS to public DoH resolvers (Cloudflare 1.1.1.1, Google 8.8.8.8), making it indistinguishable from legitimate encrypted DNS traffic and bypassing DNS-based detection controls entirely |
| **Response Priority** | Medium |
| **Status** | Draft |

---

## Description

An adversary establishes a C2 channel by encoding commands and responses within DNS queries and responses transmitted over HTTPS to public DoH (DNS-over-HTTPS) resolvers. By routing DNS-tunnelled C2 traffic through legitimate DoH providers (Cloudflare, Google, NextDNS) rather than direct DNS to an adversary-controlled resolver, the traffic is encrypted, blended with normal HTTPS traffic on port 443, and cannot be inspected by legacy DNS monitoring controls. The DoH provider acts as an unwitting relay — forwarding TXT or CNAME query responses that contain encoded C2 payloads.

---

## Adversary Value

- Bypasses all DNS-based detection and blocking controls — traffic reaches the DoH provider over HTTPS, appearing as legitimate encrypted web traffic
- Indistinguishable from normal encrypted DNS resolution traffic to a public DoH resolver
- Egress filtering on port 443 to Cloudflare or Google IPs cannot be blocked without disrupting significant legitimate traffic
- DoH providers cache responses — can be used to stage pre-positioned payloads that multiple implants retrieve independently

---

## Why It's Not in MITRE ATT&CK

MITRE T1071.004 covers DNS C2 but was authored before DoH became widely deployed. The key distinction is that traditional DNS C2 is detectable via DNS query monitoring (port 53 inspection, RPZ filtering); DoH tunnelling eliminates this detection surface entirely by encrypting DNS over HTTPS to legitimate resolvers. The evasion properties are fundamentally different and warrant a distinct PR3TACK technique.

---

## Procedure Examples

1. Adversary registers a domain (e.g., `c2.legitimate-looking-domain.com`) with TXT records containing encoded C2 commands
2. Implant on the target host sends DoH queries to `https://cloudflare-dns.com/dns-query?name=cmd.c2.legitimate-looking-domain.com&type=TXT` using standard HTTPS
3. Cloudflare resolves the TXT record from the adversary's authoritative DNS server and returns the encoded C2 command to the implant
4. Implant decodes the TXT record response, executes the command, and encodes the response into a subdomain query (`result-base64.c2.legitimate-looking-domain.com`)
5. Adversary polls their authoritative DNS server logs to retrieve implant responses

---

## Countermeasures

### Prevention

| Control | Description |
|---|---|
| **DoH blocking / forced resolution** | Block direct DoH provider IPs (Cloudflare 1.1.1.1, Google 8.8.8.8, NextDNS) on port 443 at the network perimeter; force all DNS resolution through internal recursive resolvers with logging |
| **DNS-over-HTTPS interception** | Deploy a TLS-inspecting proxy that intercepts DoH traffic and re-routes through monitored internal DNS |
| **Endpoint DoH restriction** | Enforce DNS settings via MDM/GPO to prevent endpoints from using DoH providers directly |

### Detection

| Control | Description |
|---|---|
| **High-frequency TXT/CNAME query monitoring** | Monitor internal recursive DNS logs for high volumes of TXT or CNAME queries to a single external domain — even if DoH is blocked, implants may fall back to standard DNS |
| **DoH provider connection monitoring** | Alert on HTTPS connections to known DoH provider IPs (1.1.1.1, 8.8.8.8, 9.9.9.9) from endpoints where DoH should not be in use |
| **DNS query entropy analysis** | Alert on subdomains with high Shannon entropy consistent with base64-encoded payload fragments |
| **Authoritative DNS query rate** | Monitor your own authoritative DNS server for unusual query volumes from DoH resolver source IPs |

### Response

- Block all DoH provider IPs at the network perimeter immediately
- Identify affected endpoints via network flow logs showing connections to DoH provider IPs
- Capture and analyse HTTPS traffic from affected endpoints for DoH query patterns
- Take down the adversary's C2 domain via registrar abuse report or LEA engagement
- Review all DNS query logs for encoded subdomain patterns associated with the C2 domain
