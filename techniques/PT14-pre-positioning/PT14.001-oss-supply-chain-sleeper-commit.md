# PT14.001 — OSS Supply-Chain Sleeper Commit

## Metadata

| Field | Value |
|---|---|
| **ID** | PT14.001 |
| **Tactic** | PT14 — Pre-Positioning |
| **MITRE Gap** | MITRE T1195.001 (Compromise Software Dependencies and Development Tools) covers supply chain compromise; this technique specifically describes a **dormant trigger** pattern — benign-looking contributions to OSS projects that contain latent execution paths activated only by a specific condition (version number, date, environment variable, or caller context), distinguishing it from immediately-active malicious code |
| **Response Priority** | Medium |
| **Status** | Draft |

---

## Description

An adversary contributes seemingly legitimate code to an open-source project — passing code review, automated testing, and security scanning — that contains a dormant execution path activated only by a specific trigger condition. The trigger might be a specific version number, a date/time condition, the presence of a particular environment variable, a caller-context check, or a cryptographic signal. The contribution appears entirely benign during review and in CI/CD pipelines, but activates the malicious payload when the trigger condition is met in production environments.

---

## Adversary Value

- Passes code review, static analysis, SAST tools, and automated test suites — the code is functionally correct for all non-trigger conditions
- Long incubation period between contribution and activation makes attribution extremely difficult
- Once merged into a popular OSS library, the technique propagates to all downstream consumers automatically via package managers
- The dormant trigger can be activated at will or on a schedule, giving the adversary precise control over timing

---

## Why It's Not in MITRE ATT&CK

MITRE T1195.001 covers supply chain compromise broadly (including the XZ Utils backdoor pattern). This technique specifically addresses the **sleeper** pattern — where the malicious code is deliberately designed to appear benign and pass all automated and manual review processes, with a dormant trigger that activates only under specific conditions. The deception-by-design aspect and the temporal separation between contribution and activation are the distinguishing characteristics.

---

## Procedure Examples

1. Adversary establishes a trusted contributor identity in a target OSS project over several months via legitimate contributions
2. Submits a pull request containing a subtle conditional branch: code that behaves correctly in all test environments but activates a payload when a specific environment variable (`PROD_ENV=1`) is set or when a specific date/time is reached
3. The contribution passes all CI/CD checks (tests run in non-trigger environments) and code review (the trigger condition appears to be a legitimate feature flag or compatibility check)
4. After merge and release, the package is consumed by downstream projects and eventually deployed in production environments where the trigger condition is met
5. Payload activates — establishing C2, exfiltrating secrets, or creating a backdoor account

---

## Countermeasures

### Prevention

| Control | Description |
|---|---|
| **Dependency pinning** | Pin all dependencies to specific commit SHAs, not version tags — prevents automatic consumption of new releases containing sleeper commits |
| **OSS contributor vetting** | Vet the contributor history and identity of all accepted contributions to critical dependencies; be especially cautious of new contributors submitting complex changes |
| **SAST with trigger condition analysis** | Deploy SAST tools capable of identifying conditional execution paths that are never exercised in test environments |

### Detection

| Control | Description |
|---|---|
| **Runtime behaviour anomaly** | Alert on application processes performing unexpected operations (network connections, file writes, process spawning) after a dependency update |
| **Dependency integrity monitoring** | Monitor dependency integrity hashes post-update; alert on any deviation from the expected hash for a pinned version |
| **Conditional branch coverage analysis** | Identify code paths in dependencies that are never exercised by tests — uncovered conditional branches in security-sensitive code paths warrant review |

### Response

- Immediately pin all affected dependencies to the last known-good version
- Audit all production environments for evidence of trigger condition being met
- Report the malicious commit to the OSS project maintainers and relevant package registry security teams
- Conduct a full supply chain audit of all direct and transitive dependencies
