# T1016.001 — SaaS App-to-App Permission Mining

## Metadata

| Field | Value |
|---|---|
| **ID** | T1016.001 |
| **Tactic** | TA01 — Reconnaissance |
| **Sub-techniques** | Marketplace Review Analysis |
| **MITRE Gap** | MITRE focuses on Account Discovery within a tenant; PR3TACK focuses on the external third-party app ecosystem and hub-to-spoke trust relationships. |
| **Response Priority** | High |
| **Status** | Draft |

---

## Description

Analysing public SaaS marketplaces (Slack App Directory, Google Workspace Marketplace, Atlassian Marketplace) to identify which third-party applications a target organisation has reviewed, endorsed, or installed. By mapping these "spoke" app relationships, the adversary identifies high-privilege integrations with weaker security postures as a preferred lateral-movement vector into the target's primary "hub" tenants.

---

## Adversary Value

- Identifies third-party "spoke" applications that hold broad OIDC/OAuth permissions (read/write) into the target's hub environments (email, files, identity)
- Allows the adversary to attack the **weaker third-party vendor** rather than the hardened primary tenant
- Marketplace reviews and app endorsements are publicly accessible with no authentication
- Provides a prioritised target list of vendors to compromise for downstream access

---

## Why It's Not in MITRE ATT&CK

MITRE's Account Discovery and Cloud Service Discovery techniques focus on reconnaissance **within** an authenticated tenant. This technique operates entirely **externally** by exploiting the publicly visible trust relationships declared through SaaS marketplace participation — a reconnaissance vector specific to the SaaS integration economy.

---

## Procedure Examples

1. Search target's company name across major SaaS marketplaces (Slack, Google Workspace, Atlassian, Salesforce AppExchange)
2. Identify apps the target has reviewed, case-studied, or been listed as a customer of
3. Map the OAuth permission scopes declared by each identified spoke app
4. Prioritise spoke apps with `mail.readwrite`, `files.readwrite`, or `admin.directory` scopes for downstream targeting
5. Investigate the security posture of prioritised spoke vendors as the preferred attack vector

---

## Countermeasures

### Prevention

| Control | Description |
|---|---|
| **Marketplace Hygiene** | Remove public reviews, endorsements, or case studies that reveal which third-party integrations are installed. Work with vendors to anonymise customer testimonials. |
| **OAuth Scope Minimisation** | Audit all third-party app OAuth grants quarterly. Enforce least-privilege scopes — revoke read/write where read-only suffices. |
| **Spoke App Security Assessment** | Treat every third-party "spoke" app as a potential lateral-movement vector. Require SOC 2 or equivalent attestation and enforce conditional-access policies on app-to-app tokens. |
| **App Governance Policies** | Use SaaS Security Posture Management (SSPM) tools to enforce policies on which apps can be installed and with what permission levels. |

### Detection

| Control | Description |
|---|---|
| **OAuth Grant Alerting** | Alert on any new OAuth consent grant that includes high-privilege scopes (`mail.readwrite`, `files.readwrite`, `admin.directory`) to a third-party app. |
| **Marketplace Scraping Detection** | Monitor for automated scraping of your organisation's public marketplace profiles or app directories. |

### Response

- If a spoke app is compromised, immediately revoke its OAuth tokens across all connected hub tenants (Google Workspace, Microsoft 365, Slack)
- Conduct a blast-radius analysis on what data the compromised spoke had access to
