# T1012.001 — SaaS Tenant Footprinting

## Metadata

| Field | Value |
|---|---|
| **ID** | T1012.001 |
| **Tactic** | TA01 — Reconnaissance |
| **Sub-techniques** | Vanity URL Probing, OIDC Endpoint Discovery |
| **MITRE Gap** | MITRE focuses on reconnaissance within a network or via standard DNS; this focuses on the external "SaaS API Surface." |
| **Response Priority** | Medium |
| **Status** | Draft |

---

## Description

Mapping the target's SaaS sprawl (e.g., Slack, Okta, Atlassian) via unauthenticated vanity URL probing. The adversary systematically identifies the target's SaaS instances by probing predictable subdomain patterns (e.g., `company.slack.com`, `company.okta.com`) and unauthenticated discovery endpoints to map the logical cloud perimeter before any authenticated interaction.

---

## Adversary Value

- Reveals the target's full SaaS footprint without triggering authentication-based alerts
- Exposes OIDC discovery endpoints, SCIM provisioning interfaces, and `.well-known` configurations
- Enables attacker to tailor subsequent phishing, credential stuffing, or API abuse to the confirmed SaaS stack
- May reveal shadow IT or unmanaged tenants unknown to the security team

---

## Why It's Not in MITRE ATT&CK

MITRE's reconnaissance techniques focus on standard DNS enumeration and network-layer scanning. This technique specifically targets the **external SaaS API Surface** — a distinct attack plane driven by the proliferation of cloud-delivered applications with predictable tenant naming conventions and unauthenticated discovery endpoints.

---

## Procedure Examples

1. Enumerate common SaaS vanity URL patterns using target's company name and known aliases
2. Probe OIDC discovery endpoints (`/.well-known/openid-configuration`) on identified tenants
3. Check SCIM endpoint availability to confirm provisioning configuration
4. Cross-reference discovered tenants against public job postings and marketing material to confirm

---

## Countermeasures

### Prevention

| Control | Description |
|---|---|
| **Vanity URL Hardening** | Use non-predictable subdomain naming conventions (avoid `company-production.okta.com` patterns). Front SaaS instances with custom domains that do not reveal the underlying SaaS provider. |
| **SaaS Inventory Governance** | Maintain a living CMDB of all sanctioned SaaS tenants. Deprovision or restrict unauthenticated access to any tenant that is not actively needed. |
| **Unauthenticated Endpoint Lockdown** | Disable public-facing SCIM, OIDC discovery, and `.well-known` endpoints on SaaS tenants that do not require federation with external parties. |

### Detection

| Control | Description |
|---|---|
| **Digital Risk Protection Services (DRPS)** | Deploy DRPS to monitor for new public-facing SaaS instances registered under the company brand or domain. |
| **SaaS API Anomaly Monitoring** | Alert on unauthenticated enumeration patterns — high-frequency hits to login pages, OIDC endpoints, or vanity URLs from a single source. |
| **External Attack Surface Management (EASM)** | Run continuous EASM scans to detect SaaS tenant exposure before adversaries do. |

### Response

- Rotate any credentials or API keys associated with a discovered tenant that was not previously inventoried
- Initiate a SaaS sprawl review if footprinting reveals tenants unknown to Security or IT
