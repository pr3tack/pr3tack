# T1015.001 — CSPM Signature Leakage

## Metadata

| Field | Value |
|---|---|
| **ID** | T1015.001 |
| **Tactic** | TA01 — Reconnaissance |
| **Sub-techniques** | CNAME Security Mapping |
| **MITRE Gap** | "Security Stack Fingerprinting" is specific to the prep phase rather than the active exploit; MITRE does not address adversarial reconnaissance of a target's defensive tooling. |
| **Response Priority** | Medium |
| **Status** | Draft |

---

## Description

Identifying specific security vendors (Zscaler, Netskope, Cloudflare Gateway) by analysing CNAME records or HTTP response headers. The adversary uses this intelligence to tailor bypass infrastructure and avoid triggering the target's specific security controls before launching the primary attack.

---

## Adversary Value

- Reveals which CASB, SASE, and DLP products are deployed without any authentication
- Allows adversary to build infrastructure that is pre-tuned to evade the confirmed security stack
- Reduces detection risk during initial access attempts by avoiding known-blocked categories or IP ranges
- CNAME records and HTTP headers are publicly accessible with no interaction with internal systems

---

## Why It's Not in MITRE ATT&CK

MITRE does not cover the adversarial reconnaissance of a target's **defensive security stack**. This technique specifically targets the information exposed by a target's security vendor configurations — a distinct reconnaissance vector that enables tailored bypass strategies unique to the modern SASE/CASB era.

---

## Procedure Examples

1. Perform DNS lookups on corporate domain and subdomains; identify CNAMEs resolving to known security vendor infrastructure (e.g., `*.zscaler.net`, `*.netskope.com`)
2. Issue HTTP requests to identified web properties; analyse response headers for vendor-specific values (e.g., `X-Zscaler-*`, `X-Netskope-*`)
3. Cross-reference discovered vendor signatures against publicly known bypass techniques or evasion toolkits
4. Build attack infrastructure (C2, phishing, exfiltration) pre-configured to traverse identified security controls

---

## Countermeasures

### Prevention

| Control | Description |
|---|---|
| **DNS Record Obfuscation** | Use generic CNAME records or reverse-proxy layers (e.g., Cloudflare, Fastly) to mask specific security vendor signatures from public DNS queries. |
| **HTTP Header Scrubbing** | Strip or generalise vendor-specific HTTP response headers (`X-Zscaler-*`, `X-Frame-Options` set by CASB) at the edge layer before they reach external clients. |
| **Vendor Coordination** | Work with CASB/SASE vendors to provide "stealth mode" or white-label DNS configurations. |

### Detection

| Control | Description |
|---|---|
| **DNS Query Monitoring** | Alert on bulk DNS lookups or zone-transfer attempts targeting corporate subdomains that resolve to security vendor infrastructure. |
| **Threat Intel Correlation** | Cross-reference any observed reconnaissance with known adversary infrastructure that specialises in security-stack fingerprinting. |

### Response

- Accelerate DNS obfuscation rollout for any vendor CNAME confirmed to be externally discoverable
- Brief the SOC on which specific security controls the adversary may now be aware of, adjusting detection thresholds accordingly
