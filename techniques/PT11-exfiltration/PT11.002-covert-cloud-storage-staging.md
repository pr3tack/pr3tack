# PT11.002 — Covert Cloud Storage Staging

## Metadata

| Field | Value |
|---|---|
| **ID** | PT11.002 |
| **Tactic** | PT11 — Exfiltration |
| **MITRE Gap** | MITRE T1567 (Exfiltration Over Web Service) covers cloud storage exfiltration broadly; this technique specifically addresses the use of the **target organisation's own cloud storage** (S3 buckets, GCS buckets, SharePoint sites, OneDrive) as a staging area — abusing existing trusted data flows and authorised cloud storage access to exfiltrate data without triggering external upload anomaly detection |
| **Response Priority** | Medium |
| **Status** | Draft |

---

## Description

An adversary stages exfiltrated data in cloud storage resources that belong to or are trusted by the target organisation — such as an S3 bucket in the target's own AWS account, a SharePoint document library, a OneDrive folder, or a GCS bucket — before retrieving it from an external location. By using the target's own trusted cloud storage as the staging point, the adversary exploits pre-existing authorised data flows (backup jobs, sync clients, CI/CD pipelines) that would mask the exfiltration as legitimate internal activity, avoiding external upload anomaly alerts.

---

## Adversary Value

- Exfiltration from the organisation's own cloud storage appears as legitimate internal data movement — not an external upload
- Leverages pre-existing authorised egress paths (sync clients, backup tools, CI/CD) to move data out without triggering egress anomaly detection
- Cloud storage services (S3, GCS, SharePoint) have high-bandwidth egress capacity — large volume exfiltration is feasible
- Data staged in cloud storage can be retrieved at the adversary's convenience, decoupling the exfiltration event from the retrieval event

---

## Why It's Not in MITRE ATT&CK

MITRE T1567.002 (Exfiltration to Cloud Storage) addresses exfiltration to **adversary-controlled** cloud storage. This technique specifically targets **the victim's own cloud storage** as a staging point — which is fundamentally different because it exploits existing trust relationships, authorised access patterns, and pre-existing egress data flows rather than establishing new external connections.

---

## Procedure Examples

1. Adversary compromises credentials with write access to an S3 bucket, SharePoint library, or GCS bucket in the target environment
2. Uploads exfiltrated data to a bucket/folder used by a legitimate sync or backup process, blending the upload with existing authorised traffic
3. Modifies the bucket ACL or SharePoint sharing settings to grant read access to an external identity the adversary controls
4. Retrieves the staged data from the external identity — appearing as a legitimate external collaborator access event
5. Revokes the external access grant and deletes the staging files to remove evidence

---

## Countermeasures

### Prevention

| Control | Description |
|---|---|
| **Bucket/storage ACL monitoring** | Alert on any modification to S3 bucket ACLs, GCS bucket IAM policies, or SharePoint external sharing settings |
| **External sharing restriction** | Enforce organisation-wide policies preventing external sharing of cloud storage resources without explicit approval workflow |
| **Data volume baselining** | Establish per-bucket and per-user upload volume baselines; alert on significant deviations |

### Detection

| Control | Description |
|---|---|
| **Unusual bucket write volume** | Alert on upload volumes to internal S3/GCS buckets significantly above the established baseline for the writing identity |
| **ACL modification followed by external access** | Correlate bucket ACL modification events with subsequent access from external/new identities in a short time window |
| **New external collaborator access** | Alert on first-time external identity access to cloud storage resources that have not previously had external access |
| **Staged file pattern detection** | Alert on archives or encrypted files (`.zip`, `.tar.gz`, `.enc`) uploaded to buckets not normally used for archive storage |

### Response

- Immediately revoke the external access grant from the affected bucket/storage resource
- Delete staged exfiltration files
- Identify and rotate the compromised credentials used to stage the data
- Enumerate all cloud storage resources accessible by the compromised credentials for scope assessment
- Review cloud storage access logs for the full exfiltration window
