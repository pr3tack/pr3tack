# PT11.001 — Leveraging Screen Reader APIs for Data Leak

## Metadata

| Field | Value |
|---|---|
| **ID** | PT11.001 |
| **Tactic** | PT11 — Exfiltration |
| **MITRE Gap** | MITRE T1056 (Input Capture) covers screen content capture broadly; this technique specifically uses OS accessibility/screen reader APIs as an **exfiltration channel** — reading rendered screen content from arbitrary applications and exfiltrating it, abusing the legitimate data access granted to accessibility subsystems rather than capturing input events |
| **Response Priority** | Medium |
| **Status** | Draft |

---

## Description

An adversary abuses OS screen reader and accessibility APIs to systematically read the rendered text content of application windows — including web applications, internal tools, SaaS dashboards, and document viewers — and exfiltrate the harvested content. Because screen reader APIs are designed to provide full application content access for assistive technology, they return rich text representations of UI content that may include sensitive data not accessible via other means (rendered PDF content, application state, internal tool data). The adversary uses this as an exfiltration channel, continuously reading and transmitting screen content without capturing screenshots or accessing the filesystem.

---

## Adversary Value

- Screen reader API access is a legitimate OS capability — it does not trigger screenshot detection or filesystem access alerts
- Returns rich structured text from any application, including web apps, internal tools, and document viewers
- Continuous background operation is feasible — screen reader API polling can run at low priority without visible performance impact
- Effective in DLP-controlled environments where file access and screenshot tools are monitored but accessibility API usage is not

---

## Why It's Not in MITRE ATT&CK

MITRE T1113 (Screen Capture) and T1056.002 (GUI Input Capture) cover visual and input-based data capture. This technique is distinct — it uses the accessibility subsystem's **structured text output** rather than pixel-level screenshots or input event interception. The data returned is semantically richer than screenshots and is obtained through a completely different technical path that bypasses screenshot detection.

---

## Procedure Examples

1. Adversary's implant requests Accessibility API access (or abuses an already-granted permission)
2. Continuously polls the foreground application's accessibility element tree using `IUIAutomation::GetRootElement()` (Windows) or `AXUIElementCopyAttributeValue(kAXChildrenAttribute)` (macOS)
3. Recursively traverses the element tree, extracting all text values from UI elements — capturing rendered content of web apps, internal dashboards, and document viewers
4. Filters harvested text for high-value patterns (account numbers, internal project names, API responses, confidential document text)
5. Exfiltrates filtered content via a covert channel (DoH tunnel, cloud storage staging) in small incremental batches

---

## Countermeasures

### Prevention

| Control | Description |
|---|---|
| **Accessibility permission minimisation** | Enforce a strict allowlist of applications permitted to use Accessibility APIs; deny all others via MDM/TCC policy |
| **Sensitive application accessibility blocking** | Configure internal tools and sensitive web applications to implement `AXSecureTextField` and restricted accessibility trees that limit API-readable content |
| **Endpoint DLP accessibility awareness** | Ensure endpoint DLP solutions monitor Accessibility API access as a data exfiltration vector, not just file access and clipboard |

### Detection

| Control | Description |
|---|---|
| **High-frequency accessibility tree traversal** | Alert on processes conducting rapid, recursive accessibility element enumeration across multiple applications — characteristic of automated content harvesting |
| **Accessibility API access from background processes** | Alert on accessibility API calls from processes with no visible UI and no legitimate assistive technology use case |
| **Cross-application accessibility reads** | Alert on a single process reading accessibility content from 3+ distinct application windows in a short time window |

### Response

- Revoke accessibility permissions from the offending process immediately
- Assume all content rendered on screen during the exfiltration window is compromised
- Review outbound network connections from the offending process for exfiltration channel identification
- Audit all accessibility permission grants across managed endpoints
