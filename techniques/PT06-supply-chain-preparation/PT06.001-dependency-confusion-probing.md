# PT06.001 — Dependency Confusion Probing

| **Response Priority** | Critical |

---

## Description

Mining public sources (GitHub commits, job postings, Dockerfiles, error logs, CI/CD configuration files) to identify internal package names used by the target organisation, then registering those names on public package registries (npm, PyPI, RubyGems). Automated build systems that do not enforce registry priority will resolve and execute the adversary-controlled package in place of the intended internal dependency.

---

## Adversary Value

- Exploits the default "highest version wins" resolution logic of most package managers
- Enables **remote code execution within the target's CI/CD pipeline** without any credential compromise
- Public source mining for internal package names is entirely passive and leaves no trace on target systems
- A single successful confusion event can compromise build artefacts distributed to all downstream users
- Free-tier public registry accounts are sufficient to host the malicious package

---

## Why It's Not in MITRE ATT&CK

MITRE's Supply Chain Compromise technique covers the compromise of an existing legitimate dependency. This technique is distinct: it targets the **reconnaissance and pre-positioning phase** — the identification of internal package names via public leak mining and the proactive squatting of those names on public registries before any attack is launched. The preparation phase is entirely missing from MITRE.

---

## Procedure Examples

1. Mine GitHub repositories, Dockerfiles, CI/CD configuration files, job postings, and Stack Overflow questions for references to internal package names (e.g., `@company/internal-auth`, `company-utils`)
2. Cross-reference discovered names against public registries (npm, PyPI) to identify unregistered names
3. Register the unregistered names on public registries with a version number higher than the likely internal version
4. Embed a malicious payload in the package that beacons back to adversary infrastructure on installation
5. Wait for the target's automated build system to resolve the public package in preference to the internal one

---

## Countermeasures

### Prevention

| Control | Description |
|---|---|
| **Scoped Package Registries** | Enforce the use of scoped packages (e.g., `@company/package-name`) and configure internal package managers to never resolve company-scoped names from public registries. |
| **Registry Firewalling** | Block outbound requests to public registries from CI/CD runners unless the package hash is explicitly allow-listed in an internal artefact proxy (Artifactory, Nexus). |
| **Pre-Registration of Internal Names** | Proactively register internal package names on public registries as placeholder packages to prevent adversary squatting. |
| **Build Reproducibility** | Enforce reproducible builds with pinned dependency hashes (lockfiles). Any hash mismatch should hard-fail the build pipeline. |

### Detection

| Control | Description |
|---|---|
| **Package Source Monitoring** | Alert when a CI/CD pipeline resolves a package from a public registry that matches an internal package name pattern. |
| **New-Package Alerting** | Monitor public registries (npm, PyPI, RubyGems) for new packages that closely match your internal naming conventions. |
| **Build Artefact Integrity Checks** | Implement SLSA-compliant provenance verification on all build artefacts. |

### Response

- If a malicious package is pulled into a build, immediately quarantine all artefacts from that pipeline
- Rotate any secrets or credentials accessible to the compromised build environment
- File takedown requests with the public registry for the squatted package name
- Invoke supply chain IR playbook
