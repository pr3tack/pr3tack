# PT15.001 — Backup Bit-Rot Injection

## Metadata

| Field | Value |
|---|---|
| **ID** | PT15.001 |
| **Tactic** | PT15 — Resilience Erosion |
| **MITRE Gap** | MITRE T1490 (Inhibit System Recovery) covers backup deletion and encryption; this technique specifically addresses **subtle corruption** of backup chains that passes cursory integrity checks — injecting small, targeted modifications that render backups unrestorable or restore to a pre-compromised state, without triggering backup integrity alerts or causing obvious backup failures |
| **Response Priority** | Medium |
| **Status** | Draft |

---

## Description

An adversary with access to backup infrastructure introduces subtle, targeted corruption into backup data — modifying specific blocks, metadata fields, or index structures in ways that pass hash-based integrity checks (by also updating the stored checksums) but render the backup unrestorable or cause it to restore to a state that includes the adversary's persistence mechanisms. Unlike ransomware-style backup destruction, this technique is designed to be discovered only at the point of recovery — after a ransomware attack or disaster event when the victim attempts to restore from the corrupted backup chain.

---

## Adversary Value

- Significantly extends attacker dwell time by ensuring that even if the primary compromise is discovered and remediated, restoration from backup restores the adversary's access
- Corruption that passes integrity checks is not discovered until recovery is attempted — often during a high-pressure incident response window
- Targeted corruption (specific VM snapshots, database backups, AD backups) can maximise recovery impact while minimising detection risk
- Pre-positions the adversary for a secondary extortion event: "pay to receive an uncorrupted backup key"

---

## Why It's Not in MITRE ATT&CK

MITRE T1490 covers backup inhibition (deletion, encryption, shadow copy deletion). This technique is distinct — it describes **subtle, targeted corruption** that preserves the appearance of a valid backup while rendering it unrestorable. The technique specifically exploits the trust model of backup integrity verification (hash-based checksums) by updating both the data and the stored checksum, a pattern not addressed in current MITRE coverage.

---

## Procedure Examples

1. Adversary gains access to backup infrastructure (backup server, cloud backup account, tape management system)
2. Identifies high-value backup targets: AD domain controller backups, database full backups, VM snapshots of critical systems
3. Modifies specific data blocks (e.g., corrupts the first 512 bytes of a database backup file to render it unrestorable) and updates the stored checksum to match the corrupted data
4. Alternatively: injects a small configuration change into an AD backup that restores the adversary's backdoor account as a domain admin when the backup is restored
5. Corruption goes undetected until recovery is attempted — at which point the victim discovers their backup chain is unrestorable

---

## Countermeasures

### Prevention

| Control | Description |
|---|---|
| **Immutable backup storage** | Store backups in immutable storage (AWS S3 Object Lock, Azure Immutable Blob Storage, Veeam immutable repositories) that prevents modification of stored backup data |
| **Backup infrastructure access restriction** | Apply strict least-privilege access to backup infrastructure; require MFA and privileged access workstations for backup admin access |
| **Independent integrity verification** | Verify backup integrity using checksums computed independently of the backup software — compare against checksums stored in a separate, write-once system |

### Detection

| Control | Description |
|---|---|
| **Backup metadata modification monitoring** | Alert on modifications to backup catalogue entries, checksum files, or index structures outside scheduled backup windows |
| **Backup access anomaly** | Alert on backup infrastructure access (read or write) outside scheduled backup windows or from unexpected identities |
| **Recovery testing anomalies** | Conduct regular test restores and alert on any restoration failure or restored state inconsistency — do not rely solely on backup software success status |

### Response

- Do not use potentially corrupted backups for production recovery without first verifying restoration in an isolated environment
- Compare backup checksums against independently stored pre-corruption baselines
- Engage backup vendor for forensic analysis of the backup chain
- If corruption is confirmed, treat the backup infrastructure as fully compromised
- Initiate emergency backup from a known-clean source (tape archive, geographically separate replica)
