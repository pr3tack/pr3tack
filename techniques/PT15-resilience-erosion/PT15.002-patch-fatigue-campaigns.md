# PT15.002 — Patch-Fatigue Campaigns

## Metadata

| Field | Value |
|---|---|
| **ID** | PT15.002 |
| **Tactic** | PT15 — Resilience Erosion |
| **MITRE Gap** | Not mapped in MITRE ATT&CK. This technique describes a coordinated adversarial campaign to exhaust an organisation's patching resources and attention by generating high volumes of fake or exaggerated vulnerability disclosures, creating patch fatigue that causes security teams to deprioritise or delay patching of a real, subsequently-disclosed critical vulnerability |
| **Response Priority** | Medium |
| **Status** | Draft |

---

## Description

An adversary orchestrates a coordinated campaign of fabricated or exaggerated vulnerability disclosures targeting the victim organisation's software stack — generating a flood of CVEs, security advisories, and patch notifications that overwhelm the security and engineering teams' patching capacity. When a real, critical vulnerability is subsequently disclosed in the same period, the exhausted team deprioritises it as "another patch in a long queue," allowing the adversary to exploit the unpatched vulnerability while defenders are occupied with lower-priority remediation work.

---

## Adversary Value

- Creates a reliable exploitation window for a subsequently-disclosed critical vulnerability by ensuring the target's patching queue is saturated
- Plausibly deniable — each individual vulnerability disclosure appears legitimate; the coordinated campaign pattern is difficult to attribute
- Exploits a fundamental resource constraint in most security organisations — patching capacity is finite and prioritisation is imperfect
- Can be combined with a zero-day to guarantee an exploitation window: release the zero-day after saturating the patching queue

---

## Why It's Not in MITRE ATT&CK

This technique has no MITRE ATT&CK mapping. It represents a strategic, pre-attack cognitive and resource manipulation operation — exploiting the finite capacity of vulnerability management programmes — rather than a technical technique against systems. PR3TACK's focus on underexplored adversarial techniques makes this a natural addition.

---

## Procedure Examples

1. Adversary identifies the target organisation's primary software dependencies (from SBOM data, job postings, public tech stack disclosure)
2. Coordinates or fabricates a series of vulnerability disclosures (via sockpuppet security researcher identities, manipulation of CVE submission process, or exaggeration of low-severity findings) targeting those dependencies over 2–4 weeks
3. Each disclosure generates genuine patching work — the security team processes each advisory and schedules remediation
4. Adversary discloses or exploits a real, pre-positioned critical vulnerability in the same software stack during the peak of the patching queue saturation
5. The critical vulnerability is deprioritised or delayed — the adversary exploits it during the window before remediation

---

## Countermeasures

### Prevention

| Control | Description |
|---|---|
| **Risk-based patch prioritisation** | Implement EPSS-based or CVSS v4-based patch prioritisation that distinguishes exploitability from theoretical severity — resist queue-based FIFO patching |
| **Vulnerability disclosure pattern monitoring** | Monitor the rate and source of vulnerability disclosures for your software stack; flag unusual spikes in disclosure volume |
| **Patching capacity management** | Maintain a patching capacity reserve (20–30% of patching team capacity) specifically for critical/zero-day response |

### Detection

| Control | Description |
|---|---|
| **Disclosure volume spike detection** | Alert on a significant increase in CVE disclosures for your key software dependencies over a 2–4 week period compared to historical baseline |
| **Disclosure source pattern analysis** | Identify vulnerability disclosures attributed to newly-created researcher identities or organisations with no prior disclosure history — flag for additional scrutiny |
| **Exploitation-in-the-wild monitoring** | Cross-reference your patching queue with CISA KEV (Known Exploited Vulnerabilities) and GreyNoise exploitation data — prioritise KEV entries regardless of queue position |

### Response

- Immediately re-prioritise any CVE in the queue that appears on CISA KEV or shows active exploitation evidence
- Conduct a retrospective analysis of the disclosure volume spike to identify coordinated patterns
- Report suspected fabricated vulnerability disclosures to MITRE CVE and relevant NVD contacts
- Review patching prioritisation process to ensure exploitation evidence is weighted above disclosure volume
