# PT03.001 — BIOS Config Trigger Payloads

## Metadata

| Field | Value |
|---|---|
| **ID** | PT03.001 |
| **Tactic** | PT03 — Execution |
| **MITRE Gap** | MITRE T1542 (Pre-OS Boot) covers bootkit persistence but not firmware-resident config-triggered payload execution as a standalone execution primitive activated by specific BIOS/UEFI configuration states |
| **Response Priority** | Low |
| **Status** | Draft |

---

## Description

An adversary embeds a payload within BIOS/UEFI firmware or a Unified Extensible Firmware Interface (UEFI) module that is conditionally triggered by a specific hardware configuration state — such as the presence of a particular SMBIOS value, a specific memory configuration, or a CMOS register value. The payload executes before the OS loader, enabling pre-OS code execution that persists across OS reinstalls and is invisible to all OS-level security tooling.

---

## Adversary Value

- Achieves execution that survives OS reinstalls, disk wipes, and endpoint agent removal
- Executes in a privileged ring below OS kernel (ring -1 / SMM), bypassing all OS security controls
- Trigger conditions can be set to activate only on specific hardware configurations, making the implant dormant and difficult to discover during routine analysis
- No OS-level artefacts — entirely invisible to EDR, AV, and SIEM without specialised firmware analysis tooling

---

## Why It's Not in MITRE ATT&CK

MITRE T1542.001 (System Firmware) and T1542.003 (Bootkit) cover firmware persistence but treat firmware modification as a persistence mechanism. This technique specifically describes firmware as an **execution trigger** — where a BIOS configuration state (e.g., CMOS register value, SMBIOS field, boot order) acts as a conditional execution switch for a payload, making it a novel execution primitive distinct from simple firmware persistence.

---

## Procedure Examples

1. Adversary gains physical or remote access with firmware update capability (e.g., via a vulnerable BIOS update utility, BMC/IPMI access, or supply chain access)
2. Modifies UEFI DXE driver to read a specific CMOS register value or SMBIOS field at boot
3. If the trigger value is present (e.g., set by a prior-stage implant or physical access), the embedded payload executes in SMM or DXE phase
4. Payload stages a subsequent OS-level implant or exfiltrates pre-boot secrets (disk encryption keys, TPM-sealed material)
5. Trigger value is cleared post-execution to prevent repeated firing and forensic discovery

---

## Countermeasures

### Prevention

| Control | Description |
|---|---|
| **Secure Boot enforcement** | Enable UEFI Secure Boot with vendor-signed keys; prevents unsigned UEFI modules from executing |
| **Firmware integrity measurement** | Use TPM-backed firmware measurement (PCR 0–7) and remote attestation to detect firmware modification |
| **BIOS update authentication** | Require signed firmware updates; disable BIOS update via OS interfaces in production systems |
| **Physical security** | Restrict physical access to servers; tamper-evident seals on chassis |

### Detection

| Control | Description |
|---|---|
| **Firmware hash monitoring** | Compare firmware hashes against known-good baselines using platform vendor tools (Dell BIOS Verification, HP Sure Start, Eclypsium) |
| **PCR value monitoring** | Alert on unexpected changes to TPM PCR 0 (BIOS/UEFI) or PCR 7 (Secure Boot state) between reboots |
| **UEFI variable monitoring** | Monitor UEFI variable writes via OS interfaces (`efivarfs`) for unexpected modifications |

### Response

- Immediately isolate affected hardware from the network
- Do not reinstall OS — the implant survives OS reinstalls; requires full firmware reflash from vendor-provided image
- Engage hardware vendor for firmware integrity verification and reflash procedure
- Audit physical access logs for the affected system covering the last 12 months
- Consider hardware replacement if firmware reflash cannot be verified cryptographically
