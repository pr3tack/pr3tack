## Description

Identifying recently deleted IAM roles or service accounts referenced in old documentation, public GitHub commits, Terraform state files, or CI/CD configurations. The adversary monitors these "Ghost Names" to detect if they can be re-registered in a different context or if hardcoded scripts continue to invoke that specific now-vacant role — enabling "Lateral Movement via Omission."

---

## Adversary Value

- Allows an adversary to take over a **trusted identity name** that internal systems still expect to exist
- Hardcoded references in legacy code may automatically grant trust or permissions to a re-registered role
- Ghost role monitoring requires no interaction with target systems — all intelligence is gathered from public sources
- Re-registration may be achievable in a different AWS account or GCP project, exploiting cross-account trust policies

---

## Why It's Not in MITRE ATT&CK

MITRE's Cloud Service Discovery and Valid Accounts techniques focus on active enumeration within authenticated environments. This technique exploits the **temporal gap** created by the decommissioning of IAM identities — a passive, external reconnaissance and pre-positioning activity that MITRE does not address.

---

## Procedure Examples

1. Mine public GitHub repositories, Terraform registries, and documentation for hardcoded IAM role ARNs, service account names, and identity principal references
2. Cross-reference discovered identities against cloud provider APIs to identify which are no longer active
3. Monitor for re-registration windows — particularly in cross-account trust scenarios
4. Observe whether CI/CD pipelines, cron jobs, or monitoring systems continue to generate errors referencing the ghost role name
5. Attempt re-registration of the ghost role in an adversary-controlled account to intercept trust grants

---

## Countermeasures

### Prevention

| Control | Description |
|---|---|
| **IAM Role Lifecycle Governance** | Implement a "tombstone" policy — decommissioned IAM roles and service accounts must be marked as non-re-registerable for a minimum of 12 months. Use cloud-native controls (AWS SCP, GCP Org Policies) to enforce this. |
| **Code Repository Scanning** | Scan all public and internal repositories for hardcoded references to IAM role ARNs, service account names, or identity principals. Rotate or remove any that reference decommissioned identities. |
| **Ghost Identity Auditing** | Run quarterly audits specifically looking for "dangling references" — codebases, Terraform state files, CI/CD configs, or documentation that still reference deleted roles. |

### Detection

| Control | Description |
|---|---|
| **IAM Assume-Role Monitoring** | Alert on any `sts:AssumeRole` or service-account impersonation attempt targeting a role name that was previously decommissioned. |
| **Public Leak Monitoring** | Use secret-scanning tools (e.g., TruffleHog, GitLeaks) to detect leaked IAM identifiers in public repositories. |

### Response

- If a ghost role is re-registered by an unknown party, immediately block the principal and audit all actions taken under that identity
- Patch all codebases that still reference the squatted role, and rotate any secrets those codebases may have exposed
