# PT01.005 — Vector Database Reconnaissance

## Metadata

| Field | Value |
|---|---|
| **ID** | PT01.005 |

| **Response Priority** | High |

---

## Description

Systematically probing a target's vector database (Pinecone, Weaviate, Qdrant, ChromaDB) or RAG (Retrieval-Augmented Generation) pipeline through crafted embedding queries. By issuing iterative nearest-neighbour searches, an adversary can reconstruct sensitive source documents, extract metadata containing credentials or internal architecture details, and map the knowledge base that underlies a target's AI-powered products — without ever accessing the raw document store directly.

---

## Adversary Value

- Vector databases are frequently internet-exposed without authentication, relying on obscurity rather than access control
- Metadata stored alongside embeddings often contains plaintext credentials, internal paths, and architecture details
- Iterative nearest-neighbour queries can reconstruct source documents with high fidelity — bypassing document-level access controls
- Provides adversary with the **same internal knowledge** as the AI assistant, including proprietary data and operational context
- RAG pipelines are a new and underappreciated attack surface not covered by existing security tooling

---

## Why It's Not in MITRE ATT&CK

MITRE does not currently model AI/ML infrastructure as a target of reconnaissance activity. Vector databases represent a fundamentally new data store type where the query interface itself enables data reconstruction — a capability with no equivalent in traditional database reconnaissance. The technique requires understanding of embedding semantics rather than SQL or NoSQL query languages.

---

## Procedure Examples

1. Identify target's AI-powered product surface and infer use of a RAG pipeline (via job postings, open-source code, model behaviour analysis)
2. Discover exposed vector database endpoints via EASM scanning or subdomain enumeration
3. Issue crafted embedding queries using the target's likely embedding model to probe the vector space
4. Perform iterative nearest-neighbour searches to reconstruct source document chunks
5. Extract and analyse metadata fields for credentials, internal paths, and architectural intelligence

---

## Countermeasures

### Prevention

| Control | Description |
|---|---|
| **Network Isolation** | Implement Private Link / VPC Endpoints for all vector databases (Pinecone, Weaviate, Qdrant, ChromaDB) to ensure they are never reachable via the public internet. |
| **RAG Metadata Masking** | Ensure metadata stored within vector indexes does not contain plaintext credentials, internal paths, architecture details, or PII. Treat vector DB metadata with the same classification as source documents. |
| **Authentication & Authorisation** | Enforce API-key rotation and RBAC on all vector database operations. Separate read and write permissions, and restrict embedding-upload operations to CI/CD pipelines only. |
| **Embedding Watermarking** | Where feasible, embed unique watermarks in vector embeddings that can be traced back to your organisation if exfiltrated. |

### Detection

| Control | Description |
|---|---|
| **Query Pattern Anomaly Detection** | Monitor vector database query logs for adversarial probing patterns — queries that systematically explore embedding space boundaries or attempt to reconstruct source documents via iterative nearest-neighbour searches. |
| **Unauthorised Access Alerting** | Alert on vector DB access from IPs, service accounts, or roles that are not in the authorised access list. |

### Response

- If unauthorised access is confirmed, rotate all API keys and re-evaluate network access controls
- Assess whether the adversary was able to extract meaningful information from embedding similarities
- Determine the blast radius of the exposed data and notify relevant data owners
