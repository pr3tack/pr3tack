# PT12.002 — /proc Environmental Archaeology

## Metadata

| Field | Value |
|---|---|
| **ID** | PT12.002 |
| **Tactic** | PT12 — Unix Contextual Stealth |
| **Sub-techniques** | Cross-PID Environment Variable Harvesting |
| **MITRE Gap** | MITRE focuses on credential access via files and memory dumps; this technique specifically exploits the process-level environment variable exposure unique to the Unix `/proc` filesystem in multi-tenant or shared-kernel environments, including container orchestration platforms. |
| **Response Priority** | High |
| **Status** | Draft |

---

## Description

Mining `/proc/[pid]/environ` for secrets (API keys, database credentials, OAuth tokens) passed into ephemeral containers or processes as environment variables during the provisioning or startup phase. In environments where PID namespace isolation is absent or misconfigured, a compromised process can read the full environment of any other process on the host — including init processes, sidecars, and service meshes that may hold high-value credentials injected at deployment time.

---

## Adversary Value

- Environment variables are a common, well-documented anti-pattern for secret injection, widely practiced despite guidance against it
- `/proc/[pid]/environ` is world-readable by default on many Linux distributions unless `hidepid` is enforced
- A single compromised container on a shared node can harvest secrets from all co-located workloads
- Credentials obtained this way are live (not rotated), valid, and often have broad permissions
- The technique leaves minimal forensic artefacts — file reads on `/proc` are not logged by default

---

## Why It's Not in MITRE ATT&CK

MITRE's Unsecured Credentials technique covers credentials in files, command history, and configuration files. This technique specifically exploits the **`/proc` virtual filesystem** — a Linux kernel interface that exposes process environment at runtime. The cross-process, cross-container nature of this exposure in shared-kernel environments (Kubernetes nodes, LXC hosts) is not addressed in MITRE's current credential access coverage.

---

## Procedure Examples

1. Gain initial foothold on a shared Linux host or container with access to the host's PID namespace
2. Enumerate running processes: `ls /proc/ | grep -E '^[0-9]+$'`
3. Read environment variables of target processes: `cat /proc/[pid]/environ | tr '\0' '\n'`
4. Filter for high-value secrets: API keys, `AWS_SECRET_ACCESS_KEY`, `DATABASE_URL`, OAuth tokens, service account credentials
5. Extract and exfiltrate discovered credentials for use in subsequent phases

---

## Countermeasures

### Prevention

| Control | Description |
|---|---|
| **Secret Injection Hardening** | Never pass secrets as environment variables to containers. Use mounted secret volumes (Kubernetes Secrets with `projected` volumes, Vault Agent) with file-based injection that can be revoked and rotated independently. |
| **procfs Restriction** | Use `hidepid=2` (or `hidepid=invisible` on newer kernels) mount option on `/proc` in multi-tenant or shared-kernel environments to prevent cross-process environment snooping. |
| **PID Namespace Isolation** | Enforce PID namespace isolation so containers cannot traverse `/proc` of processes outside their own namespace. |
| **Ephemeral Secret Lifetime** | Ensure secrets injected into containers have the shortest possible TTL, and are rotated automatically on each deployment. |

### Detection

| Control | Description |
|---|---|
| **Procfs Access Monitoring** | Alert on any process accessing `/proc/[pid]/environ` for PIDs outside its own process group — this is almost always malicious in production environments. |
| **EDR Behavioural Rules (CrowdStrike Falcon)** | Create detection rules for `open()` syscalls targeting `/proc/*/environ` with cross-PID access patterns. |
| **Wiz Cloud Workload /proc Monitoring** | Wiz runtime monitoring for anomalous `/proc` filesystem access patterns in cloud workloads. |

### Response

- If environment variable exfiltration is confirmed, immediately rotate all secrets present in the compromised container's environment
- Audit all containers on the same node for similar exposure
- Isolate the affected host; conduct forensic capture of `/proc` state and process list at time of compromise
