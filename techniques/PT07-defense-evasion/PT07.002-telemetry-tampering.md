# PT07.002 — Telemetry Tampering

## Metadata

| Field | Value |
|---|---|
| **ID** | PT07.002 |
| **Tactic** | PT07 — Defense Evasion |
| **MITRE Gap** | MITRE T1562.006 (Indicator Blocking) covers telemetry suppression broadly; this technique specifically addresses in-place modification of local telemetry agent configuration, event streams, or buffered log data to selectively suppress or rewrite specific events before they reach the SIEM — a more surgical approach than agent killing that is harder to detect via agent health monitoring |
| **Response Priority** | Medium |
| **Status** | Draft |

---

## Description

An adversary with local administrative access modifies the configuration, filter rules, or event buffer of a local telemetry agent (EDR sensor, Syslog forwarder, Windows Event Forwarding subscription, or cloud logging agent) to suppress or rewrite specific event types before they are transmitted to the centralised logging platform. Unlike disabling the agent entirely (which triggers an availability alert), selective telemetry tampering allows the agent to continue operating and reporting benign events while silently dropping or modifying events related to the adversary's activity.

---

## Adversary Value

- Avoids triggering agent health/availability alerts that would fire if the agent were killed or stopped
- Selectively suppresses only the specific event types relevant to the adversary's activity — minimising detection of the tampering itself
- Modifying event buffer files or filter configs leaves no obvious process-level artefact
- Effective against SIEM-dependent detection stacks that have no independent validation of event completeness

---

## Why It's Not in MITRE ATT&CK

MITRE T1562.006 (Indicator Blocking) covers event log clearing and blocking but does not address the specific pattern of in-place telemetry agent **configuration modification** or **event stream rewriting** — where the agent continues to function and forward events selectively. This is a distinct and more sophisticated evasion technique that exploits the trust model of agent-based telemetry pipelines.

---

## Procedure Examples

1. Adversary gains local admin access on the target host
2. Locates the telemetry agent configuration file (e.g., Splunk Universal Forwarder `inputs.conf`, Elastic Agent `agent.yml`, Syslog-ng filter rules, Windows Event Forwarding subscription XML)
3. Adds a filter or exclusion rule targeting specific Event IDs, process names, or source paths associated with their activity (e.g., excludes Event ID 4688 for specific process names, or excludes Sysmon Event ID 1 for their malware process)
4. Restarts or signals the agent to reload configuration — agent continues operating normally, suppressing only targeted events
5. Alternatively: directly modifies the agent's event queue buffer files on disk to remove specific entries before they are transmitted

---

## Countermeasures

### Prevention

| Control | Description |
|---|---|
| **Agent config file integrity** | Monitor telemetry agent configuration files with a file integrity monitoring (FIM) solution; alert on any modification |
| **Least-privilege agent accounts** | Run telemetry agents under dedicated service accounts with minimal permissions; prevent non-agent processes from writing to agent config directories |
| **Centralised config management** | Manage agent configurations centrally (via Fleet, Ansible, or MDM); any config drift from the golden image is a finding |

### Detection

| Control | Description |
|---|---|
| **Event volume baseline anomaly** | Alert on hosts where event volume drops significantly below the established baseline without a corresponding change in workload — selective suppression causes volume anomalies |
| **Agent config file modification** | Alert on writes to agent configuration files (Splunk `inputs.conf`, Elastic `agent.yml`, Windows Event Forwarding subscription registry keys) |
| **Cross-source event gap detection** | Cross-correlate events from multiple telemetry sources (EDR + network + cloud) for the same host; gaps in one source that are not reflected in others indicate selective suppression |
| **Agent process file descriptor monitoring** | Monitor file descriptors opened by the agent process; alert if the agent opens its own config file for writing (self-modification) |

### Response

- Compare current agent configuration against the centralised golden configuration baseline
- Restore configuration from the central management system immediately
- Review all events from the affected host in the period since the config modification — assume events in that window may be incomplete
- Treat the host as fully compromised and initiate full incident response
- Audit all hosts managed by the same agent deployment for similar config modifications
