# PT07.001 — Cloud Metadata API Overloading

## Metadata

| Field | Value |
|---|---|
| **ID** | PT07.001 |
| **Tactic** | PT07 — Defense Evasion |
| **MITRE Gap** | MITRE T1562 (Impair Defenses) covers defense impairment broadly; this technique specifically targets the cloud instance metadata service (IMDS) as an evasion mechanism — flooding or manipulating IMDS responses to hide credential theft, mask instance identity in logs, or prevent metadata-based anomaly detection from correlating events correctly |
| **Response Priority** | Medium |
| **Status** | Draft |

---

## Description

An adversary floods or manipulates requests to the cloud instance metadata service (AWS IMDSv1/v2, GCP metadata server, Azure IMDS) to corrupt metadata-dependent logging pipelines, mask the true identity of a compromised instance in cloud audit logs, or prevent metadata-based anomaly detection from correctly correlating malicious activity with a specific instance or role. By overloading the metadata service, the adversary can cause log entries to appear with null, stale, or incorrect instance identity metadata, effectively anonymising their cloud-side activity.

---

## Adversary Value

- Cloud audit logs that rely on IMDS-derived instance identity (instance ID, account ID, assumed role) become unreliable when IMDS responses are manipulated
- IMDSv1 (still widely deployed) has no request authentication — any process on the instance can flood or manipulate metadata requests
- Causes false attribution of malicious API calls to incorrect instances or roles, hampering incident response correlation
- Can be combined with credential theft (IMDS token theft) to mask the source of subsequent API abuse

---

## Why It's Not in MITRE ATT&CK

MITRE T1552.005 (Cloud Instance Metadata API) covers metadata credential theft; T1562 covers defense impairment. Neither specifically addresses using IMDS request flooding or response manipulation as a **log corruption and attribution evasion** mechanism — the specific pattern of targeting metadata-dependent logging infrastructure to anonymise cloud-side activity.

---

## Procedure Examples

1. Adversary establishes execution on a cloud instance (via compromised credentials, SSRF, or RCE)
2. On IMDSv1 instances: floods `http://169.254.169.254/latest/meta-data/` with high-rate requests to cause rate limiting or response corruption in downstream logging agents that pull instance identity from IMDS
3. On IMDSv2 instances: repeatedly rotates the IMDSv2 session token to invalidate tokens held by monitoring agents, causing them to log events with stale or null instance identity
4. Simultaneously performs malicious API calls — cloud audit logs attribute the calls to a null or incorrect instance identity, preventing correlation with the compromised instance
5. Alternatively: manipulates local `cloud-init` or metadata caching layers to inject false instance identity data into the logging agent's metadata cache

---

## Countermeasures

### Prevention

| Control | Description |
|---|---|
| **IMDSv2 enforcement** | Enforce IMDSv2 (token-required mode) on all instances; disable IMDSv1; set hop limit to 1 to prevent SSRF-based metadata theft |
| **IMDS access restriction** | Use iptables/nftables or AWS instance metadata endpoint policies to restrict which processes can access IMDS |
| **Out-of-band instance identity** | Do not rely solely on IMDS for instance identity in logging pipelines — augment with VPC Flow Log source IPs and CloudTrail `sourceIPAddress` fields |

### Detection

| Control | Description |
|---|---|
| **IMDS request rate anomaly** | Alert on processes generating > 100 IMDS requests per minute from a single instance — normal workloads rarely require high-frequency metadata polling |
| **Log entries with null instance identity** | Alert on CloudTrail or cloud audit log entries where the `instanceId` or `principalId` field is null or mismatched with the expected instance roster |
| **IMDSv2 token rotation rate** | Alert on unusually high IMDSv2 session token generation rates from a single instance |

### Response

- Immediately isolate the affected instance from the network
- Pull CloudTrail logs using the source IP address (not instance ID) for correlation — instance ID metadata may be corrupted
- Review VPC Flow Logs for outbound connections from the instance IP during the anomaly window
- Rotate all IAM role credentials associated with the instance profile
- Snapshot and forensically analyse the instance before termination
