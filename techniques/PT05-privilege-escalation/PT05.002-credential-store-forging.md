# PT05.002 — Credential Store Forging

## Metadata

| Field | Value |
|---|---|
| **ID** | PT05.002 |
| **Tactic** | PT05 — Privilege Escalation |
| **MITRE Gap** | MITRE T1556 (Modify Authentication Process) covers authentication manipulation; this technique specifically describes the pre-positioning of crafted credential cache entries (forged cached tokens, synthetic Kerberos tickets, or planted DPAPI blobs) to be retrieved and elevated by a privileged process at a later time — a temporal separation between planting and exploitation not addressed in current MITRE mappings |
| **Response Priority** | Medium |
| **Status** | Draft |

---

## Description

An adversary with limited initial access plants crafted or forged credential material — synthetic cached domain credential hashes, forged Kerberos tickets (golden/silver ticket variants), or DPAPI-encrypted blobs containing elevated tokens — into OS credential stores accessible to a higher-privileged process. When the privileged process (OS scheduler, authentication service, or credential renewal daemon) subsequently reads and processes the planted material, the adversary's payload is evaluated with elevated privilege, achieving privilege escalation through credential store manipulation rather than direct exploitation.

---

## Adversary Value

- Temporal separation between planting and exploitation reduces detection correlation probability
- Forged Kerberos tickets and cached credentials are processed by trusted OS subsystems — the escalation event appears as a legitimate authentication operation
- Avoids direct exploitation of a vulnerability — the escalation path is entirely through legitimate OS authentication flows
- Effective in environments where direct privilege escalation exploits are blocked by EDR but credential store access is available to lower-privileged accounts

---

## Why It's Not in MITRE ATT&CK

MITRE T1558 (Steal or Forge Kerberos Tickets) focuses on credential theft for lateral movement; T1556 covers authentication process modification. Neither specifically addresses the pattern of **pre-positioning** forged credential material in a store for later retrieval and processing by a privileged subsystem as a **privilege escalation** vector — the key distinction being the temporal gap and the exploitation of trusted OS credential processing.

---

## Procedure Examples

1. Adversary gains access to a low-privilege account with write access to the Windows Credential Manager or DPAPI store
2. Plants a crafted DPAPI blob containing a forged token or elevated credential material, encrypted under the target user's DPAPI master key (obtained via DPAPI blob decryption or shadow copy)
3. Waits for a credential renewal or authentication event that causes a privileged process to process the planted blob
4. Alternatively: plants forged cached domain credentials (MS-CACHE v2 hashes) in the LSA cache, which are then used by a domain-joined system during offline authentication to grant elevated access
5. On trigger, the privileged process decrypts and uses the forged material, granting the adversary elevated session or token access

---

## Countermeasures

### Prevention

| Control | Description |
|---|---|
| **Credential Manager access restriction** | Restrict access to Windows Credential Manager and DPAPI stores via GPO; require elevation for any credential store write operation |
| **LSA protection** | Enable LSA Protected Process Light (PPL) to prevent unprivileged access to LSASS memory and credential cache |
| **Cached credential minimisation** | Set `CachedLogonsCount` to 0 on workstations where offline authentication is not required |
| **Kerberos ticket validation** | Enforce Kerberos PAC validation; monitor for tickets with unusual PAC contents or lifetimes |

### Detection

| Control | Description |
|---|---|
| **Credential Manager write monitoring** | Alert on writes to Windows Credential Manager (`vaultcmd`, `CredWrite` API calls) from non-standard processes |
| **LSA cache modification** | Alert on modifications to HKLM\SECURITY\Cache (requires SYSTEM-level monitoring) — changes outside scheduled credential refresh windows are suspicious |
| **Kerberos ticket anomalies** | Alert on Kerberos TGT or service ticket requests with lifetimes, PAC fields, or encryption types inconsistent with your DC configuration |

### Response

- Flush all cached credentials (`cmdkey /list`, then remove all entries; or Group Policy enforcement)
- Purge Kerberos ticket cache (`klist purge`) across affected systems
- Reset passwords for all accounts whose credential material may have been targeted
- Engage Active Directory team to reset the KRBTGT account password (twice) if golden ticket is suspected
- Review DPAPI master key access logs
