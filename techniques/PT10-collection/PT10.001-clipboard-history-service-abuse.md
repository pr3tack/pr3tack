# PT10.001 — Abuse of Clipboard History Services

## Metadata

| Field | Value |
|---|---|
| **ID** | PT10.001 |
| **Tactic** | PT10 — Collection |
| **MITRE Gap** | MITRE T1115 (Clipboard Data) covers real-time clipboard capture; this technique specifically targets OS-level clipboard history services (Windows Clipboard History, macOS Universal Clipboard, Linux clipboard managers like Klipper/Clipman) that persist clipboard contents across sessions and applications — enabling retrospective credential harvesting beyond the single-copy-paste window addressed by T1115 |
| **Response Priority** | High |
| **Status** | Draft |

---

## Description

An adversary reads from OS-level clipboard history services that persist clipboard contents across time — going beyond real-time clipboard capture to harvest all clipboard entries stored since the service was enabled. Windows 10/11 Clipboard History (`WIN+V`) stores up to 25 entries per session including text, images, and HTML; macOS Universal Clipboard syncs across Apple devices; Linux clipboard managers (Klipper, Clipman, CopyQ) can retain thousands of entries. This creates a rich retrospective cache of sensitive data — passwords, API keys, OTPs, internal URLs, and confidential text — that the adversary can harvest in a single operation.

---

## Adversary Value

- Single operation yields potentially hundreds of historical clipboard entries — far more valuable than real-time capture
- Clipboard history frequently contains credentials (passwords copied from password managers), API keys, OTPs, and sensitive business data
- Clipboard history databases are stored in predictable filesystem locations accessible to user-level processes
- The historical nature means credentials entered days ago remain harvestable even if the user has since changed their workflow

---

## Why It's Not in MITRE ATT&CK

MITRE T1115 covers real-time clipboard data capture. This technique specifically exploits **persistent clipboard history stores** — a distinct attack surface from real-time clipboard monitoring. The key difference is retrospective access: an adversary can harvest weeks of clipboard history in a single operation without requiring persistent monitoring.

---

## Procedure Examples

1. **Windows:** Adversary accesses `%LOCALAPPDATA%\Microsoft\Windows\Clipboard\` — the SQLite database storing Clipboard History entries — and extracts all stored text, HTML, and image entries
2. **macOS:** Reads the Universal Clipboard SQLite database at `~/Library/Application Support/com.apple.CloudClipboard/` or queries the `NSPasteboard` API for historical entries
3. **Linux (Klipper):** Reads `~/.local/share/klipper/history2.lst` or `~/.config/klipper/klipperrc` which contain the full clipboard history in plaintext or XML
4. Parses extracted entries for credential patterns (password manager paste format, API key patterns, OTP codes), internal URLs, and confidential document excerpts
5. Exfiltrates harvested credentials for offline use

---

## Countermeasures

### Prevention

| Control | Description |
|---|---|
| **Disable Clipboard History** | Disable Windows Clipboard History via GPO (`User Configuration > Administrative Templates > System > OS Policies > Allow Clipboard History`) |
| **Password manager clipboard clearing** | Configure password managers to auto-clear clipboard after 30–60 seconds; most enterprise password managers (1Password, Bitwarden) support this |
| **Clipboard manager restriction** | Prevent installation of third-party clipboard managers on managed endpoints via application allowlisting |

### Detection

| Control | Description |
|---|---|
| **Clipboard history database access** | Alert on processes other than the OS clipboard service or approved password managers reading the clipboard history database files |
| **Clipboard history API calls from unexpected processes** | Monitor calls to `IDataObject::GetData()` for CF_HISTORYCLIPBOARD format from non-standard processes |
| **Bulk clipboard entry access** | Alert on processes reading more than 5 clipboard history entries in a single session |

### Response

- Immediately clear clipboard history on all affected hosts (`Settings > System > Clipboard > Clear` or via PowerShell `Remove-Item "$env:LOCALAPPDATA\Microsoft\Windows\Clipboard\*"`)
- Assume all clipboard entries from the history window are compromised — rotate any credentials that may have been copied
- Disable clipboard history via GPO pending security review
- Investigate the process that accessed the clipboard history database
