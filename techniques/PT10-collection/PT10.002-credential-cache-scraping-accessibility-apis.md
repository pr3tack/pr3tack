# PT10.002 — Credential Cache Scraping via Accessibility APIs

## Metadata

| Field | Value |
|---|---|
| **ID** | PT10.002 |
| **Tactic** | PT10 — Collection |
| **MITRE Gap** | MITRE T1056.002 (GUI Input Capture) covers keylogging; this technique specifically abuses OS accessibility APIs (Windows UI Automation, macOS Accessibility API, AT-SPI on Linux) to read credential material directly from application UI elements — bypassing input capture by reading from the application's memory-backed UI state rather than intercepting keystrokes |
| **Response Priority** | Medium |
| **Status** | Draft |

---

## Description

An adversary abuses OS-level accessibility APIs — designed for screen readers and assistive technology — to programmatically read the text content of UI elements in other applications, including password fields, token display areas, and authentication dialogs. Unlike keylogging, accessibility API scraping reads credential material directly from the application's UI element tree, capturing values that are already present in memory without intercepting input events. This technique is effective against software tokens, OTP display apps, password manager auto-fill UI, and authentication dialogs.

---

## Adversary Value

- Captures credential material already in memory — not dependent on the user actively typing
- Accessibility APIs are legitimate OS features; their use is not blocked by application sandboxing or most EDR solutions
- Can capture OTP/TOTP codes displayed in authenticator apps, session tokens shown in browser dev tools, and API keys displayed in SaaS dashboards
- macOS Accessibility API access requires a one-time TCC permission grant — but many enterprise applications grant this permission to management tools

---

## Why It's Not in MITRE ATT&CK

MITRE T1056.002 covers GUI input capture (keylogging via UI hooks). This technique is distinct — it reads from the **application's UI element state** via accessibility APIs rather than intercepting input events. The mechanism bypasses input event monitoring and captures data that the user has not actively typed (e.g., auto-filled passwords, displayed tokens, session values rendered in UI).

---

## Procedure Examples

1. **Windows (UI Automation):** Adversary's process calls `IUIAutomation::ElementFromPoint()` or `FindAll()` to traverse the UI element tree of a target application (browser, password manager, authenticator app) and reads `ValuePattern.CurrentValue` from password fields and token display elements
2. **macOS (Accessibility API):** Requests `AXUIElementCopyAttributeValue(kAXValueAttribute)` on UI elements in the target application's accessibility tree — returning plaintext values from password fields if the application does not implement `AXSecureTextField` correctly
3. **Linux (AT-SPI):** Uses the AT-SPI2 D-Bus interface to enumerate UI elements and read accessible text values from target application windows
4. Targets authenticator apps (Google Authenticator, Authy) to capture TOTP codes, and password manager UI to capture auto-filled credential values before they are cleared

## Countermeasures

### Prevention

| Control | Description |
|---|---|
| **Accessibility API restriction** | Restrict which applications are granted Accessibility API access via MDM/TCC policy (macOS); audit and minimise the accessibility permission allowlist |
| **Sensitive field masking** | Ensure password fields and token display elements use OS-protected UI element types (`AXSecureTextField`, `PasswordBox`) that prevent accessibility API value reads |
| **Application sandboxing** | Sandbox sensitive applications (authenticators, password managers) to prevent cross-process accessibility API access |

### Detection

| Control | Description |
|---|---|
| **Accessibility API access from unexpected processes** | Alert on processes calling Windows UI Automation or macOS Accessibility APIs to access UI elements of sensitive applications (browsers, password managers, authenticators) |
| **TCC database modification (macOS)** | Alert on modifications to the TCC database granting accessibility permissions to new processes |
| **AT-SPI D-Bus enumeration (Linux)** | Monitor D-Bus calls to AT-SPI2 interfaces from non-standard accessibility consumers |

### Response

- Revoke accessibility permissions from all non-essential applications immediately
- Assume all credential material visible in the UI of targeted applications during the attack window is compromised — rotate affected credentials
- Review macOS TCC database (`~/Library/Application Support/com.apple.TCC/TCC.db`) for unauthorised accessibility grants
- Investigate the process that accessed accessibility APIs for target applications
