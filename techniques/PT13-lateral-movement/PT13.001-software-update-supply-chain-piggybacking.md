# PT13.001 — Software Update Supply Chain Piggybacking

## Metadata

| Field | Value |
|---|---|
| **ID** | PT13.001 |
| **Tactic** | PT13 — Lateral Movement |
| **MITRE Gap** | MITRE T1195.002 (Compromise Software Supply Chain) covers supply chain compromise as an initial access vector; this technique specifically describes piggybacking on the **update mechanism** of a low-popularity, already-installed application to move laterally across an organisation — exploiting the trusted, authenticated update channel of a legitimate application to deliver lateral movement payloads to all endpoints running that software |
| **Response Priority** | Medium |
| **Status** | Draft |

---

## Description

An adversary compromises the software update infrastructure or signing key of a low-popularity application already deployed across the target organisation — such as an internal tool, niche productivity app, or rarely-audited third-party utility — and delivers a malicious update payload through the application's trusted, authenticated update channel. Because the update is signed by the legitimate vendor's certificate and delivered through the expected update mechanism, it passes endpoint security controls and is automatically installed across all endpoints running the software, achieving broad lateral movement in a single operation.

---

## Adversary Value

- Legitimate code signing and trusted update channels bypass application allowlisting, EDR behavioural controls, and user awareness
- Single compromise of one update server achieves simultaneous lateral movement to all endpoints running the target software
- Low-popularity applications have less security scrutiny of their update pipelines and signing infrastructure
- Update delivery appears as expected, routine application behaviour — no unusual network connections or process launches

---

## Why It's Not in MITRE ATT&CK

MITRE T1195.002 covers supply chain compromise as an initial access vector (e.g., SolarWinds). This technique specifically addresses supply chain piggybacking as a **lateral movement** mechanism — where the adversary is already inside the network and exploits an installed application's update channel to spread to additional endpoints, rather than using the supply chain for initial access.

---

## Procedure Examples

1. Adversary identifies a low-popularity application deployed across most endpoints in the target environment (internal monitoring tool, niche PDF viewer, legacy line-of-business utility)
2. Compromises the vendor's update server or code signing key (via credential compromise, CI/CD pipeline compromise, or vendor network access)
3. Packages a malicious payload alongside or as a replacement for a legitimate update, signed with the vendor's code signing certificate
4. The application's auto-update mechanism downloads and installs the malicious update on all endpoints during the next update cycle
5. The payload executes with the application's privilege level, establishing persistence and exfiltrating credentials or staging further lateral movement

---

## Countermeasures

### Prevention

| Control | Description |
|---|---|
| **Software inventory and update source review** | Maintain a complete software inventory; for each application, document the update server URL and signing certificate; alert on any changes |
| **Update traffic monitoring** | Monitor outbound connections to application update servers; alert on unexpected update server URLs or certificate changes |
| **Vendor security assessment** | Include update infrastructure security in third-party vendor assessments; require vendors to document their code signing and update delivery security controls |

### Detection

| Control | Description |
|---|---|
| **Application update behaviour anomaly** | Alert on applications launching new processes, writing to unexpected filesystem locations, or establishing network connections immediately after an update event |
| **Code signing certificate change** | Alert on application updates signed by a different certificate than the previously observed signing certificate |
| **Mass simultaneous update execution** | Alert on the same application update installing across a large number of endpoints simultaneously — especially for low-popularity applications |

### Response

- Immediately block the update server URL at the network perimeter
- Roll back the malicious update on all affected endpoints
- Isolate and forensically analyse a representative sample of affected endpoints
- Engage the vendor to investigate and revoke the compromised signing key
- Assess the full blast radius — all endpoints running the application during the update window are potentially compromised
