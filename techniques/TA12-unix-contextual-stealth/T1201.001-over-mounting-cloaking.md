# T1201.001 — Over-Mounting Cloaking

## Metadata

| Field | Value |
|---|---|
| **ID** | T1201.001 |
| **Tactic** | TA12 — Unix Contextual Stealth |
| **Sub-techniques** | Mount Point Obfuscation |
| **MITRE Gap** | MITRE covers "Hidden Files and Directories" but not the specific Unix behaviour of using mount points to hide "warm" staged data beneath a temporary filesystem overlay — a technique invisible to most file-based scanners. |
| **Response Priority** | High |
| **Status** | Draft |

---

## Description

Staging reconnaissance data, tools, or exfiltration payloads in a target directory and then mounting a `tmpfs` (temporary filesystem) over that directory. The staged data remains physically on disk but is masked from the operating system, running processes, and file-based security scanners until the mount is removed. This technique exploits standard Unix/Linux filesystem semantics rather than any vulnerability, making it broadly applicable across Linux hosts and container environments.

---

## Adversary Value

- Staged data is invisible to all file-system-based scanners while the mount is active
- Requires no kernel exploit — exploits standard, expected Unix filesystem behaviour
- `tmpfs` layer appears empty or contains innocuous decoy files to casual inspection
- Unmounting instantly "reveals" the staged data for use without leaving forensic artefacts on the overlay
- Effective in container environments where filesystem isolation is expected but mount capabilities may be over-granted

---

## Why It's Not in MITRE ATT&CK

MITRE's Hide Artefacts technique (T1564) covers hidden files and rootkit-based concealment. This technique is distinct: it uses the **mount namespace layer** — a standard Unix kernel feature — to achieve concealment. No kernel modification, rootkit, or file attribute change is required. The technique exploits the expected behaviour of the `mount` syscall, which is not addressed in MITRE's current coverage.

---

## Procedure Examples

1. Stage reconnaissance data, tooling, or payload files in a chosen target directory (e.g., `/var/lib/app-data/`)
2. Mount a `tmpfs` overlay over the directory: `mount -t tmpfs tmpfs /var/lib/app-data/`
3. Optionally populate the `tmpfs` layer with innocuous decoy files to pass casual inspection
4. The underlying staged data is now hidden from the OS, running processes, and file-based scanners
5. When ready to use the staged data, unmount the `tmpfs`: `umount /var/lib/app-data/` — the staged content is instantly accessible

---

## Countermeasures

### Prevention

| Control | Description |
|---|---|
| **Mount Namespace Restrictions** | Use seccomp profiles and AppArmor/SELinux policies to restrict which processes can create new mount points, especially `tmpfs` mounts over non-standard directories. |
| **Container Security Hardening** | In containerised environments, drop the `SYS_ADMIN` capability and enforce `readOnlyRootFilesystem` where possible to prevent mount operations entirely. |
| **Immutable Infrastructure** | Favour immutable container images and serverless architectures where filesystem modifications are inherently restricted. |

### Detection

| Control | Description |
|---|---|
| **Mount Point Auditing (EDR)** | Deploy EDR rules (CrowdStrike Falcon) to alert on the creation of `tmpfs` or overlay filesystems over non-empty directories — a strong indicator of data staging. |
| **Wiz Runtime Mount Anomaly** | Use Wiz Runtime Sensor detection of anomalous mount events in cloud workloads. |
| **Kernel Audit Rules** | Enable `auditd` rules for `mount`/`umount` syscalls, filtering for non-standard mount targets. |
| **Filesystem Drift Detection** | In container environments, use runtime security tools to detect any filesystem changes that deviate from the base image. |

### Response

- On alert, immediately capture the underlying directory contents **before** unmounting the cloaking filesystem
- Isolate the affected host and conduct a full forensic review of what was staged beneath the mount point
