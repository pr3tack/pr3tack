# PT02.004 — LLM Mirroring & Local Jailbreak Prep

## Metadata

| Field | Value |
|---|---|
| **ID** | PT02.004 |
| **Tactic** | PT02 — Resource Development |
| **Sub-techniques** | Offline Safety Filter Probing |
| **MITRE Gap** | MITRE does not currently codify the preparation of LLM-specific exploits in offline, adversary-controlled environments — a critical blind spot as AI-powered applications become primary attack surfaces. |
| **Response Priority** | Critical |
| **Status** | Draft |

---

## Description

Downloading a quantised version of a target's suspected or confirmed LLM (e.g., Llama-3-70B, Mistral) to a local adversary-controlled machine. The adversary performs thousands of automated "jailbreak" or "prompt injection" attempts locally to find a bypass for the model's safety filters — without ever triggering the target's API-based rate limits, monitoring systems, or abuse detection. The resulting injection string is validated offline before being deployed against the live target environment on the first attempt.

---

## Adversary Value

- Develops prompt injection payloads with **zero detection risk** against the target's live monitoring systems
- Exploits the fact that many commercial LLMs are fine-tuned from open-weight base models, making offline rehearsal directly applicable
- Provides a "ready-to-use" injection string that is tuned to work on the first live attempt
- Enables large-scale, automated jailbreak research that would be impossible at low cost against rate-limited production APIs
- Applicable against any AI-powered product surface: chatbots, AI coding assistants, AI security tooling, RAG pipelines

---

## Why It's Not in MITRE ATT&CK

MITRE does not codify the **preparation phase of LLM-specific exploits**. The use of open-weight model mirrors as a rehearsal environment for developing prompt injections is a novel adversary capability with no equivalent in traditional software exploitation — it has no MITRE analogue because AI systems as primary attack surfaces are not yet represented in the ATT&CK framework.

---

## Procedure Examples

1. Identify the target's AI-powered product surface and infer the likely underlying model (via public disclosures, model behaviour fingerprinting, or open-source intelligence)
2. Download the quantised open-weight equivalent (e.g., via Hugging Face) and deploy locally using llama.cpp, Ollama, or equivalent
3. Automate thousands of jailbreak and prompt-injection attempts using a fuzzing framework (e.g., Garak, PyRIT)
4. Identify injection patterns that consistently bypass the offline model's safety filters
5. Validate the winning injection string against the live target API, relying on the offline rehearsal to minimise detection-triggering attempts

---

## Countermeasures

### Prevention

| Control | Description |
|---|---|
| **Model Access Controls** | Restrict access to model weights and architecture documentation. Use watermarking or fingerprinting on model weights to detect unauthorised downloads. |
| **Input Validation & Prompt Hardening** | Deploy multi-layered input validation on all LLM-facing APIs — structural checks, semantic anomaly detection, and known-jailbreak-pattern matching. |
| **Rate Limiting with Behavioural Analysis** | Go beyond simple rate limiting — use behavioural analysis to detect prompt-injection probing patterns (systematic variation of inputs, boundary-testing sequences) even at low volumes. |
| **Model Versioning & Drift Detection** | Maintain strict versioning of deployed models and monitor for attempts to probe specific version behaviours. |
| **Safety Filter Fingerprinting** | Implement distinct safety-filter fingerprints per deployment so offline rehearsals against the base model diverge from live behaviour. |
| **Canary Tokens in Model Responses** | Embed canary tokens in model responses that trigger alerts if replayed from a different context. |

### Detection

| Control | Description |
|---|---|
| **Prompt Injection Signature Detection** | Deploy real-time classifiers on LLM input streams that detect known jailbreak patterns, role-injection attempts, and instruction-override sequences. |
| **Output Anomaly Monitoring** | Monitor LLM outputs for anomalous content — responses that contain system prompts, internal instructions, or content violating safety policy may indicate a successful injection. |
| **Canary Tokens in System Prompts** | Embed unique canary strings in system prompts that, if they appear in outputs, indicate a prompt-extraction attack. |
| **First-Call Anomaly Detection** | Alert if the very first API call from a client bypasses safety filters — indicative of an offline-rehearsed payload. |

### Response

- If a jailbreak succeeds in production, immediately rotate the system prompt and deploy an updated input filter
- Conduct forensic analysis on the successful injection to update the local detection classifier
- Block offending API keys; rotate model safety-filter configurations; invoke AI IR playbook
- Consider deploying a secondary "guardian" LLM that reviews outputs before they reach the user
