# PT02.003 — Cloud Account Incubation

---

## Description

Creating and aging AWS, GCP, or Azure accounts over weeks or months to build a "warm" reputation with cloud provider trust systems, WAF/SIEM enrichment services, and anomaly-detection platforms that use account age as a risk signal. The adversary uses the incubated account to conduct operations that appear to originate from an established, low-risk cloud tenant rather than a newly registered, high-risk one.

---

## Adversary Value

- Bypasses anomaly detection rules that flag "New Tenant" activity as high-risk
- Cloud provider reputation feeds (GreyNoise, Spur, IP Quality Score) score older accounts as lower risk
- Free-tier accounts can be warmed at near-zero cost before activation for attack use
- Account age signals are widely trusted by WAF and API gateway rules as a proxy for legitimacy
- Enables infrastructure that passes "cloud reputation" checks during initial access attempts

---

## Why It's Not in MITRE ATT&CK

MITRE's Acquire Infrastructure technique focuses on the act of acquiring resources, not on the deliberate, long-term **incubation of trust signals** within those resources. The temporal dimension — the patience required to build cloud account reputation — is a distinct, underappreciated adversary capability that modern detection stacks are not designed to counter.

---

## Procedure Examples

1. Register free-tier cloud accounts (AWS Free Tier, GCP Sandbox) under synthetic or compromised identities
2. Conduct low-volume, benign-looking activity over 60–180 days: hosting static web content, running small compute jobs, using cloud storage
3. Avoid triggering abuse detection by staying within free-tier limits and mimicking legitimate startup or developer usage patterns
4. Once the account has aged sufficiently and passed reputation checks, repurpose it for attack infrastructure (C2, phishing, data staging)
5. Route attack traffic through the warmed account to inherit its reputation score with downstream security controls

---

## Countermeasures

### Prevention

| Control | Description |
|---|---|
| **Tenant Age as a Risk Signal** | Implement security policies that apply heightened scrutiny to inbound traffic or API calls from cloud tenants younger than 90 days, regardless of provider. |
| **Cloud Reputation Scoring** | Integrate third-party cloud reputation feeds (e.g., Spur, GreyNoise, IP Quality Score) into WAF and API gateway rules to score inbound cloud-originating traffic. |
| **Allow-Listed Cloud Tenants** | For critical integrations, maintain an explicit allow-list of trusted cloud tenant IDs rather than relying on IP-based filtering. |

### Detection

| Control | Description |
|---|---|
| **Cloud Tenant Metadata Enrichment** | Enrich firewall and SIEM logs with cloud-provider tenant metadata (account age, organisational association, ASN registration date). |
| **Behavioural Anomaly on New Tenants** | Alert on any cloud tenant that transitions from zero activity to high-volume API calls against your infrastructure within a short window. |

### Response

- Automatically throttle or block traffic from cloud tenants that fail reputation scoring
- Feed confirmed malicious tenant IDs back into shared threat-intel feeds for community defence
